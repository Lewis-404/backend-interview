MQ / 消息重复消费与幂等性

消息重复消费 — 为什么有，怎么解决？

编号：MQ-01难度：⭐⭐⭐频率：🔥🔥🔥🔥🔥
💡 一句话结论
MQ 为保证可靠性（at-least-once 语义）会导致消息重复。解决方案：消费者做幂等处理——唯一消息 ID + Redis SetNX 防重、数据库唯一约束兜底、状态机前置判断。
🎬 生活类比 — 重复消费 = 快递送两次
用「收快递」看懂消息去重
📦
场景1：快递员送了两趟同一件货
你在家等快递，快递员按门铃——你签收了。过了一会儿，快递员又来了，还是同一件（网络超时重投、Broker 重发）。你拿起快递单一看：单号 XD20240524（消息 ID）——这本签收册上已经记过了！SETNX "XD20240524" = 签到表上写名字——写上了说明第一次收，写不上说明已经收了，直接说"不收"。
📋
场景2：签收册可能丢 —— 数据库兜底
万一签收册（Redis）被小孩撕了（重启/过期），怎么保证不重复收？数据库唯一约束 = 物业的系统里有记录。INSERT 时发现单号已存在（唯一索引冲突），系统自动拒绝重复录入。Redis SETNX 做快速判断 + DB 唯一索引兜底，双重保险。
🔄
场景3：状态机 = 快递状态追踪
快递有明确状态：待签收 → 已签收。收到快递时先查状态：已经是"已签收"？直接跳过。只有"待签收"才处理。UPDATE ... SET status='signed' WHERE id=? AND status='pending'——这是 CAS（Compare And Swap），防两个人同时签收同一件快递。

📋 7步答题模板

第一步：为什么会重复？

MQ 的投递语义有三种：at-most-once（可能丢）、at-least-once（可能重）、exactly-once（精确一次）。大多数 MQ 默认 at-least-once——生产者超时重发、Broker 主从切换重发、消费者未 ACK 重投。网络抖动也会导致重复。

第二步：幂等性定义

幂等性（Idempotence）：同一个操作执行一次和执行多次的效果相同。数学上 f(f(x)) = f(x)。MQ 场景下：同一条消息消费多次不会产生副作用（如重复扣款、重复下单）。

第三步：方案一——唯一消息 ID + Redis

生产者给每条消息生成全局唯一 ID（雪花算法/UUID）。消费者收到消息后：SETNX msg:consumer:{msgId} EX 3600，返回 1 说明第一次消费，处理业务；返回 0 说明已消费，直接 ACK 跳过。注意设置合理的过期时间。

第四步：方案二——数据库唯一约束兜底

利用数据库唯一索引作为最后防线。例如订单表 order_id 设唯一索引，INSERT 时用 INSERT IGNORE 或 ON DUPLICATE KEY UPDATE。即使 Redis 失效，数据库层面也不会产生重复数据。

第五步：方案三——状态机前置判断

在业务表中加状态字段（如 pending → processing → completed）。消费时先查状态：如果已经是 completed 则跳过；只有 pending 才执行。用 CAS（Compare And Swap）保证并发安全：UPDATE ... SET status='processing' WHERE id=xxx AND status='pending'。

第六步：方案对比

Redis 方案：性能最好（O(1)），但有丢失风险（重启/过期）。DB 唯一约束：最强保证，但有性能开销。状态机：适合有明确状态的业务，需要额外字段。生产环境常组合使用：Redis 做快速去重 + DB 唯一索引兜底。

第七步：各 MQ 的可靠性支持

RocketMQ：支持事务消息（半消息+回查），消费端返回 CONSUME_SUCCESS 才确认。Kafka：acks=all 保证生产可靠，手动提交 offset 保证消费可靠。RabbitMQ：publisher confirm + consumer ack。

"MQ 为了保证可靠性默认 at-least-once 语义，所以重复消费是必然的。解决方案是做消费者幂等，三种方式：一是唯一消息 ID + Redis SetNX，最快但可能丢失；二是数据库唯一索引兜底，最强保证但性能差一点；三是状态机前置判断，适合有明确状态的业务。生产环境我用 Redisson 分布式锁 + 唯一索引双向保证。"

⚠️ 面试常见错误 vs 正确回答

❌ 错误回答

"MQ 不会重复消费"

"加个锁就行"

"Redis 就够了不需要 DB"

✅ 正确回答

"MQ 默认 at-least-once，重复消费是常态，消费者必须做幂等"

"锁能防止并发，但不能防止消息重投——SETNX 才是防重"

"Redis 有丢失风险，DB 唯一约束做最终兜底才是完整方案"
