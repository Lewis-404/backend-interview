幂等性设计 — 分布式系统的基石

编号：MQ-03难度：⭐⭐⭐频率：🔥🔥🔥🔥🔥
💡 一句话结论
幂等性是指同一操作执行多次结果不变。分布式系统幂等设计：唯一 ID 生成（雪花算法/UUID）、数据库去重表/唯一索引、Redis SetNX、状态机判断（已处理则跳过）。需考虑并发安全。
🎬 生活类比 — 幂等性 = 电梯关门按钮
用「电梯按钮」看懂幂等性设计
🛗
场景1：电梯关门按钮按 10 次，门也只关一次
你进了电梯按关门键——连按 10 次，门也不会关 10 次。这就是幂等性——同一操作执行多次，结果和执行一次完全一样。电梯的逻辑：门已经是"关闭中"状态，再按关门键就忽略。如果没有幂等性：按一次扣一次钱——按 10 次扣 10 次钱，这就出大事了。
📝
场景2：去重表 = 签到表，SETNX = 签到簿
公司开会，进门先签到。去重表（idempotent_record） = 签到表——每人（每条消息 ID）只能签一次，重复签会被拒绝（唯一索引冲突）。Redis SETNX = 电子签到簿——扫一下工卡，系统说"已签到"就直接通过，不用去翻纸质签到表。签到簿可能没电（Redis 重启），纸质签到表还在（DB 兜底）。
🔐
场景3：并发安全 = 两个人同时按关门键
两个人同时进电梯，同时按关门键。如果没有防护，电梯可能"关两次"——这就不是幂等性的事了，是并发安全的问题。解决方案：SETNX 天然原子（一次只能一个人签到）、DB 唯一索引串行化（同时 INSERT 只有一个成功）、CAS 状态机（UPDATE WHERE status='pending'，affected_rows=0 说明被并发处理了）。

📋 7步答题模板

第一步：数学定义与意义

幂等性（Idempotence）：f(f(x)) = f(x)，同输入执行多次与执行一次结果相同。HTTP 方法中 GET/PUT/DELETE 是幂等的，POST 不是。分布式系统中网络重试、消息重投、超时重发是常态，没有幂等性就会产生副作用（重复扣款、重复创建资源）。

第二步：幂等 ID 生成

每条请求/消息需要一个全局唯一标识。雪花算法（Snowflake）：64 位 long，1 位符号+41 位时间戳+10 位机器 ID+12 位序列号，单机 QPS 可达 400 万。UUID：128 位，简单但有碰撞概率（极小）且索引性能差。业务 ID：如 order_id、trade_no 天然唯一，更推荐优先用业务 ID。

第三步：数据库去重表

建一张 idempotent_record 表：idempotent_key VARCHAR(128) UNIQUE、created_at TIMESTAMP。每次请求先 INSERT INTO idempotent_record VALUES(key, NOW())，唯一索引冲突说明已处理。可以和业务操作放在同一事务中保证原子性。INSERT IGNORE 或 ON DUPLICATE KEY 也能用。

第四步：Redis SetNX 方案

SET key value NX EX ttl：key 为业务 ID（如 order:{id}:pay），NX 表示只有不存在时才设置成功。返回 OK → 第一次，执行；返回 nil → 已处理，跳过。注意设置合理过期时间（不能短于业务处理时间）。缺点：Redis 重启可能丢 key，需要 DB 唯一索引兜底。

第五步：状态机判断

业务对象加状态字段（如订单：待支付→已支付→已发货→已完成）。处理前先查状态：if status == "待支付" then 扣款并更新为已支付；else return "已处理"。用数据库乐观锁保证并发安全：UPDATE order SET status='paid' WHERE id=? AND status='pending'，affected_rows=0 说明已被并发处理。

第六步：并发安全

幂等性不仅要防重复，还要防并发。Redis SETNX 本身是原子的。DB 去重表 + 唯一索引天然串行化。状态机 + CAS（Compare And Swap）也是原子的。最差情况可以加分布式锁（Redisson），但锁只解决并发不解决重复，需要和幂等方案配合使用。

第七步：RPC 幂等方案

不只是 MQ，RPC 调用也可能重试（超时、负载均衡重试）。方案：①客户端生成 TraceID 传给服务端，服务端做去重；②网关层统一幂等（基于 TraceID + Redis）；③gRPC 拦截器做幂等校验。金融场景最好做端到端幂等。

⏳
60 秒内说清
"幂等性就是同一次操作执行多次结果不变。实现分四层：第一层 ID 生成——雪花算法或业务 ID；第二层去重表——DB 唯一索引做最强保证；第三层 Redis SetNX——性能好但可能丢 key；第四层状态机——适合有明确状态的业务。关键是并发安全——SETNX 和唯一索引天然原子，状态机用 CAS 乐观锁。生产环境 Redis+DB 双保险。"

⚠️ 面试常见错误 vs 正确回答

❌ 错误回答

"幂等就是加个锁"

"Redis SETNX 就够了"

"UUID 做幂等 ID 最好"

✅ 正确回答

"锁解决并发，不解决重复——两次非并发的请求锁不住"

"Redis 可能丢 key，必须 DB 唯一索引兜底"

"UUID 索引性能差，业务 ID 或雪花算法更优"

🌲 追问树

幂等性
├─ 什么是幂等性？
→ f(f(x))=f(x)，多次执行结果相同
├─ 为什么 MQ/RPC 需要幂等？
→ 网络重试/超时重发/消息重投 不可避免
├─ ID 生成方案有哪些？
→ 雪花算法(64位long)/UUID(128位)/业务ID(最推荐)
├─ DB 去重怎么做？
→ 唯一索引 + INSERT IGNORE / ON DUPLICATE KEY
├─ Redis SETNX 怎么用？
→ SET key val NX EX ttl，原子操作，高性能
├─ 状态机方案？
→ 先查状态 + CAS 更新：UPDATE ... WHERE status='pending'
└─ 怎么保证并发安全？
→ SETNX 天然原子 / 唯一索引 / CAS / 分布式锁保底
