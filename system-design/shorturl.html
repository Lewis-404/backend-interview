🔗 短链系统设计

编号 SD-05 分类：URL 服务 难度：⭐⭐⭐ 频率：中高频
💡 一句话结论
短链核心：长 URL → 哈希/发号 → 短码（Base62 编码）→ 302 重定向。关键：哈希冲突（MurmurHash + 布隆过滤器）、发号器（雪花/DB 自增）、缓存策略（热点缓存在 Redis）、301 永久 vs 302 临时、过期策略（TTL + 惰性删除）。

场景2：短码 = 索书号
图书馆给每本书一个索书号：I247.5/100（7 个字符），拿着这个号就能找到对应的详细位置（长 URL）。Base62 编码就是把数字 ID 转成 0-9a-zA-Z 的短码——就像 ISBN 号把长书名压缩成标准编号。
👨‍💼
场景3：302 = 管理员告诉你去哪
你拿着索书号问管理员，管理员查了电脑说："三楼 A 区 12 排 5 架 3 层"（302 重定向），你自己走过去拿书。为什么用 302 不用 301（永久）？因为管理员要统计"这本书今天被借了几次"（访问统计），还要检查"这本书还在不在馆内"（过期检测）。301 会让读者下次直接去位置，不看管理员了。
🔢
场景4：Base62 = 索书号编码规则
图书馆索书号用字母+数字，短链用 Base62：0-9, a-z, A-Z = 62 个字符。7 位索书号能表示 62^7 = 3.5 万亿 本书！对比：纯数字 7 位只能表示 1000 万本。这就是为什么短链可以只用 7 个字符——编码空间足够用几十年。

🗺️ 7 步回答模板

Step 1 — 是什么

短链服务将长的 URL（可能几百个字符）压缩为短的唯一标识符（6-8 个字符），用户访问短链时服务端 302 重定向到原始长 URL。典型产品：t.cn（新浪微博）、bit.ly、TinyURL。核心需求：短码唯一（不能两个 URL 映射到同一个短码）、快速重定向（毫秒级）、高并发（每天数十亿次访问）。

Step 2 — 为什么需要

① 字符限制：微博 140 字限制，SMS 短信 70 字，长 URL 会占满全部空间；② 美观与信任：短链比一长串参数更美观，也更容易记；③ 统计追踪：短链可以做点击统计、来源分析、防钓鱼检测（预览真实 URL）；④ 二维码友好：短链生成的二维码更简单，识别更快。

Step 3 — 解决什么核心问题

三个核心：① 短码唯一性——不同长 URL 不能生成相同的短码（哈希冲突处理）；② 高并发重定向——访问短链时毫秒级返回 302，热点 URL 需要缓存；③ 大规模存储——每天可能生成上亿短链，存储和查询需要高效设计（DB 分片 + 布隆过滤器）。

Step 4 — 核心原理

生成短码两种方案：
① 哈希法：长 URL 经 MurmurHash 生成 64 位哈希值，取前/后 42 位（2^42 = 4 万亿）用 Base62 编码为 7 位字符。冲突处理：检测到冲突后在 URL 末尾追加随机串重新哈希。
② 发号器法：自增 ID（雪花算法/DB auto_increment），将 10 进制 ID 转 Base62 编码。如 ID=12345 → Base62("dnh")。优点是无冲突，缺点是 ID 可预测（可加密或随机化）。

访问流程：浏览器请求 short.url/abc123 → 服务端查 Redis 缓存 → 缓存命中返回 302 + Location → 缓存未命中查 DB → 更新缓存 → 302 重定向。

Step 5 — 缺点/局限

① 哈希冲突：不同 URL 可能产生相同哈希（需冲突处理 + 布隆过滤器预判）；② 安全风险：短链隐藏了真实 URL，可能被用于钓鱼/恶意网站（需要安全检测 + 预览页）；③ 存储增长：每天亿级短链，存储随时间线性增长（需要过期 + 归档策略）；④ 热点问题：热门短链（如微博热搜）瞬时数百万 QPS，单点 Redis 不够（需要 CDN + 本地缓存）。

Step 6 — 如何解决缺点

① 冲突处理：用布隆过滤器快速判断短码是否存在，存在则重试（长 URL + 随机盐 → 再哈希）；② 安全：创建短链时调安全 API 检测目标 URL，发现恶意链接拒绝生成；③ 过期清理：设置 TTL（如 30 天未访问），定时任务 + 惰性删除（访问时检查是否过期）；④ 热点高并发：CDN 边缘节点缓存 302 响应（Location 头），源站压力降低 99%。

Step 7 — 实际落地

TinyURL 方案：MySQL 自增 ID + Base62 编码，简单可靠。新浪微博 t.cn：发号器 + Base62 + Redis 缓存 + CDN 重定向。最小可行方案：Go HTTP Server + MySQL（短码 → 长 URL 映射表）+ Redis 缓存热点 + 302 响应。生产优化：CDN 缓存 302 重定向（Cache-Control: public, max-age=86400），只有首次访问到源站。

60 秒内说清
"短链 = 长 URL → 哈希/发号 → Base62 编码 → 短码 → 302 重定向。生成有两种：哈希法用 MurmurHash 产生 64 位哈希取 42 位 Base62 编码成 7 位字符，冲突了加随机盐重试；发号器法用自增 ID 转 Base62，无冲突。访问时：先查 Redis 缓存，命中直接 302，未命中查 MySQL 再缓存。热点 URL 用 CDN 缓存 302 响应，防钓鱼用安全检测 API，过期链接用 TTL + 定时清理。"

⚠️ 千万别这么回答

❌ 错误表达

  • "用 MD5 取前 6 位就行"（MD5 碰撞 + 只取 6 位字符集小，冲突概率极高）
  • "每次查 MySQL"（热点 URL 几十万 QPS 会打挂 DB，必须加缓存）
  • "301 永久重定向"（浏览器会缓存 301，后续访问不再经过服务端，无法统计）
  • "用 Redis 随机生成 6 位字符串"（没有唯一性保证，可能重复覆盖）
  • "短码越长越好"（7 位 Base62 = 3.5 万亿组合，完全够用，没必要更长）

✅ 正确表达

  • "用 MurmurHash（速度快分布好）或发号器（自增 ID → Base62），冲突率极低"
  • "302 临时重定向，每次请求都经过服务端，可以做访问统计和过期检查"
  • "Redis 缓存热点 URL 映射，CDN 缓存 302 响应，源站只需处理首次请求"
  • "Base62（0-9a-zA-Z）= 62 个字符，7 位 = 62^7 = 3.5 万亿，足够用几十年"
  • "短链过期：TTL + 定时任务 + 惰性删除，访问时检查是否过期，过期返回 410"

🌲 追问树

🔗 短链系统
├─ → 短码生成方案
│ ├─ 哈希法：长 URL → MurmurHash(64bit) → 取 42bit → Base62 → 7 位短码
│ ├─ 发号器法：雪花 ID / DB 自增 → 十进制转 Base62 → 无冲突
│ └─ 混合法：发号器生成 ID → 加盐哈希打乱 → Base62（既唯一又不可预测）
├─ → Base62 编码详解
│ ├─ 字符集：0-9, a-z, A-Z = 62 个字符
│ ├─ 7 位 = 62^7 = 3.52 × 10^12 ≈ 3.5 万亿种组合
│ └─ 转换：十进制 ID 不断对 62 取模，余数映射字符，商继续除
├─ → 哈希冲突处理
│ ├─ MurmurHash 碰撞率约 1/2^64，几乎可忽略但仍需处理
│ ├─ 布隆过滤器：快速判断短码是否已被占用（可能有假阳性）
│ └─ 冲突时：长 URL + 随机盐 → 再哈希 → 重试直到不冲突（概率极低，< 3 次）
├─ → 301 vs 302
│ ├─ 301（永久）：浏览器缓存重定向 URL，后续直接跳转不过服务端
│ ├─ 302（临时）：每次请求都到服务端，可统计点击、检查过期
│ └─ 短链场景用 302：需要统计访问次数 + 动态更新目标 URL + 过期检测
├─ → 缓存策略
│ ├─ 热点预取：统计 Top 1000 高频短链，提前加载到 Redis
│ ├─ CDN 边缘缓存：302 响应缓存在 CDN 节点，命中的直接返回
│ └─ 本地缓存：服务内存 Guava Cache 缓存最近 1000 个热门映射
├─ → 高并发重定向优化
│ ├─ CDN 层缓存 302（Cache-Control: public, max-age=86400）
│ ├─ Nginx 层直接返回（Lua 脚本读 Redis 后构造 302）
│ └─ 应用层：Go HTTP Server 极轻量处理（只读 Redis + 写 Header）
├─ → 过期清理
│ ├─ 创建时设置 TTL（如 30 天未访问即过期）
│ ├─ 惰性删除：访问时检查 created_at，过期返回 410 Gone
│ └─ 定时任务：每天凌晨扫描过期记录 → DELETE FROM short_urls
└─ → 域名选择
├─ 短域名：如 t.cn / bit.ly，域名越短短链越短
├─ 独立域名：避免与主站 cookie 混杂（安全隔离）
└─ 多域名分片：按短码首字符路由到不同域名，分散热点
🧠 记忆钩子
"短链 = 图书馆索书号系统：长 URL = 书的详细存放位置（三楼 A 区 12 排 5 架 3 层），短码 = 索书号（I247.5/100），302 = 管理员根据索书号告诉你去哪个位置拿书。哈希法 = 给每本书按内容算一个编号（可能会重复），发号器法 = 新书入库挨个发编号（不会重复），Base62 = 把纯数字编号变成字母数字混合的短编号。"
