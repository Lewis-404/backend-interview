💰 支付系统设计

编号 SD-03 分类：金融系统 难度：⭐⭐⭐⭐⭐ 频率：必问
💡 一句话结论
支付系统核心：幂等性（支付单号唯一防重）、对账（T+1 对账文件）、分布式事务（本地消息表 + 最终一致性 + 补偿）、资金安全（状态机 + 审计日志 + 风控）。流程：生成支付单 → 调用支付渠道 → 回调通知 → 更新订单 → 通知业务。

场景1：支付单号 = 银行流水号
你去银行转账，银行给你一个唯一的流水号（支付单号）。这个流水号绝对不能重复——重复了就等于一笔钱转了两次。所以用雪花算法生成全局唯一 ID + 数据库唯一索引兜底，双重防重。这是支付系统最基础的保障。

场景2：对账 = 月底对账本
每个月银行（支付渠道）出一份对账单（T+1 对账文件 CSV），你拿着自己的记账本逐行比对：你有银行没有（长款）——钱到账了但没记录？银行有你没有（短款）——被偷偷扣了？金额对不上（差异账）——到底转了多少？这是资金安全的最后一道防线。
📝
场景3：分布式事务 = 多人记账先记草稿
支付成功后要同时通知三个部门：财务（订单状态）、仓库（发货）、客服（通知用户）。不能"打了个电话就以为都知道了"——用本地消息表：在同一个事务里写一条"待通知"消息，异步推给各个系统，每个都要确认收到（ACK），一个没收到就重试（最多 5 次），全失败就人工介入。
📞
场景4：回调 + 主动查询 = 双重确认
你转了钱，银行说"到账了会通知你"（渠道异步回调）。但你敢只等着通知吗？万一通知丢了怎么办？所以还要定时主动查询——扫描所有 PENDING 超过 30 秒的支付单，直接打电话问银行"这笔到底成没成"。回调 + 主动查询双保险，钱不会凭空消失。

🗺️ 7 步回答模板

Step 1 — 是什么

支付系统是电商/金融平台的核心基础设施，负责处理用户付款请求、对接第三方支付渠道（微信/支付宝/银联）、管理资金流转。和普通业务系统最大的区别：钱的事情，差一分都不行。这意味着系统设计需要在 CAP 定理中优先保证 C（一致性），对账和补偿机制是生命线。

Step 2 — 为什么需要

普通业务系统可以接受"最终一致性"（如订单状态晚几秒更新），但支付系统涉及真金白银：① 同一笔支付不能扣两次（需要幂等）；② 支付成功但订单未更新会引发客诉（需要可靠回调+补偿）；③ 渠道方和我们的账对不上就是资损（需要 T+1 对账）；④ 资金流向必须有据可查（需要审计日志 + 状态机跟踪）。

Step 3 — 解决什么核心问题

三个核心问题：① 幂等性——支付单号全局唯一，同一笔订单无论请求多少次只扣一次款（唯一索引 + 状态机判断）；② 一致性——支付渠道回调可能丢失、延迟或重复，需可靠回调处理 + 主动查询补偿（定时查询渠道订单状态）；③ 资金安全——防止篡改金额（签名验证）、防止内部作弊（操作审计日志）、异常交易阻断（风控规则引擎）。

Step 4 — 核心原理

支付全流程：
① 生成支付单：业务侧传入订单号和金额，支付系统生成全局唯一支付单号（雪花算法 + 业务前缀），状态 = CREATED；
② 调用支付渠道：通过渠道 SDK（微信 JSAPI/支付宝 APP）发起支付请求，返回 prepay_id 给前端拉起收银台；
③ 回调处理：渠道异步通知支付结果到回调 URL，系统必须：验签（用渠道公钥验证回调参数）、幂等处理（按支付单号查状态，已终态的忽略）、更新支付单 + 订单状态；
④ 主动查询补偿：定时任务扫描超过 30 秒仍为 PENDING 的支付单，主动调渠道查询接口确认状态，关单超时未支付的订单；
⑤ T+1 对账：渠道每天出对账文件（CSV），系统下载后逐行比对：我方有渠道无（长款），渠道有我方无（短款），金额不一致（差异账），生成差错报告人工处理。

Step 5 — 缺点/局限

① 跨系统一致性难保证：支付成功后更新订单、积分、优惠券等多个系统，任一失败需全部补偿（用本地消息表保证最终一致性）；② 对账滞后：T+1 对账意味着前一天的差错要到第二天才发现，实时监控重要；③ 渠道差异大：微信和支付宝的 API 设计、回调格式、对账文件格式各不相同，需要适配层；④ 合规压力：存储敏感信息需 PCI-DSS 认证，近年人民银行对支付机构的监管趋严。

Step 6 — 如何解决缺点

① 本地消息表：支付完成后在同一事务中写入"待通知业务"消息，异步推送业务系统，失败则重试（最多 5 次），超过次数人工介入；② 实时监控：对账前做实时数据比对（每 10 分钟比对一次交易量），发现异常即时告警；③ 渠道适配器模式：统一支付接口 + 各渠道实现类（微信/支付宝/银联），新增渠道只需加一个适配器；④ 合规：支付数据与业务数据物理隔离存储，卡号等敏感信息 token 化（用渠道返回的 token 代替明文）。

Step 7 — 实际落地

业界实践：支付宝自研 OceanBase 做分布式事务（XA + 两阶段提交）；微信支付用 PaxosStore 保证单笔交易的强一致性。中小公司推荐方案：① 支付单号用雪花算法保证全局唯一；② 幂等靠 DB 唯一索引（pay_no UNIQUE）+ 状态机；③ 分布式事务用本地消息表（同一 DB 内 ACID，跨系统最终一致）；④ 对账用定时任务下载对账文件 + 分页比对。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"支付系统的核心就三件事：不重复扣款（幂等），钱不会丢（对账），状态是对的（一致性）。怎么做呢？支付单号全局唯一 + DB 唯一索引保证幂等；渠道回调来了先验签再处理，同时定时主动查询补充可能丢失的回调；每天 T+1 对账，逐行比对渠道和我们的交易记录发现差错。跨系统的一致性用本地消息表：支付成功后在事务里插一条通知消息，异步推给订单/积分/优惠券，失败就重试。"

⚠️ 千万别这么回答

❌ 错误表达

  • "调微信/支付宝 SDK 就行了"（只管了支付渠道，没考虑幂等、对账、补偿）
  • "用分布式事务/2PC 保证一致性"（支付场景不适合强一致分布式事务，性能差且渠道不支持）
  • "MQ 发个消息就完事了"（MQ 可能丢消息，没有对账机制兜底）
  • "前端轮询等支付结果"（不该只依赖前端，回调+主动查询才是正道）
  • "回调来了直接更新数据库"（没有验签，没有幂等，安全性和正确性都没有）

✅ 正确表达

  • "支付单号全局唯一 + DB 唯一索引 + 状态机，三者配合保证幂等"
  • "渠道回调必须验签 + 幂等处理，不能直接信任回调数据"
  • "本地消息表 + 最终一致性 + 补偿重试，是支付系统跨服务数据一致的标准方案"
  • "T+1 对账是资金安全的最后一道防线：逐行比对对账文件，发现平账/长款/短款/差异"
  • "主动查询补偿：定时扫描 PENDING 状态的支付单，调渠道查询接口确认真实状态"

🌲 追问树

💰 支付系统
├─ → 支付单号生成（防重/幂等）
│ ├─ 雪花算法：machine_id + timestamp + sequence，全局递增唯一
│ ├─ DB 唯一索引：pay_no UNIQUE KEY，重复插入报错
│ └─ 状态机：CREATED → PENDING → SUCCESS/FAILED/CLOSED，终态不可逆
├─ → 支付渠道对接
│ ├─ 微信 JSAPI/APP/H5/Native 多种支付方式
│ ├─ 支付宝 APP/PC/H5 支付
│ └─ 适配器模式：统一接口 + 各渠道实现，新增渠道零侵入
├─ → 回调处理（关键环节）
│ ├─ 验签：用渠道公钥验证回调参数签名，防伪造请求
│ ├─ 幂等：按 pay_no 查当前状态，已终态直接返回成功
│ └─ 更新：在同一事务中更新支付单 + 写入本地消息表
├─ → 对账（T+1 文件对账）
│ ├─ 下载渠道对账文件（微信每天凌晨生成前一天交易 CSV）
│ ├─ 逐行比对：我方有渠道无 = 长款，渠道有我方无 = 短款
│ └─ 差异处理：生成差错报告 → 人工确认 → 调账/补单/退款
├─ → 分布式事务方案
│ ├─ 本地消息表：支付成功后事务内写入消息，异步推业务系统
│ ├─ 最终一致性：接收方保证幂等，发送方保证至少一次投递
│ └─ 补偿机制：推送失败 5 次重试 → 人工介入（工单/告警）
├─ → 资金安全
│ ├─ 状态机：严格定义所有合法状态转换（如 SUCCESS 不能回退到 PENDING）
│ ├─ 审计日志：每次状态变更记录操作人/时间/IP/变更前/变更后
│ └─ 风控：单笔限额、日累计限额、异地登录拦截、设备指纹
└─ → 退款处理
├─ 生成退款单（关联原支付单号）
├─ 原路退回（微信付的退回微信，支付宝付的退回支付宝）
├─ 退款幂等（退款单号唯一 + 状态机）
└─ 退款对账（退款流水也参与对账）
🧠 记忆钩子
"支付 = 银行转账流程：支付单号 = 转账流水号（绝对不能重复），对账 = 每月底拿银行对账单和你的记账本逐笔对，分布式事务 = 你同时要通知财务部、仓库、客服三个人，不能打个电话就以为都知道了——得挨个确认，一个没通知到就记下来等下再通知（本地消息表 + 重试补偿）。"
