💬 IM 即时通讯系统

编号 SD-02 分类：实时通信 难度：⭐⭐⭐⭐ 频率：高频
💡 一句话结论
IM 核心：长连接（WebSocket/TCP）+ 心跳保活（30s）+ 消息可靠投递（ACK + 重传 + seq 去重）+ 离线消息存储。架构：接入层（长连接网关）→ 逻辑层（消息路由 / 状态同步）→ 存储层（Redis 缓存 / DB 持久化 / COS 媒体）。

场景1：长连接 = 电话线一直通着
传统 HTTP 像传呼机——你 ping 一下我回一下，每轮都要重新拨号。WebSocket 长连接就是电话打通了不挂——电话线（TCP 连接）一直连着，双方随时说话，延迟从秒级降到毫秒级。这就是 IM 实时性的根基。

场景2：心跳 = 问"你还在吗"
电话虽然通着，但万一对方睡着了呢？每隔 30 秒问一声——"喂，还在吗？"（ping/pong）。问 3 次没人应（90 秒无响应），挂电话，标记离线。这就是心跳保活——不断连才知对方在线。
✅
场景3：ACK = 对方说"听到了"
你说了一句话，对方回"听到了"（ACK 确认）。万一对方没回？等 3 秒再重复一遍（超时重传），再等 9 秒、再等 27 秒（指数退避）。3 次都没回应？留个言存起来（离线消息队列），等对方上线再听。TCP 只保证网络层送达，应用层必须自己确认。
📋
场景4：seq 序号 = 每句话前面编号
电话里说了一大段话，万一网络抖动，后说的先到了怎么办？每条消息前面加个递增长号——对方按号排序，保证不乱。收到过的号（重传的）就扔掉（去重）。这就是 TIMELINE 模型——每个人的聊天记录是一条按 seq 序号排列的时间线。

🗺️ 7 步回答模板

Step 1 — 是什么

IM（Instant Messaging）即时通讯系统，提供用户间实时文本、图片、语音、视频消息的收发能力。典型产品：微信、钉钉、Slack。核心要求：低延迟（< 100ms 送达）、高可靠（消息不丢不重）、高并发（万人群/亿级用户）。

Step 2 — 为什么需要

HTTP 短轮询每 2 秒请求一次，N 个用户在线就是 N/2 QPS 的无效请求，且延迟最低 2 秒。长轮询（Comet）虽然节省连接，但服务端持有连接加重资源消耗。WebSocket 是真正意义上的全双工长连接，一个 TCP 连接双向通信，延迟降到毫秒级，服务端能主动推送。TCP 自定义协议更极致（无 HTTP 头部开销），适合千万级在线场景（如微信）。

Step 3 — 解决什么核心问题

三个核心：① 实时性（消息毫秒级送达）——长连接 + 服务端主动推送；② 可靠性（消息不丢不重不乱序）——ACK 确认 + 超时重传 + seq 序号去重 + 接收方按 seq 排序；③ 大规模连接管理（百万在线）——接入层网关集群 + 一致性哈希路由 + 连接迁移。

Step 4 — 核心原理

① 长连接网关：用户建立 WebSocket 连接到接入层，网关维护 uid→connection 映射（存储在本地内存 + Redis 作为备份）；
② 心跳保活：客户端每 30 秒发 ping，服务端回 pong，超时 3 次（90 秒无响应）断开连接并清理映射；
③ 消息投递流程：发送方→逻辑层（生成全局递增 seq + 存储消息到 Timeline）→路由层（查接收方在线状态）→如果在线推送→接收方收到后回 ACK；如果离线→写入离线消息队列；
④ TIMELINE 模型：每个用户/群有一条消息 Timeline（按 seq 递增的消息链表），写操作追加新消息，读操作按 seq 范围拉取；
⑤ 群聊扩散：写扩散（发一条写 N 条 Timeline，适合小群）vs 读扩散（发一条写一条群 Timeline，适合大群/万人大群）。

Step 5 — 缺点/局限

① 长连接开销：百万连接占用大量内存（每连接 4KB buffer → 4GB）；② 连接迁移：网关宕机时连接全部断开，用户需重连（通过一致性哈希 + 网关无状态设计缓解）；③ 消息时序：分布式环境下不同网关接收到的消息顺序可能不一致，依赖全局 seq 服务（雪花算法 / Redis incr）；④ 弱网环境：移动端网络切换（WiFi↔4G）导致断连，需快速重连 + 离线消息同步。

Step 6 — 如何解决缺点

① 连接数优化：epoll/kqueue IO 多路复用（Go netpoller），单机轻松 10 万+连接；② 网关高可用：无状态设计 + 一致性哈希路由（用户重连到新节点后从 Redis 恢复状态）；③ 绝对时序：使用 Lamport 逻辑时钟或全局 ID 生成器（雪花算法，递增趋势）；④ 弱网优化：QUIC 协议（0-RTT 重连）、消息分帧 + 批量 ACK、根据网络类型动态调整心跳间隔。

Step 7 — 实际落地

微信架构：长连接用 TCP 自定义协议 + Protobuf 序列化，消息存储用 PaxosStore（自研分布式 KV），群聊大群用读扩散。钉钉架构：接入层用 LVS + Nginx → 长连接网关（Netty/Go），存储用 MySQL 分库分表 + Redis。最小可行方案：Go goroutine + WebSocket + Redis Pub/Sub（广播）+ MySQL 存历史消息，适合 10 万以下在线用户。

60 秒内说清
"IM 核心是长连接 + 可靠投递。用户连到网关建立 WebSocket，网关维护 uid→连接映射。发消息时：逻辑层生成全局 seq 号存 Timeline，查到对方在线就实时推送，离线就存离线队列等上线拉。可靠性靠三样：ACK 确认（对方确认收到）、seq 去重（同一条消息不显示两次）、超时重传（没收到 ACK 就重发）。群聊写扩散存 N 条适合小群，读扩散存 1 条适合大群。心跳 30 秒保活，3 次超时断连。"

⚠️ 千万别这么回答

❌ 错误表达

  • "WebSocket 就是 HTTP 升级一下就行了"（没说长连接维护、心跳、断线重连）
  • "用 Redis Pub/Sub 就能做 IM"（Pub/Sub 不持久化，丢消息，没有 ACK）
  • "消息存 MySQL 就行"（没考虑读扩散/写扩散、分库分表、Timeline 模型）
  • "TCP 就是可靠的，不需要 ACK"（TCP 只保证网络层可靠，应用层仍需业务 ACK）
  • "群聊就是循环发"（大群 1000 人就是 1000 次 IO，性能灾难）

✅ 正确表达

  • "长连接需要心跳保活机制，30s ping/pong，超时 3 次断开"
  • "消息可靠性需要业务层 ACK + 重传 + seq 去重，不能只靠 TCP"
  • "TIMELINE 模型：每人一条时间线，按 seq 号追加和拉取消息"
  • "群聊扩散分写扩散（发 1 条写 N 条，适合小群）和读扩散（发 1 条写 1 条，适合大群）"
  • "离线消息存在离线队列 + TIMELINE，用户上线后按 seq 增量拉取"

🌲 追问树

💬 IM 即时通讯
├─ → 长连接方案选型
│ ├─ WebSocket：基于 HTTP 升级，浏览器友好，适合 Web/小程序
│ ├─ TCP 自定义协议：更轻量（无 HTTP 头），适合移动端
│ └─ QUIC：0-RTT 重连，弱网优化，Google 出品
├─ → 心跳保活机制
│ ├─ 客户端每 30s 发 Ping，服务端回 Pong
│ ├─ 超时 3 次（90s 无响应）→ 断开连接 + 更新在线状态
│ └─ 自适应心跳：WiFi 下 60s，4G 下 30s，弱网 15s
├─ → 消息可靠投递（核心）
│ ├─ ACK 机制：接收方收到消息后返回 ACK，发送方 3s 内未收到则重传
│ ├─ Seq 去重：每条消息有全局递增 seq，客户端收到后去重
│ └─ 指数退避重试：1s → 3s → 9s → 27s，最多 3 次
├─ → TIMELINE 模型
│ ├─ 单聊：双方各一条 Timeline（发送方写 + 接收方写）
│ ├─ 按 seq 分段拉取：client 带着 last_seq 请求增量消息
│ └─ 冷热分离：热数据 Redis（最近 7 天），冷数据 MySQL（归档）
├─ → 群聊扩散策略
│ ├─ 写扩散：1 条消息 → 群内 N 个成员的 N 条 Timeline（小群 <200 人）
│ ├─ 读扩散：1 条消息 → 群 Timeline（1 条），成员拉取时读群 Timeline（大群）
│ └─ 混合策略：活跃成员写扩散，沉默成员读扩散
├─ → 已读/未读管理
│ ├─ 每个会话维护 last_read_seq（用户读到哪了）
│ ├─ 未读数 = 会话 max_seq - last_read_seq
│ └─ 群聊已读：维护每个成员的最后已读 seq，判断消息是否被所有人读
└─ → 万人在线优化
├─ 网关集群：Nginx/HAProxy 四层负载 → 一致性哈希路由
├─ IO 模型：epoll（Linux）/ kqueue（macOS）/ Go netpoller
├─ 消息合并：1 秒内的多条消息合并成 1 次推送
└─ 连接迁移：网关无状态 + Redis 存储 session，宕机后自动迁移
🧠 记忆钩子
"IM 像电话交换机：长连接 = 电话线一直通着不挂断，心跳 = 隔一会儿问'你还在吗'，ACK = 对方说'听到了我听到了'，seq = 每句话前面加个编号防止听乱，离线消息 = 你不在家我留个言在答录机里，群聊写扩散 = 我挨个打电话通知（人少时快），读扩散 = 我在公告栏贴通知大家自己看（人多时快）。"
