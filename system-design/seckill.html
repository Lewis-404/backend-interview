⚡ 秒杀系统设计

编号 SD-01 分类：高并发系统 难度：⭐⭐⭐⭐⭐ 频率：必问
💡 一句话结论
秒杀核心是"层层削峰"：CDN 静态化 → Nginx 限流（令牌桶）→ 网关鉴权+防刷（验证码 / IP 限频）→ Redis 预扣库存（Lua 原子操作）→ MQ 异步下单 → DB 最终扣减。关键：读写分离、缓存预热、限流熔断、降级预案。
🎬 生活类比 — 用「春运抢票」看懂秒杀系统层层削峰
用生活场景看懂全部概念

场景1：CDN = 遍布全国的代售点
春运前，12306 先把列车时刻表（商品详情页/静态资源）分发给全国代售点（CDN 节点）。旅客不用全挤到火车站（源站）——在家门口代售点就能看信息。90% 的流量在 CDN 层就被挡掉了。

👮
场景2：Nginx = 售票厅门口的警察
每个售票厅门口站一个警察（Nginx），用警戒线（令牌桶限流 limit_req）控制进厅人数。每秒只放 10 个人进去，超出的全拦外面："对不起，售票厅满员，请稍后再试"（返回 429）。刷票脚本（恶意高频请求）在这里就被干掉一大半。
🎫
场景3：Redis = 售票员看座位图
真正进了售票厅的人，售票员查的不是全国铁路数据库——他在看桌面上的实时座位图（Redis 预扣库存）。有没有票、扣一张，全在一条 Lua 脚本里原子完成：get → if > 0 → decr → 出票。一个人查完下一个人马上查，绝不超卖。
🎟️
场景4：MQ = 排号小票
售票员出了小票（Redis 扣成功），旅客拿着小票到旁边等（MQ 异步处理）。后台出票系统（DB）按自己的节奏一张张出票，不会被售票厅的人流冲垮。这就是削峰填谷——100 万人抢，DB 按自己能承受的速率慢慢处理。
🖨️
场景5：DB = 最终出票打座位号
后台出票系统（DB）消费 MQ 消息，用 乐观锁（version 字段） 最终扣库存，打印带座位号的正式车票。如果发现座位已经没了（乐观锁失败），标记异常——定时对账任务会找出 Redis 和 DB 的差值，人工处理退款。

🗺️ 7 步回答模板

Step 1 — 是什么

秒杀是高并发场景下，瞬间大量用户争抢极少量商品的电商活动。典型特征：瞬时 QPS 是平时的百倍以上、读多写少、热点商品集中。架构目标是保护后端服务不被冲垮，而非"承接所有流量"——大多数请求在到达 DB 前就应被拦截或排队。

Step 2 — 为什么需要

假设平时 1000 QPS 的系统，秒杀开始瞬间飙升到 100 万 QPS，如果请求直达 DB：① 连接池打满 → 所有请求超时；② 锁竞争（行锁）加剧 → 死锁；③ 缓存雪崩（大量 key 同时过期）→ DB 直接被压垮。必须用分层削峰策略把 100 万压到 DB 能承受的几千。

Step 3 — 解决什么核心问题

三个核心：① 超卖（卖出的数量超过库存）——是最严重的事故，必须通过 Redis Lua 原子操作 + DB 乐观锁双重保障；② 性能（系统不能被瞬间流量打垮）——逐层限流、异步化、缓存预热；③ 公平性（防刷/防脚本/防僵尸号）——验证码、IP 限频、用户风控。

Step 4 — 核心原理

分层削峰架构（从外到内）：
① CDN 层：商品详情页、图片等静态资源推到 CDN，用户请求不经过源站；
② Nginx 层：limit_req 模块按 IP+URL 做令牌桶限流（如每秒 10 个请求），超出直接返回 429；
③ 网关层：验证码验证（滑块）+ IP 黑名单 + 用户风控（同一用户 ID 1 秒内只能请求一次）；
④ Redis 层：库存预热到 Redis，下单时执行 Lua 脚本原子检查+扣减（get stock → if stock>0 → decr stock → 返回结果），全部在单次原子操作中完成；
⑤ MQ 层：扣减成功的请求异步写入 RocketMQ，削峰填谷（DB 按自己的节奏消费）；
⑥ DB 层：消费 MQ 消息，执行最终扣减（UPDATE ... SET stock=stock-? WHERE stock>=? AND version=?），乐观锁防止超卖。

Step 5 — 缺点/局限

① Redis 单节点瓶颈：预扣库存集中在单个 Redis key，热点问题严重（可用本地缓存 + Redis 集群分片缓解）；② 数据一致性：Redis 扣减成功但 DB 写入失败时需对账补偿（定时任务扫描+人工介入）；③ 延迟问题：MQ 异步化后用户不会立即看到结果（前端需轮询或 WebSocket 推送最终结果）；④ 恶意流量难以完全防范：代理 IP 池、打码平台可绕过部分风控（需持续优化规则引擎）。

Step 6 — 如何解决缺点

① Redis 热点：用本地 Guava Cache/Caffeine 做 L1 缓存 + Redis 做 L2，热 key 在应用内存中提前判断库存是否耗尽；② 对账补偿：定时任务扫描 Redis 和 DB 的差值，发现不一致时人工审核退款或补发；③ 用户体验：前端下单后展示"排队中"，WebSocket 推送结果，超时自动退款；④ 风控升级：结合设备指纹 + 行为分析（鼠标轨迹 / 点击速度）识别机器人。

Step 7 — 实际落地

淘宝双十一架构演进：早期用 Oracle + 存储过程硬扛；后来迁移到 MySQL + Redis；再后来自研 Tair（分布式缓存）+ RocketMQ + OceanBase。关键里程碑：全链路压测（提前找出瓶颈）、异地多活（多地同时秒杀）、单元化架构（用户按 ID 路由到固定单元）。中小公司落地最小方案：Nginx limit_req + Redis Lua + RocketMQ + MySQL，成本低效果好。

⏳
60 秒内说清
"秒杀的核心打法是层层削峰。最外层 CDN 挡住 90% 静态请求，Nginx 令牌桶限流挡住 9% 的恶意高频请求，网关做验证码和 IP 限频。真正进到业务的只有 1%，用 Redis Lua 脚本原子扣库存，扣成功的发 MQ 异步落 DB。超卖靠 Redis Lua + DB 乐观锁双重保障，数据一致性靠定时对账补偿。一句话：把 100 万 QPS 削到 DB 能扛的几千。"

⚠️ 千万别这么回答

❌ 错误表达

  • "加机器扩容就行了"（没考虑成本，也没解决超卖问题）
  • "用 Redis 做库存就行"（没说 Lua 原子操作，并发下会超卖）
  • "用消息队列异步下单"（只说了一个环节，缺少全链路视角）
  • "数据库加锁就解决了"（行锁在 10 万并发下直接死锁）
  • "前端按钮置灰防重复点击"（这只是前端防重，不能解决超卖）

✅ 正确表达

  • "分层削峰是核心思想：CDN → Nginx → 网关 → Redis → MQ → DB，每层削减一个数量级"
  • "Redis 扣库存用 Lua 脚本保证原子性，单次操作完成：检查+扣减+返回"
  • "MQ 不是所有请求都进，只有 Redis 扣减成功的才发 MQ，保证了削峰效果"
  • "超卖用 Redis Lua（预扣）+ DB 乐观锁 version（最终扣减）双重保障"
  • "全链路压测 + 降级预案（关非核心服务、页面静态化）是上线前的必须"

🌲 追问树

⚡ 秒杀系统
├─ → 前端防刷
│ ├─ 验证码（滑块/点选）—— 过滤简单脚本
│ ├─ 按钮防重（点击后置灰 + 前端计数）
│ └─ URL 动态化（秒杀链接每次不同，防提前构造请求）
├─ → CDN 静态化
│ ├─ 商品详情页纯静态（HTML/CSS/JS 推 CDN）
│ ├─ 库存数量通过 Ajax 异步获取
│ └─ 回源率极低（CDN 命中率 > 99%）
├─ → Nginx 限流
│ ├─ limit_req_zone：基于 IP 的令牌桶（rate=10r/s）
│ ├─ limit_conn：限制单 IP 并发连接数
│ └─ 超出限流返回 429 Too Many Requests
├─ → Redis 预扣库存（核心）
│ ├─ Lua 脚本：get stock → if stock>0 → decr → return OK
│ ├─ 原子性保证：单线程执行，无并发问题
│ └─ 库存预热：活动开始前将 DB 库存 load 到 Redis
├─ → MQ 削峰（RocketMQ）
│ ├─ 只有 Redis 扣减成功的请求才发 MQ
│ ├─ 消费者按 DB 处理能力消费（多线程/批量）
│ └─ 失败重试 + 死信队列兜底
├─ → DB 扣减 + 对账
│ ├─ 乐观锁：UPDATE SET stock=stock-1, version=version+1 WHERE id=? AND stock>=1 AND version=?
│ ├─ 定时对账任务：比对 Redis 和 DB 的库存差异
│ └─ 不一致时：人工介入 + 补偿退款
├─ → 超卖解决方案
│ ├─ Redis Lua 原子操作（第一道防线）
│ ├─ DB 乐观锁 version 字段（第二道防线）
│ └─ 兜底：对账发现超卖 → 人工退款 + 补发优惠券
├─ → 热点隔离
│ ├─ 热点商品单独部署（独立 Redis + 独立服务）
│ ├─ 本地缓存预判：应用内存中维护"已售罄"状态
│ └─ Redis 分片：按商品 ID 哈希分散到多个 Redis 节点
└─ → 降级预案
├─ 关停非核心服务（推荐/评论/物流查询）释放资源
├─ 限流降级：达到阈值后返回"活动太火爆，请稍后再试"
└─ 熔断：下游服务超时率达到阈值 → 快速失败 → 保护上游
🧠 记忆钩子
"秒杀像春运抢票：CDN = 遍布全国的代售点（分流），Nginx = 售票厅门口的警察控制入场人数（限流），Redis = 售票员实时看座位图还剩多少票（预扣库存），MQ = 排号小票让你在旁边等（异步削峰），DB = 最终出票打座位号（持久化）。你不需要一波人全挤进售票厅——大部分在门外就被分流了。"
