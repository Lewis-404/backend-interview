MySQL锁机制全解

TOP 04 ⭐⭐⭐⭐⭐

InnoDB锁体系：行锁(Record/Gap/Next-Key Lock)+表锁+意向锁；RR下Next-Key Lock（行锁+间隙锁）解决幻读；死锁检测通过wait-for graph超时回滚

🎬 生活类比 — 停车场车位管理
为什么停车场不会有两辆车停同一个车位？
🅿️
场景：行锁(Record Lock) = 车位锁
每个车位一个地锁，只有你降锁才能停——MySQL的行锁只锁你操作的那一行，别人其他行照常访问，互不干扰
⛓️
场景：间隙锁(Gap Lock) = 车位间的铁链
两个车位之间拉了铁链——虽然中间没车位，但也不能让新人插队停进来！这就是防幻读的本质：不让你INSERT新行进来
🔐
场景：临键锁(Next-Key Lock) = 车位锁+铁链
锁住车位本身+前面的铁链——彻底防御：你不能改我车位上的车，也不能在我前面插队停一辆。RR级别的默认锁机制
🚗💥
场景：死锁 = 两车互不相让堵死
A车想进B车位，B车想进A车位——两车互不相让→交通瘫痪。MySQL检测到后，交警(死锁检测)拖走代价最小的一辆(回滚事务)
🔒
行锁 = 车位锁
只锁一个位置，别人自由进出其他车位
⛓️
间隙锁 = 铁链
锁住空位，阻止插队，防幻读全靠它
🔐
临键锁 = 锁+链
双重保护，RR默认策略，彻底防幻读
💀
死锁 = 两车卡死
互等对方释放→交警拖走代价最小的

🗺️ 7 步回答模板

Step 1 — 是什么

锁是多用户并发访问数据库时保证数据一致性的机制。InnoDB支持行级锁、间隙锁、临键锁、意向锁、自增锁等

Step 2 — 为什么

没有锁，并发写同一条数据会导致数据错乱（丢失更新）。锁通过互斥访问保证并发安全性

Step 3 — 解决的核心问题

写-写冲突（丢失更新）；RR下幻读（间隙锁阻止INSERT）；元数据修改与DML冲突（元数据锁）

Step 4 — 核心原理

行锁在索引记录上加锁；间隙锁锁索引记录之间的间隙（不锁记录本身）；Next-Key Lock=行锁+前面间隙锁（左开右闭区间）

Step 5 — 缺点/局限

锁冲突导致等待和超时；间隙锁降低并发插入性能；死锁需要检测和回滚；锁太多可能导致锁升级到表锁

Step 6 — 如何解决

使用合适索引避免锁全表；控制事务大小和时间；死锁重试机制；乐观锁替代悲观锁（读多写少场景）

Step 7 — 实际落地

常用参数：innodb_lock_wait_timeout=50s（锁等待超时）；死锁检测：innodb_deadlock_detect=ON；监控：SHOW ENGINE INNODB STATUS\G

⏱️ 60 秒极简表达

⏳
60 秒内说清
InnoDB的锁分三层。最底层是行锁：Record Lock锁住索引记录本身，Gap Lock锁住索引记录之间的间隙（比如id=5和id=10之间），Next-Key Lock=Record+Gap（左开右闭区间）。RR默认用Next-Key Lock防幻读。中间层是意向锁(IS/IX)：行锁加之前先在表级标记，方便其他事务快速判断表上是否有行锁。上层是元数据锁(MDL)和自增锁：MDL保护表结构不被并发修改，自增锁保证AUTO_INCREMENT唯一性。死锁检测通过wait-for graph——如果发现环就回滚代价最小的事务。

⚠️ 千万别这么回答

❌ 错误表达

  • "行锁就是锁住一行" — 行锁加在索引上，没索引会锁全表
  • "Gap Lock锁的是记录" — 间隙锁不锁任何记录，锁的是索引间隙
  • "死锁会自动超时解决" — 死锁检测是主动发现循环等待并回滚，不是被动等超时
  • "乐观锁就是不加锁" — 乐观锁是应用层的版本号控制，不是数据库不锁

✅ 正确表达

  • "InnoDB行锁加在索引上，不是记录本身。没有索引的UPDATE会锁全表，这是隐式锁升级"
  • "Next-Key Lock = Record Lock + Gap Lock，锁定的是(上一个索引值, 当前索引值]的左开右闭区间"
  • "意向锁是表级锁，用来和表锁协调：行锁前先加意向锁，表锁只需检查意向锁就知道表上有无行锁"
  • "死锁处理分两条线：检测(wait-for graph主动发现环→回滚)和超时(innodb_lock_wait_timeout→回滚)"

🌲 追问树

MySQL锁体系
├─ → 行锁分哪三种？
│ └─ Record Lock：锁索引记录；Gap Lock：锁索引间隙(开区间)；Next-Key Lock：Record+Gap(左开右闭)。RC只用Record Lock，RR默认Next-Key
├─ → Next-Key Lock如何防幻读？
│ └─ WHERE id>5 FOR UPDATE→锁住(5,∞)的间隙，其他事务无法INSERT id=6/7/8...。但快照读(SELECT)不触发行锁，幻读仍可能发生
├─ → 意向锁(IS/IX)干什么？
│ └─ 行锁和表锁的"通告机制"。加行锁前先加IX意向锁，表锁加锁时只需检查意向锁是否冲突，不必遍历所有行锁
├─ → 元数据锁(MDL)是什么？
│ └─ MySQL 5.5引入。DML自动加MDL读锁，DDL自动加MDL写锁。防止SELECT时表被ALTER/DROP。常见问题：DDL等DML释放MDL导致阻塞
├─ → 死锁怎么发现和处理？
│ └─ wait-for graph：事务T1等T2→T2等T1→成环=死锁。检测到环后回滚undo量最小的事务(undo log越小代价越低)。超时是兜底(默认50s)
├─ → 乐观锁 vs 悲观锁怎么选？
│ └─ 悲观锁(SELECT FOR UPDATE)：冲突率高时用，保证不出错。乐观锁(version字段+CAS)：冲突率低时用，性能好但需要重试
└─ → 为什么没索引的UPDATE会锁全表？
└─ 行锁加在索引上。WHERE列无索引→全表扫描→每一行都要判断→所有行都被X锁锁住。所以DBA强调WHERE列必须有索引
🧠 记忆钩子
行锁=停车场车位锁（只锁自己的车位），间隙锁=车位间的铁链（没车也不能停在这个间隙），临键锁=车位锁+铁链——彻底保证不会有车插队（彻底防幻读）
3种 — 行锁：Record/Gap/Next-Key | 2种 — 意向锁：IS(共享)/IX(排他) | RC — 只使用Record Lock | RR — 默认Next-Key Lock
