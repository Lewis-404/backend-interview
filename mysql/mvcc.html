MVCC多版本并发控制

TOP 02 ⭐⭐⭐⭐⭐

MVCC通过Undo Log保留数据历史版本，每个事务读取快照(ReadView)，实现读不阻塞写、写不阻塞读；RC每次读生成新ReadView，RR事务内复用同一ReadView

🎬 生活类比 — Git 版本管理
多人同时改代码却不互相干扰 — 这就是 Git，也是 MVCC
💾
场景：每次 UPDATE = git commit
你改一行代码就commit一次——MySQL每UPDATE一行就产生一个新版本。不像Word那样覆盖保存，而是像Git一样保留完整历史
🔖
场景：RR 隔离级别 = git checkout 固定 commit
事务开始时"checkout"到某个commit，全程只看这个快照。别人后来提交的代码你看不到——这就是可重复读：整个事务看到同一份数据
🔄
场景：RC 隔离级别 = 每次都 git pull
每次读数据都先拉最新——别人刚提交的你能看到。这次读到3条，下次5条——因为别人在你读的时候又提交了2条
📜
场景：Undo Log = git log 历史记录
git log记录每次commit的变更——Undo Log记录每次修改前的旧值。回滚就像git revert：拿出历史版本还原
📜
Undo Log = git log
每次修改产生一条历史，串联成版本链
👁️
ReadView = 当前 checkout
事务看到的世界取决于checkout到哪个点
#️⃣
trx_id = commit hash
唯一标识每次修改，判断可见性全靠它
🔀
快照读 = 看commit日志
不干扰别人，只读自己版本的代码

🗺️ 7 步回答模板

Step 1 — 是什么

MVCC(Multi-Version Concurrency Control)是一种并发控制机制，通过维护数据的多个版本来实现事务隔离，让读操作不加锁也能读到一致数据

Step 2 — 为什么

传统锁机制下读-写互相阻塞，高并发场景吞吐量急剧下降。MVCC让读写互不阻塞，大幅提升并发性能

Step 3 — 解决的核心问题

解决"读-写冲突"——一个事务读取时另一个事务修改同一条数据，不用加锁也能保证读到一致性视图

Step 4 — 核心原理

每行记录有隐藏列DB_TRX_ID(最近修改事务ID)和DB_ROLL_PTR(指向Undo Log的回滚指针)。事务开始时创建ReadView(活跃事务列表)，读时沿着版本链找到对当前ReadView可见的版本

Step 5 — 缺点/局限

Undo Log占用空间且不能随便清理（长事务导致Undo堆积）；RR下可能出现"快照太旧"读到过期数据；不能完全替代锁（写-写冲突仍需锁）

Step 6 — 如何解决

避免长事务、监控Undo Log大小、对于需要最新数据的场景用SELECT ... FOR UPDATE（当前读）

Step 7 — 实际落地

RR隔离级别下快照读默认使用MVCC；当前读(SELECT FOR UPDATE/UPDATE/DELETE)走锁机制。核心参数：innodb_undo_tablespaces、innodb_max_undo_log_size

⏱️ 60 秒极简表达

⏳
60 秒内说清
MVCC让MySQL实现"读写不互斥"。每条记录有隐藏的事务ID和回滚指针，修改时不是原地更新而是通过Undo Log保存旧版本，形成版本链。每个事务开始时拍一张"快照"(ReadView)，记录当时活跃的事务ID列表。读数据时沿着版本链找到第一个对当前ReadView可见的版本——这就是快照读。RC和RR的区别在于：RC每次SELECT都生成新ReadView(能看到已提交的最新版本)，RR只在事务开始时生成一次(整个事务看到同一快照)。

⚠️ 千万别这么回答

❌ 错误表达

  • "MVCC就是不加锁" — 错误，写-写冲突仍需要行锁
  • "MVCC是MySQL特有的" — PostgreSQL、Oracle等也实现了MVCC
  • "ReadView就是事务开始时间戳" — ReadView是活跃事务ID数组，不是时间戳
  • "RR下MVCC能解决所有并发问题" — 不能解决写偏斜(Write Skew)

✅ 正确表达

  • "MVCC的核心是快照读——每个事务在某一时刻获得数据库的一致性视图"
  • "版本链通过DB_TRX_ID + DB_ROLL_PTR两个隐藏列连接Undo Log中的历史版本"
  • "RC和RR的区别本质是ReadView生成策略不同：RC每次读新建，RR事务级复用"
  • "MVCC解决读-写冲突，锁解决写-写冲突，两者配合才是完整方案"

🌲 追问树

MVCC 核心原理
├─ → 为什么需要MVCC？
│ └─ 读不加锁也能读一致性数据。没有MVCC时要么用锁(读写互斥→低并发)，要么脏读（读到未提交数据）
├─ → Undo Log版本链怎么工作？
│ └─ 每次UPDATE不会覆盖原数据，而是写新版本到Undo Log，原数据roll_ptr指向旧版本，形成单向链表
├─ → 隐藏列 trx_id + roll_ptr 做什么？
│ └─ trx_id=最后修改此行的事务ID，roll_ptr=指向Undo Log旧版本的指针。还有DB_ROW_ID(无主键时的隐藏主键)
├─ → ReadView可见性判断规则？
│ └─ ①trx_id<最小活跃事务ID→可见 ②trx_id>=最大活跃事务ID→不可见 ③在活跃列表中→不可见(未提交) ④不在且<最大→可见(已提交)
├─ → RC vs RR 本质区别？
│ └─ RC:每次快照读生成新ReadView→能看到别的事务已提交的修改。RR:第一次快照读生成ReadView后复用→整个事务看到同一快照
├─ → 当前读 vs 快照读？
│ └─ 快照读=普通SELECT(MVCC)；当前读=SELECT FOR UPDATE/UPDATE/DELETE/INSERT，读最新版本+加锁，不走MVCC
└─ → MVCC和锁怎么配合？
└─ 读→MVCC快照读（不加锁）；写→当前读+行锁。两者互补：MVCC处理读-写并发，锁处理写-写并发
🧠 记忆钩子
MVCC像Git版本管理：每次修改产生一个commit（版本），读操作看当时的快照而不是最新代码；RR = 一个事务的git checkout固定commit，RC = 每次读都git pull看看有没有新提交的
3个 — 隐藏列(trx_id/roll_ptr/row_id) | RC — 每次SELECT新建ReadView | RR — 事务级复用ReadView
