慢查询优化实战

TOP 06 ⭐⭐⭐⭐⭐

慢查询优化流程：①开启slow_query_log ②分析慢查询日志(pt-query-digest) ③EXPLAIN定位问题 ④优化策略:加索引/重写SQL/读多写少场景加缓存/数据量过大分库分表

🎬 生活类比 — 交通治理
一条 SQL 堵了 3 秒，就像早高峰的一个路口瘫痪了整条街
📷
场景：慢查询日志 = 交通摄像头
摄像头记录所有超时路口——慢查询日志记录所有执行超过0.5秒的SQL。没有摄像头你根本不知道哪条路堵了
🔍
场景：EXPLAIN = 堵点分析
交警站在路口数车流——EXPLAIN告诉你：哪个查询在用全表扫描(type=ALL)、扫描了多少行(rows=100万)、有没有建临时表(Using temporary)
🛣️
场景：加索引 = 加车道
路口太窄→拓宽+加车道。索引就像多修几条车道：原来一条路串行，加索引后多条路并行，车流速度瞬间翻倍
🚌
场景：缓存 = 公交专线
高频热点数据开Redis缓存——相当于给热门路线开公交专线(BRT)。大家都走快速通道，不用在普通车道排队
📷
日志 = 摄像头
记录所有超时的SQL，发现拥堵源头
🔍
EXPLAIN = 堵点分析
定位具体哪个路口(查询)最堵、原因是什么
🛣️
索引 = 加车道
拓宽路口，让车流(查询)并行通过
🌉
分库分表 = 建新路
单条路扛不住→建多条路分流，最后的手段

🗺️ 7 步回答模板

Step 1 — 是什么

慢查询指执行时间超过long_query_time阈值的SQL语句。MySQL提供慢查询日志记录所有超时SQL

Step 2 — 为什么

单个慢SQL可能拖垮整个数据库——占CPU、抢锁、占连接池，导致雪崩。线上99%的数据库性能问题都能追溯到几条慢SQL

Step 3 — 解决的核心问题

系统性发现、分析、优化数据库性能瓶颈。从"用户反馈系统变慢"到"精确定位哪条SQL慢了200ms"

Step 4 — 核心原理

慢查询日志记录SQL文本+执行时间+扫描行数+锁等待时间。pt-query-digest统计分析出TOP N慢SQL。EXPLAIN定位具体原因（没索引/索引失效/全表扫描）

Step 5 — 缺点/局限

慢查询日志本身有性能开销(约5%)；只记录超时的，不超时的"潜在慢SQL"发现不了；日志文件会很大需要定期清理

Step 6 — 如何解决

使用performance_schema的events_statements_summary_by_digest表替代慢查询日志（开销更低，覆盖所有SQL）。慢查询日志开启按天切分+自动清理

Step 7 — 实际落地

开启慢查询日志：set global slow_query_log=ON; set global long_query_time=0.5;。分析工具：pt-query-digest slow.log。优化策略：加索引→改写SQL→加缓存→分库分表

⏱️ 60 秒极简表达

⏳
60 秒内说清
慢查询优化是四步法。第一步开日志：把long_query_time设成0.5~1秒，记录所有慢SQL。第二步分析：用pt-query-digest跑日志，它会按执行总时长排名——找出耗时最多的几条SQL，这就是靶子。第三步定位：每条慢SQL用EXPLAIN看，99%的问题都是"没走索引""索引失效""扫描行数太大"三类。第四步对症：索引问题加索引、SQL写的烂就重写（比如把OR改成UNION、避免SELECT *、LIMIT大偏移量用子查询）、读多写少的加热点缓存、数据量实在太大考虑分区或分表。

⚠️ 千万别这么回答

❌ 错误表达

  • "慢查询优化就是加索引" — 只加索引是偷懒，很多慢SQL是因为SQL写法问题
  • "long_query_time设的越小越好" — 太小会导致日志爆炸且开销大，生产一般0.5~2秒
  • "发现慢SQL直接加force index" — 绕过优化器治标不治本，数据分布变了索引适得其反
  • "分库分表就能解决所有慢查询" — 分库分表是最后手段，引入分布式事务、跨库JOIN等新问题

✅ 正确表达

  • "先量化：pt-query-digest找出执行次数×平均耗时的TOP SQL；再定性：EXPLAIN定位具体原因"
  • "优化有优先级：加索引(成本最低)→改写SQL→加缓存→读写分离→分库分表(成本最高)"
  • "索引失效的经典场景：函数操作(WHERE DATE(create_time))、隐式类型转换(varchar列用数字查)、OR条件不全有索引、LIKE '%xxx'"
  • "profiling可以精确到微秒级：SET profiling=1; SHOW PROFILES; SHOW PROFILE FOR QUERY N; 看到Sending data占比高=回表太多"

🌲 追问树

慢查询优化四步法
├─ → 如何开启和分析慢查询日志？
│ └─ slow_query_log=ON，long_query_time=N秒，log_queries_not_using_indexes记录未用索引的SQL。分析：pt-query-digest生成TOP报告，mysqldumpslow快速预览
├─ → 慢查询常见原因有哪些？
│ └─ ①没走索引(没有/失效) ②索引不合适(选择性低) ③扫描行数太大(LIMIT大偏移) ④锁等待 ⑤大数据量排序/分组 ⑥返回数据太多(SELECT *) ⑦关联表太多
├─ → 索引失效的经典场景？
│ └─ WHERE列上使用函数(DATE(col))、隐式类型转换、LIKE '%xxx'前缀%、OR条件不同列索引、联合索引不满足最左前缀、!=/<>/NOT IN、IS NULL/IS NOT NULL(视索引情况)
├─ → 如何改写SQL做优化？
│ └─ 避免SELECT *；大OFFSET用子查询(WHERE id>上次max_id)；OR改UNION；IN改EXISTS(子查询是大表时)；COUNT(*)替代COUNT(col)；分页用延迟关联
├─ → 什么时候用缓存？
│ └─ 读多写少+实时性要求不高→Redis缓存。注意：缓存穿透(查不存在的数据)→布隆过滤器；缓存击穿(热点过期)→互斥锁/永不过期+异步更新
├─ → Profiling 怎么精确定位？
│ └─ SET profiling=1→执行SQL→SHOW PROFILES→SHOW PROFILE CPU,BLOCK IO FOR QUERY N。Sending data占比高=回表多；Creating tmp table=排序/分组有问题
└─ → 什么时候分库分表？
└─ 单表>2000万行且索引优化到极限后仍慢→分表(水平拆分)。单库QPS/连接数达到瓶颈→分库。先优化再拆分，别用分库分表掩盖设计问题
🧠 记忆钩子
慢查询优化像交通治理：先看监控（慢查询日志）→找堵点（EXPLAIN定位）→加车道（建索引）→限流（加缓存）→修新路（分库分表）。大多数拥堵只要加一条车道（索引）就解决了
4步 — 日志→分析→定位→优化 | 0.5~2s — 生产long_query_time | pt工具 — Percona Toolkit分析 | 2000万 — 单表分表参考阈值
