Explain执行计划解读

TOP 05 ⭐⭐⭐⭐⭐

EXPLAIN分析SQL的核心字段：type（访问类型从好到差：system→const→eq_ref→ref→range→index→ALL）、key_len（索引使用长度）、rows（估计扫描行数）、Extra（Using index/Using filesort/Using temporary）

🎬 生活类比 — 体检报告
一份体检报告，告诉你身体哪里出了问题
🩺
场景：type 字段 = 体检指标
ALL = 癌症晚期（全表扫描，最差）；index = 三高（全索引扫，也不健康）；range = 亚健康（范围扫描，及格线）；ref = 正常（索引查找，不错）；const = 运动员（主键等值，最优）
📊
场景：rows = 预估工作量
医生说"预估要处理1000个细胞样本"——rows=1000。如果实际需要处理的只有10个，那999个都是白费的——索引不合适，扫描了太多无用数据
⚠️
场景：Extra = 医生的特别嘱咐
Using filesort = "你需要额外做康复训练"；Using temporary = "需要住院观察"（建临时表）；Using index = "一切正常，身体很棒"（覆盖索引）
🔑
场景：key_len = 索引用到了多少
像药方里实际起效的药材克数——key_len越接近索引定义的总长度，说明索引利用率越高。key_len=4只用了第一列int，key_len=14可能三列全用上了
📋
type = 指标等级
ALL(癌症)→index(三高)→range(亚健康)→ref→const(最优)
📈
rows = 预估天数
rows越大→处理越多→越慢，优化目标越小越好
🏥
Extra = 医嘱
filesort=需康复/temporary=需住院/Using index=健康
💊
key_len = 起效剂量
越接近总长度=索引用得越充分

🗺️ 7 步回答模板

Step 1 — 是什么

EXPLAIN是MySQL提供的SQL执行计划分析工具，展示优化器如何执行这条SQL——用什么索引、扫描多少行、是否排序等

Step 2 — 为什么

不会看执行计划=盲人开刀。一条SQL某天突然慢10倍，EXPLAIN告诉你是否索引失效、是否全表扫描、是否产生临时表

Step 3 — 解决的核心问题

定位慢SQL的瓶颈点：是索引没用到？是用了但扫描行数太大？是产生了文件排序？还是创建了磁盘临时表？

Step 4 — 核心原理

MySQL优化器根据统计信息(索引基数、表大小)生成多个执行计划，选代价最小的。EXPLAIN展示的就是这个最终选择的计划

Step 5 — 缺点/局限

EXPLAIN给出的是估计值，实际执行可能偏差大（统计信息不准）。不执行SQL，无法感知锁等待、BufferPool命中率等运行时因素

Step 6 — 如何解决

MySQL 5.6+的EXPLAIN FORMAT=JSON给出更详细信息；MySQL 8.0的EXPLAIN ANALYZE实际执行并统计每一步的耗时

Step 7 — 实际落地

日常开发中EXPLAIN是必须步骤。关注type<=range、key_len合理、Extra无Using filesort/Using temporary

⏱️ 60 秒极简表达

⏳
60 秒内说清
看EXPLAIN结果，三字段最关键。第一看type：从const(主键等值，最优)→eq_ref(关联表唯一匹配)→ref(普通索引等值)→range(范围扫描)→index(全索引扫)→ALL(全表扫，最差)，至少要到range级别。第二看key_len：索引用了几列、每列用了几字节。key_len=4+2+1表示三列联合索引全用到了（int 4+ varchar(1) utf8mb4 5+ tinyint 1...具体跟字段类型相关）。第三看Extra：Using index=好(覆盖索引)、Using where=正常、Using filesort=危险(需要额外排序)、Using temporary=严重(建临时表)。

⚠️ 千万别这么回答

❌ 错误表达

  • "type=index就是走索引了，说明没问题" — type=index是全索引扫描，和全表扫描一样慢，只是扫的是索引树
  • "Extra=Using where说明有问题" — Using where只是表示有过滤条件，完全正常
  • "possible_keys有值就说明走索引了" — possible_keys只是候选，key字段才是实际使用的索引
  • "rows就是实际返回行数" — rows是优化器的估计值，可能差几个数量级

✅ 正确表达

  • "EXPLAIN的type从system > const > eq_ref > ref > range > index > ALL逐级变差，优化目标是至少达到range"
  • "Using filesort不等于一定用文件排序，很多时候在内存排序；但它是排序不走索引的信号，需要优化"
  • "key_len反映索引使用情况：联合索引用到的列越多，key_len越大。key_len比定义的总长小=部分列没用到"
  • "filtered字段(5.7+)很重要：rows × filtered% = 最终返回行数，值很小说明回表过滤掉大量数据，可能需要调整索引"

🌲 追问树

EXPLAIN 核心解读
├─ → type字段的完整等级？
│ └─ system(系统表1行) > const(主键/唯一索引等值，1行) > eq_ref(关联表唯一匹配，1对1) > ref(非唯一索引等值) > range(范围扫描，>/BETWEEN/IN) > index(全索引扫) > ALL(全表扫)
├─ → key_len怎么算？
│ └─ int=4字节；bigint=8；varchar(n) utf8mb4=n×4+2(变长标记)；char(n) utf8mb4=n×4；datetime=5(5.6.4+)。可为NULL+1字节
├─ → Extra关键值含义？
│ └─ Using index=覆盖索引(最优)；Using index condition=ICP索引下推；Using where=有WHERE过滤；Using filesort=额外排序；Using temporary=临时表；Using join buffer=BNL关联
├─ → possible_keys vs key？
│ └─ possible_keys=候选索引列表（优化器考虑过的）。key=最终选择的索引。possible_keys有值但key=NULL→索引失效
├─ → 如何判断是否走索引？
│ └─ type不ALL/index=走了索引。key有值=用了具体索引。key_len和索引定义吻合=充分利用。rows远大于实际返回→有过滤
├─ → filtered字段怎么看？
│ └─ 表示存储引擎返回的行中，满足WHERE条件的百分比。filtered=10%→引擎返回100行只有10行满足条件→回表浪费90行→考虑加联合索引
└─ → EXPLAIN ANALYZE 和 EXPLAIN 区别？
└─ EXPLAIN不执行SQL，给估计值。EXPLAIN ANALYZE(8.0.18+)真实执行SQL，统计每步实际耗时和行数，但不能用于INSERT/UPDATE/DELETE
🧠 记忆钩子
EXPLAIN就是SQL的体检报告：type=各项指标（好/中/差），rows=预估要处理多少行（工作量预估），Extra=医生的特别提示（有异常一定要处理），key=用了什么药（索引），filtered=有效率
7级 — type从system到ALL | ≥range — 可接受的type底线 | Using index — 最优Extra(覆盖索引) | filesort — 需要优化的信号
