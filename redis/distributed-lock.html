#8 分布式锁实现

📚 难度：高级 ⏱️ 建议作答：4-5 分钟 🔖 标签：分布式锁 / SETNX / RedLock / Lua
💡 一句话结论
单Redis分布式锁：SET key value NX PX ttl（原子加锁+设超时） + Lua脚本释放（判断value防误删）。Redisson看门狗自动续期解决业务超时问题。多节点RedLock（向多数独立Redis实例成功加锁）提供更高可靠性，但有学术争议（Martin Kleppmann批判），一般不建议使用，更推荐ZooKeeper方案或单Redis+合理超时。
🏮 生活类比：公共厕所门锁
把技术概念映射到「上厕所」的日常场景，秒懂分布式锁
🚪 SET NX PX = 关门+定时器

你走到厕所门口，如果门开着（锁不存在），你就进去、锁门、同时设定闹钟（NX + PX）。注意：关门和设定闹钟必须是一口气完成的原子操作——不能先关门再调闹钟，否则万一你关门后晕倒了，闹钟没设，厕所永远被锁。

🔑 Lua 释放 = 确认身份再开门

你上完厕所出来开门前，先确认「是我自己锁的门」（判断 client_id 是否匹配）。如果不确认身份直接开门——万一你上太久闹钟响了、别人锁门进去了，你把别人的门打开了，那就尴尬了。

🐕 看门狗 = 自动续时

你在厕所里待着，看门狗每隔一段时间就喊一嗓子：「你还在吗？在的话我再给你续 30 秒！」防止闹钟在你还没完事时就响了（看门狗自动续期）。你出来后锁自动释放，看门狗下班。

🗺️ 7 步回答模板

是什么（定义）

分布式锁是控制分布式系统中多个进程（或线程）对共享资源互斥访问的机制。Redis分布式锁利用Redis的单线程+原子命令特性，用SETNX（SET if Not eXists）实现抢锁，用Lua脚本保证释放锁的原子性。

为什么需要（应用场景）

① 避免重复执行（如定时任务在多实例部署时只执行一次）。② 库存扣减防超卖（虽然更推荐用Redis原子操作或Lua）。③ 分布式ID生成器避免冲突。④ 资源竞争（如多个实例同时操作同一文件）。

解决什么问题

单机锁（synchronized/Lock）无法跨JVM，分布式锁提供了跨进程/跨主机的互斥机制。关键挑战：① 互斥性（同一时刻只有一个客户端持有锁）② 防死锁（持有锁的客户端崩溃后锁能被释放）③ 防误删（A不能释放B持有的锁）④ 高可用（锁服务本身不能单点故障）。

原理（单Redis实现）

加锁（正确姿势）：SET lock_key client_id NX PX 30000 — 原子操作：只有当Key不存在时才能SET成功，同时设置30秒过期。错误做法：先SETNX再EXPIRE（非原子，如果SETNX之后进程崩溃，锁永远不会过期→死锁）。

释放锁（Lua脚本）：

if redis.call("get", KEYS[1]) == ARGV[1] then
  return redis.call("del", KEYS[1])
else
  return 0
end

必须用Lua保证"判断+删除"原子性。错误做法：先GET判断value再DEL——中间可能被其他线程插入，导致释放了别人的锁。

锁续期（看门狗Watchdog）：Redisson的看门狗每10秒检查锁是否还被当前线程持有，如果是则自动延长锁的过期时间（默认30秒）。防止业务执行时间超过锁TTL导致锁被提前释放。

缺点与权衡

① 单Redis方案有单点故障风险（Redis挂了锁服务不可用）。② 主从架构下主节点宕机后锁可能丢失（异步复制，从节点升主时可能缺少锁记录）。③ 业务执行超时时即使有看门狗也可能需要调整TTL策略。④ 性能开销：每次加锁/释放都需要一次Redis网络往返（1-2ms）。

RedLock及其争议

RedLock算法（推荐跳过此方案）：向N个独立Redis实例（非主从，各为单点）顺序尝试加锁，如果超过半数（N/2+1）成功且在锁有效期TTL内完成，则认为获取锁成功。实际有效时间 = TTL - 获取锁耗时 - 时钟漂移补偿。

Martin Kleppmann的批判：① RedLock依赖时钟同步假设，但分布式系统中时钟不可靠（时钟跳跃、GC停顿）。② 客户端获取锁后GC停顿超TTL，锁自动过期，另一个客户端获取锁，造成两个客户端同时持有锁。③ fencing token方案更可靠：锁服务返回递增token，资源层检查token拒绝旧请求。④ 结论：如果只是效率优化（减少重复工作），单Redis够了；如果需要绝对正确性，用ZooKeeper（CP系统）。

落地建议

实用选型指南：① 99%场景：用Redisson的RLock，自动处理看门狗+可重入+公平锁。② 需要绝对正确性（如金融账务）：放弃Redis，用ZooKeeper临时顺序节点+Watcher。③ 高并发+允许偶发冲突：单Redis+合理TTL+监控。④ 业务执行时间不确定：看门狗 + 合理的锁超时上限。⑤ 关键代码段：用fencing token机制在业务层做幂等防护。

⚡
面试官："Redis怎么实现分布式锁？"
"加锁用SET key value NX PX 30000，一个原子命令同时完成抢锁和设超时。释放用Lua脚本先判断value是不是自己设的client_id，是才DEL，防止误删别人的锁。Redisson封装了看门狗自动续期，防止业务执行超时锁提前释放。至于RedLock多Redis实例方案有争议——依赖时钟同步，可能GC停顿导致两个客户端同时持有锁。一般用Redisson就够了，绝对正确性场景用ZooKeeper。"

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"SETNX + EXPIRE就行了"

"释放锁直接DEL就行"

"RedLock就是最好的分布式锁方案"

"分布式锁就是synchronized用Redis实现"

✅ 正确表达

"SETNX + EXPIRE分两步执行不是原子操作！如果SETNX成功后进程崩溃，EXPIRE没执行，锁永远不会释放——死锁。必须用 SET key value NX PX ttl 一条命令"

"直接DEL会误删别人的锁：A的锁超时释放→B加锁→A执行完去DEL→B的锁被A误删。必须用Lua脚本判断value确认是自己加的锁才DEL"

"RedLock有严重学术争议，Martin Kleppmann专门写文章批判过。依赖时钟同步假设在分布式系统中不可靠。面试表达：了解RedLock但一般不用"

"分布式锁的挑战远超单机锁：网络分区、时钟漂移、GC停顿、进程崩溃都要考虑。单机synchronized只保护JVM内互斥，跨进程完全无效"

🌲 追问树

分布式锁
├─ 单Redis实现
├─ 正确加锁：SET key client_uuid NX PX 30000（一条命令，原子性）
├─ 正确释放：Lua脚本 if get(key)==client_uuid then del(key) end（判断+删除原子性）
├─ client_uuid建议：UUID或机器ID+线程ID，保证唯一性
└─ PX时间设置：根据业务最大执行时间，留20-50%余量
├─ Redisson看门狗
├─ 原理：加锁时不设TTL（或看门狗覆盖），后台定时任务每10秒续期
├─ 触发：锁持有期间自动续期，锁释放时停止续期
├─ 局限：如果Redisson客户端崩溃/网络断开，续期停止，锁最多30秒自动释放
└─ 可重入：Redisson用hash结构实现（field=线程ID，value=重入次数）
├─ RedLock（了解即可）
├─ 要求N个独立Redis实例（N≥3奇数，不能是主从）
├─ 客户端顺序向每个实例加锁（SET NX PX），超时时间远小于锁TTL
├─ 超过N/2+1实例成功→获得锁；否则释放已获得的锁重试
└─ 缺点：依赖时钟同步、GC停顿导致锁超时、复杂度高
├─ ZooKeeper方案对比
├─ 临时顺序节点：每个客户端创建znode，序号最小的获得锁
├─ Watcher机制：前一个节点删除时通知下一个客户端
├─ 优势：CP系统（一致性强）、自带会话超时（客户端崩溃自动释放）
└─ 劣势：性能不如Redis（ZAB协议开销）、运维更复杂
└─ 选型决策树
├─ 性能敏感+非绝对正确 → Redisson RLock（单Redis）
├─ 绝对正确（金融/账务）→ ZooKeeper临时顺序节点
├─ 已有etcd/consul基础设施 → 用它们的分布式锁API
└─ 简单场景（防重复执行定时任务）→ SET NX PX + Lua释放 足够
🧠 记忆钩子
"分布式锁像公共厕所的门锁：SET NX PX = 关门+设定时器防止厕所蹲太久（原子操作，不能先关门再调定时器）。释放锁 = 确认是自己进去的（判断client_id）再开门（Lua原子）。看门狗 = 如果你还在里面（业务还在执行），自动续时防止被赶出去。RedLock = 同时在3个厕所锁门，多数锁上才放心——但万一你蹲着蹲着睡了一觉（GC停顿），定时器到了别人也能进去。"
