#3 缓存雪崩 (Cache Avalanche)

📚 难度：中高级 ⏱️ 建议作答：4-5 分钟 🔖 标签：缓存问题 / 高可用 / 限流熔断
💡 一句话结论
缓存雪崩是大量Key同时过期或Redis宕机，瞬间所有流量打到数据库导致系统崩溃。核心解法：TTL加随机值打散过期时间 + 多级缓存（本地Caffeine + Redis） + 限流熔断（Sentinel/Hystrix） + Redis高可用（哨兵/Cluster）。
🏮 生活类比：雪山雪崩
把技术概念映射到「雪崩」的自然现象，秒懂缓存雪崩
❄️ 一片雪滑落

一片雪花从山坡滑落——没人会在意。一个Key过期，缓存miss一次，数据库完全扛得住。

🏔️ 所有雪同时滑落

但如果整个山坡的积雪在同一秒全部滑落——那就是雪崩。大量Key集体过期，所有请求同时砸向数据库，瞬时 QPS 可能从 5 万飙升到 50 万，数据库直接瘫痪。

⏰ 让雪分批融化

给每堆积雪设置不同的融化时间（TTL 随机化）——有的 1 小时后化，有的 1 小时 5 分钟后化——错开时间，永远不会同时滑落。

🗺️ 7 步回答模板

是什么（定义）

缓存雪崩指大量缓存Key在同一时刻（或极短时间窗口内）集体过期，或者Redis服务宕机，导致海量请求直接打到数据库，造成数据库压力骤增甚至崩溃。与穿透和击穿不同，雪崩影响范围更广（全量或大批Key）。

为什么发生（根因）

① 大量Key用了相同的TTL，到期时间高度集中（如统一设了1小时过期）→ 集体过期。② 缓存预热脚本故障，大量数据同时加载 → 过期时间自然就趋同。③ Redis集群宕机或网络分区 → 缓存层完全不可用。④ 缓存实例重启导致数据全量丢失。

解决什么问题（危害）

雪崩的杀伤力最大：① 数据库瞬间承受平时数十倍的QPS → ② 连接池耗尽/慢查询堆积 → ③ 整个服务不可用 → ④ 如果没有熔断降级，调用链路层层超时，最终全链路崩溃。是缓存三大问题中最严重的一个。

原理（方案机制）

① TTL随机化：给每个Key的过期时间增加一个随机偏移量（如基础TTL + random(0,300)秒），让过期时间分布在不同时刻。
② 多级缓存：本地缓存（Caffeine/Guava Cache）作为第一道防线，Redis作为第二道。本地缓存设置了不同的TTL，即使Redis全挂，本地还能扛一段时间。
③ 限流熔断：使用Sentinel或Hystrix对数据库访问做限流和熔断。当错误率达到阈值时快速失败，防止级联故障。
④ Redis高可用：哨兵模式实现主从自动切换，Cluster模式实现数据分片+故障转移。

缺点与权衡

TTL随机化：只能缓解不能根治，极端情况仍可能集中。多级缓存：数据一致性问题——本地缓存更新延迟比Redis大，需要合理的缓存更新策略。限流熔断：被限流的请求会直接失败，影响用户体验，需要配合降级方案。

如何解决缺点

多级缓存一致性 → 用Redis Pub/Sub或Canal监听binlog做本地缓存主动失效。限流熔断影响用户体验 → 降级数据（默认值、静态兜底页、最后成功的缓存数据）。TTL随机化不足 → 结合后台定时刷新任务，在低峰期预热热点Key。

落地建议

生产环境多层防御体系：L1（Nginx本地缓存/降级页）→ L2（Caffeine本地缓存，TTL 10-30s）→ L3（Redis集群，哨兵/Cluster）→ L4（数据库，前面有Sentinel限流）。监控：缓存命中率、过期Key分布热力图、数据库QPS告警。

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"雪崩就是缓存挂了，重启就行"

"TTL随机化就够了"

"限流就行了，不需要降级"

"主从复制就能防雪崩"

✅ 正确表达

"雪崩分两种场景：① 大量Key同时过期（TTL趋同导致）② Redis服务宕机（单点故障）。重启虽然能恢复，但冷启动期间数据库压力巨大"

"TTL随机化只能缓解过期集中问题，不能解决Redis宕机导致的雪崩。必须多级缓存+限流+高可用组合"

"限流保证数据库不被打死，降级保证用户体验不完全丢失。二者配合才是完整方案——限流是防，降级是兜"

"主从复制解决数据冗余，但故障切换需要哨兵或Cluster。只有主从没有自动切换，主挂了照样雪崩"

🌲 追问树

缓存雪崩
├─ 为什么大量同步过期？
├─ 批量加载数据时统一设了相同TTL
├─ 缓存预热脚本一次性写入没有随机化TTL
└─ 根本原因：缺乏TTL差异化设计
├─ TTL随机化怎么做？
├─ 公式：TTL = base_TTL + random(0, max_offset)
├─ 举例：1小时基础 + 随机0-600秒 → 过期分布在60-70分钟内
└─ 注意：偏移量不能太小（至少base_TTL的10-20%）
├─ 多级缓存架构？
├─ L1 本地缓存（Caffeine）：TTL 10-30秒，JVM内，纳秒级访问
├─ L2 分布式缓存（Redis）：TTL 30-60分钟，分片/哨兵
├─ L3 数据库（MySQL）：分库分表+读写分离
└─ 一致性保证：本地缓存用Redis Pub/Sub或Canal做主动失效
├─ 限流熔断方案？
├─ Sentinel：资源名（DB查询）+ 流控规则（QPS阈值）+ 降级规则（慢调用/异常比例）
├─ 熔断状态：Closed → Open（熔断） → Half-Open（探测） → Closed
└─ 降级返回值：默认对象、空列表、缓存中最后一条有效数据
└─ Redis高可用
├─ 哨兵模式：≥3个哨兵，自动故障转移，适用于读多写少
├─ Cluster模式：16384槽分片+主从自动切换，适用于大数据量+高并发
└─ 是否需要持久化：需要！宕机后从RDB/AOF恢复，否则冷启动等于雪崩
🧠 记忆钩子
"缓存雪崩像真雪崩：一片雪（一个Key过期）没问题，所有雪同时滑落（大量Key同时过期）就灾难了。解决方法：让雪分批融化（TTL随机），准备第二道防线（多级缓存），设路障减缓冲击（限流熔断），保证山体本身不塌（Redis高可用）。"
