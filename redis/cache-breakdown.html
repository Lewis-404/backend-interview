#2 缓存击穿 (Cache Breakdown)

📚 难度：中级 ⏱️ 建议作答：3-4 分钟 🔖 标签：缓存问题 / 互斥锁 / 逻辑过期
💡 一句话结论
缓存击穿是一个热点Key过期瞬间，大量并发请求同时穿透缓存打到数据库。数据本身是存在的（与穿透的本质区别）。核心解法：互斥锁（一个线程查DB回种，其他等待）+ 逻辑过期（缓存永不过期，值里带过期时间，异步刷新）。

🗺️ 7 步回答模板

是什么（定义）

缓存击穿是指某个热点Key在过期的一瞬间，大量并发请求同时穿透缓存，直接打到数据库。与穿透不同，这个Key对应的数据在数据库中是存在的。

为什么发生（根因）

热点Key（如秒杀商品的库存、热门博文的详情）承载着极高的并发读流量。缓存过期后，在第一个请求完成"查DB→回种缓存"的窗口期内，后续请求全部打到数据库，形成瞬时流量尖峰。

解决什么问题（危害）

热点Key通常QPS极高（几千到几十万），集中在1-2秒内同时打库 → 数据库连接池瞬间打满 → 导致整个服务不可用。与穿透不同，击穿是"有效请求在极短时间窗口集中打库"。

原理（方案机制）

方案一：互斥锁（Mutex Lock） — 使用Redis的SETNX实现分布式锁。第一个发现缓存失效的线程获取锁去查DB重建缓存，其他线程等待锁释放后从缓存读取。
方案二：逻辑过期（Logical Expiration） — 缓存永不过期（或TTL设置很长），但在Value中自带逻辑过期时间。线程读取后发现逻辑过期，返回旧值的同时异步起一个线程获取锁去刷新缓存。
方案三：提前预热 — 在Key过期前主动刷新，或在系统启动时/定时任务预热热点数据。

缺点与权衡

互斥锁：会有短暂的等待时间，锁未释放期间请求被阻塞；极端情况下获取锁的线程挂了需要锁超时兜底。逻辑过期：返回的是旧数据（最终一致性），"永不过期"的Key会占用更多内存；实现复杂度更高（需要额外线程池异步刷新）。

如何解决缺点

互斥锁 → PX设置锁超时时间（如10s）+ 获取锁失败不阻塞而是直接返回旧值（降级策略）。逻辑过期 → 用Caffeine本地缓存做第一层（更快） + Redis做第二层；异步线程池做隔离防止OOM。

落地建议

对于一致性要求高的场景（如库存扣减）：互斥锁 + 双检。对于高并发读多写少（如商品详情）：逻辑过期 + 提前预热。Redisson的RLock已封装了看门狗和自动续期，可以直接用。生产环境建议加上热点自动发现（基于访问频率统计）。

⏱️ 60 秒极简表达

⚡
面试官："缓存击穿怎么解决？"
"缓存击穿是热点Key过期的瞬间大量并发打库。两种方案：一是互斥锁——用SETNX对Key加锁，拿到锁的线程查DB回种缓存，其他线程等待或降级返回旧值。二是逻辑过期——缓存不设过期时间，Value里带过期字段，发现过期后返回旧值并异步刷新。互斥锁保证一致性但会阻塞，逻辑过期保证可用性但返回旧数据，根据业务场景选。"

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"击穿就是缓存没了"

"加个synchronized就行"

"逻辑过期就是TTL设大一点"

"互斥锁方案不需要双检"

✅ 正确表达

"击穿的特指热点Key过期瞬间的并发打库，关键是'热点+过期窗口'两个条件同时满足"

"synchronized是本地锁，分布式环境必须用分布式锁（SETNX），因为多个实例各自有各自的JVM锁"

"逻辑过期是在Value中额外存一个时间戳，缓存本身不设或设超大TTL。读到旧数据时异步刷新，不等刷新完成"

"必须双检：拿到锁后再次检查缓存是否已被其他线程重建，避免重复查DB"

🌲 追问树

缓存击穿
├─ 与穿透的区别？
├─ 穿透：数据在数据库中也不存在（无效请求）
└─ 击穿：数据在数据库中存在（有效请求），只是热点Key过期了
├─ 互斥锁方案细节？
├─ SET key value NX PX 10000：原子获取锁+设超时
├─ 双检（Double Check）：获取锁后再次查缓存，可能其他线程已重建
├─ 获取锁失败的处理：① 重试等待（自旋+sleep）② 降级返回旧值 ③ 直接查询走DB限流
└─ 锁超时：获取锁的线程查DB失败/超时，锁自动释放避免死锁
├─ 逻辑过期方案细节？
├─ Value结构：{"data": ..., "expireTime": 1716451200000}
├─ 读到逻辑过期：先返回旧data，然后尝试获取互斥锁异步刷新
├─ 如果获取锁失败：说明已有线程在刷新，直接返回旧值即可
└─ 线程池隔离：异步刷新用独立线程池，避免阻塞主业务线程
└─ 对比：互斥锁 vs 逻辑过期
├─ 一致性：互斥锁（强一致）> 逻辑过期（最终一致）
├─ 可用性：逻辑过期（完全不阻塞）> 互斥锁（获取锁时可能等待）
├─ 复杂度：互斥锁（简单）< 逻辑过期（需异步线程池+过期时间戳）
└─ 选型：金融/库存→互斥锁；资讯/商品详情→逻辑过期
🧠 记忆钩子
"缓存击穿像超市鸡蛋打折：折扣牌（热点Key）被撕掉的一瞬间，所有人涌到仓库（数据库）去抢。互斥锁 = 留一个人补库存（锁门去仓库取货），其他人在门口排队等。逻辑过期 = 先用旧鸡蛋（旧数据）顶着，同时派个人悄悄去补货（异步刷新）。"
