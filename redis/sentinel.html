#6 哨兵模式详解 (Sentinel)

📚 难度：高级 ⏱️ 建议作答：4-5 分钟 🔖 标签：高可用 / 哨兵 / 故障转移
💡 一句话结论
Sentinel是Redis高可用方案的核心组件，多哨兵（≥3个奇数）监控主从节点，通过SDOWN→ODOWN→Leader选举→故障转移四步完成主节点自动切换。核心机制：超过quorum个哨兵认为主观下线后形成客观下线，随后用Raft简化版选举Leader Sentinel执行故障转移。
🏮 生活类比：小区保安团
把技术概念映射到「小区保安」的日常场景，秒懂哨兵模式
👀 一个保安说「人没了」

保安小王发现业主老张三天没出门（PING 超时），报告说“老张可能不在人世了”——但这只是小王的个人判断（主观下线 SDOWN），小区不能凭一个人的话就发讣告。

🗳️ 过半保安投票确认

其他保安也去敲门确认，结果老张真的没回应。超过半数（quorum）的保安都确认后，小区才正式公告“老张去世”（客观下线 ODOWN），启动后续程序。

👑 选队长安排后事

保安们投票选出一个队长（Leader 选举），队长负责：① 从老张的儿子中选一个最靠谱的继承家业（最优从节点→新主）② 通知其他家人和邻居（通知客户端）③ 老张如果复活了，自动降级为儿子（旧主变从）。

🗺️ 7 步回答模板

是什么（定义）

Sentinel（哨兵）是一个独立的Redis进程，不存储数据，专门负责监控Redis主从节点的健康状态、自动故障转移、通知客户端。通常部署≥3个哨兵实例（分布在不同的物理机/可用区）组成哨兵集群。

为什么需要（解决了什么问题）

单独的主从复制没有自动故障转移能力——主节点宕机后，需要人工介入将某个从节点提升为主节点，修改其他从节点的主节点配置，通知所有客户端新的主节点地址。Sentinel自动化了这整个过程，实现无人值守的高可用。

原理（四步故障转移流程）

步骤1 — 主观下线（SDOWN）：每个哨兵通过PING命令（每秒一次）检测主节点。如果在down-after-milliseconds（默认30s）内未收到有效回复，该哨兵标记主节点为"主观下线"。注：单个哨兵的判断不具权威性。

步骤2 — 客观下线（ODOWN）：哨兵通过哨兵间的Gossip协议交换信息，当超过quorum个哨兵都认为主节点SDOWN时，主节点被标记为"客观下线"。这是一个分布式共识过程。

步骤3 — Leader选举：所有认为ODOWN的哨兵参与Leader选举，使用Raft协议的简化版：哨兵向其他哨兵发送投票请求（Sentinel is-master-down-by-addr），先到先得，获得多数票（>N/2）的成为Leader。

步骤4 — 故障转移：Leader Sentinel执行故障转移：① 从所有从节点中选择一个最优的（优先级高>复制偏移量大>runId小）② 发送SLAVEOF NO ONE将其提升为主节点 ③ 更新其他从节点的主节点配置 ④ 通知客户端新主节点地址（通过Pub/Sub或客户端sentinel驱动自动感知）。

关键技术细节

哨兵之间如何发现？哨兵通过订阅主节点的__sentinel__:hello频道互相发现，交换各自监控的主从节点信息。
哨兵如何发现从节点？哨兵连接主节点后执行INFO replication命令获取所有从节点信息。
客户端如何感知切换？客户端通过Sentinel API（SENTINEL get-master-addr-by-name）获取当前主节点地址，或订阅+switch-master频道接收通知。Java客户端如Jedis/Lettuce已内置Sentinel自动切换。

缺点与权衡

① 哨兵集群本身可能网络分区（脑裂）—多个哨兵同时认为自己应该领导故障转移。② 异步复制导致切换时可能丢数据——原主节点宕机前未同步到从节点的数据会丢失。③ 哨兵只有监控和切换能力，不负责数据分片——数据量超过单机内存上限无能为力。④ 切换期间短暂不可写（秒级）。

如何解决缺点

脑裂 → 配置合理的quorum（>N/2）+ 奇数个哨兵（3/5/7），分布在多个可用区。数据丢失 → 配置min-replicas-to-write和min-replicas-max-lag（如果从节点太少或延迟太大，主节点拒绝写入）。数据量问题 → 升级到Redis Cluster（分片）。切换不可写 → 配置合理的down-after-milliseconds（太快误判多，太慢恢复慢）。

落地建议

标准部署方案：3台物理机（或3个可用区），每台同时跑一个Redis实例（一主两从）+ 一个Sentinel实例。quorum=2，down-after-milliseconds=5000-10000。生产环境必须配置Sentinel的密码认证（sentinel auth-pass）和客户端密码。监控：哨兵日志、故障转移次数、主从角色变更告警。切记：哨兵模式不解决数据分片，数据量大就用Cluster。

⏱️ 60 秒极简表达

⚡
面试官："哨兵模式怎么工作的？"
"哨兵是一个独立的监控进程，不存业务数据。多个哨兵（≥3）组成集群，通过PING检测主节点健康。单个哨兵发现断连标记主观下线（SDOWN），超过quorum个哨兵都确认后形成客观下线（ODOWN）。接着哨兵之间用Raft简化版选举一个Leader，Leader从所有从节点中选一个最优的（优先级高+数据新）提升为新主节点，然后通知其他从节点和客户端。整个过程自动完成，客户端通过Sentinel API获取新主地址。"

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"哨兵就是一台监控机器"

"主观下线就是挂了"

"哨兵模式支持数据分片"

"哨兵选举和Raft完全一样"

✅ 正确表达

"哨兵必须是集群（≥3个奇数），单哨兵自己挂了就无法故障转移，哨兵本身成为单点故障"

"主观下线（SDOWN）是单个哨兵的判断，可能是网络抖动导致；客观下线（ODOWN）是集群达成共识后的确认，只有ODOWN才触发故障转移"

"哨兵只负责高可用（故障自动切换），不负责数据分片。数据量超单机内存上限需要Redis Cluster"

"哨兵Leader选举借鉴了Raft的思想但做了简化：没有Term概念的强约束，没有日志复制，只用于单次Leader选举"

🌲 追问树

哨兵模式
├─ SDOWN vs ODOWN
├─ SDOWN（Subjective Down）：单哨兵视角，down-after-milliseconds超时未回复
├─ ODOWN（Objective Down）：quorum个哨兵都认为SDOWN，达成集群共识
├─ 触发：只有ODOWN才触发故障转移
└─ 配置陷阱：quorum=1 → 一个哨兵就能ODOWN → 极易误判
├─ quorum设置策略
├─ 原则：quorum > N/2（多数），保证不会有两个阵营同时ODOWN
├─ 3哨兵 → quorum=2；5哨兵 → quorum=3；7哨兵 → quorum=4
├─ quorum只影响ODOWN判断，不影响Leader选举（选举需要>N/2票）
└─ 建议quorum和选举多数保持一致
├─ Leader选举（Raft简化版）
├─ 每个哨兵有一票，先请求先投票（First Come First Served）
├─ 获得≥N/2+1票的哨兵成为Leader
├─ 与Raft区别：无Term持久化，无日志复制协议，仅一次选举
└─ 选举超时：sentinel failover-timeout（默认3分钟），超时后重新选举
├─ 故障转移流程细节
├─ 选主策略：① replica-priority最低 ② 复制偏移量最大（数据最新）③ runId最小
├─ SLAVEOF NO ONE → 等待新主完成角色切换 → CONFIG REWRITE持久化
├─ 其他从节点执行SLAVEOF 新主IP 新主端口
└─ 旧主恢复后自动成为新主的从节点（sentinel会修改其配置）
└─ 防脑裂与防丢数据
├─ min-replicas-to-write 1：至少1个从节点在线才接受写入
├─ min-replicas-max-lag 10：从节点复制延迟超过10秒拒绝写入
├─ 哨兵需分布在不同物理机/可用区（避免机房级故障）
└─ 与Cluster对比：哨兵不解决水平扩展，数据量超内存 → 选Cluster
🧠 记忆钩子
"哨兵像小区保安团（≥3人）：一个人说你死了不行（SDOWN主观），过半数都确认才发讣告（ODOWN客观）。然后保安之间投票选出一个队长（Leader选举，Raft简化），队长去安排后事：选一个最好的继任者（最优从节点），宣布新任业主（提升为主），通知所有相关方（其他从节点和客户端）。旧业主复活后自动降级为租客（变成新主的从节点）。"
