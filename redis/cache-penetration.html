#1 缓存穿透 (Cache Penetration)

📚 难度：中级 ⏱️ 建议作答：3-4 分钟 🔖 标签：缓存问题 / 布隆过滤器
💡 一句话结论
缓存穿透是查询既不在缓存也不在数据库中的数据（通常是恶意构造的不存在Key），请求绕过缓存直接击穿到数据库层。核心解法：布隆过滤器预判Key是否存在 + 缓存空值设短TTL + 入口参数校验拦截非法请求。

🗺️ 7 步回答模板

是什么（定义）

缓存穿透是指查询一个数据库中也不存在的数据。由于缓存中没有，每次请求都会穿透缓存直接打到数据库。典型场景：恶意攻击者用随机ID（如负数、UUID）反复请求。

为什么发生（根因）

缓存层的设计前提是"数据在数据库中存在"——缓存未命中就去查数据库。如果攻击者利用"数据本就不存在"这个特点，缓存永远无法拦截，所有请求都直达DB。

解决什么问题（危害）

① 数据库压力激增（大量无效查询）→ ② 连接池耗尽 → ③ 正常请求被阻塞 → ④ 数据库宕机（雪崩的导火索之一）。对于高并发系统，几万QPS的穿透可以直接拖垮MySQL。

原理（方案机制）

方案一：布隆过滤器。将所有存在的Key哈希映射到位数组中。请求来时先查布隆过滤器，返回"不存在"则直接拒绝。空间效率极高（10亿数据约1.2GB）。
方案二：缓存空值。对查不到的Key也缓存一个Null标记，设置短TTL（30-60秒），后续相同Key在TTL内直接返回空，不穿透。
方案三：参数校验。在入口层（网关/Controller）对请求参数做合法性校验，如ID必须为正整数、格式校验等。

缺点与权衡

布隆过滤器：有误判率（说存在但实际不存在的概率约1%）、不支持删除（或需计数布隆）。缓存空值：占用额外内存、短TTL期间若数据恰好被写入会出现短暂不一致。

如何解决缺点

误判率→调整哈希函数数量和位数组大小（trade-off空间和误判）；不支持删除→升级为计数布隆过滤器（Counting Bloom Filter）；短暂不一致→业务可接受的窗口内不一致，或写入时主动删除空值缓存。

落地建议

生产环境标准组合拳：① Nginx/网关层做参数校验（过滤明显非法请求）→ ② 布隆过滤器做Key存在性预判（RedisBloom模块或Guava本地布隆）→ ③ 缓存空值作兜底（短TTL，如30s）→ ④ 监控告警：穿透率突增触发安全策略。

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"缓存穿透就是缓存过期了，请求打到数据库"

"布隆过滤器很简单，就一个bit数组"

"用互斥锁解决缓存穿透"

"缓存穿透和击穿是一回事"

✅ 正确表达

"缓存穿透是查不存在的数据（与过期无关），请求穿透缓存直接打库"

"布隆过滤器使用多个哈希函数映射到bit数组，保证空间效率，误判率可控（公式 k=(m/n)*ln2）"

"互斥锁是解决击穿的方案，穿透不需要锁——因为数据本来就不存在，锁了也没用"

"穿透（数据不存在）、击穿（热点Key过期）、雪崩（大量Key同时过期），三者根因和方案完全不同"

🌲 追问树

缓存穿透
├─ 与击穿、雪崩的区别？
├─ 穿透：数据不存在 ⇄ 击穿：热点Key过期 ⇄ 雪崩：大量Key同时过期
└─ 穿透的请求都是"无效请求"，击穿和雪崩是"有效请求"但缓存miss
├─ 布隆过滤器原理？
├─ 初始化：m位数组 + k个哈希函数 → 插入时k个位置全部置1
├─ 查询时k个位置全为1则"可能存在"，有0则"一定不存在"
└─ 误判率公式：p ≈ (1-e^(-kn/m))^k，最优k=(m/n)*ln2 ≈ 0.7*m/n
├─ 缓存空值方案？
├─ SET key "" EX 30：空值TTL建议30-60秒
├─ 风险：大量不同Key的空值会占内存 → 设置最大内存上限
└─ 不一致：空值TTL期间数据被写入 → 写入时主动DEL空值Key
├─ 布隆过滤器不支持删除怎么办？
├─ Counting Bloom Filter：把bit位换成计数器（int），插入+1删除-1
├─ 代价：空间膨胀4-32倍（取决于计数器位数）
└─ 定时重建：用全量数据重建一个新的布隆过滤器然后替换
└─ 场景与落地
├─ 恶意攻击/爬虫：参数校验+布隆+空值 三层防御
├─ 布隆过滤器放哪：本地Guava（低延迟）或RedisBloom（集中管理）
└─ 监控指标：穿透率 = 查DB返回空次数/总请求数，超过阈值告警
