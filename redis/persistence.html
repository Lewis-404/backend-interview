#4 持久化：RDB vs AOF

📚 难度：中高级 ⏱️ 建议作答：4-5 分钟 🔖 标签：持久化 / RDB / AOF / 混合持久化
💡 一句话结论
RDB是快照持久化（fork子进程+COW写时复制，恢复快但会丢数据），AOF是追加写命令日志（数据安全但恢复慢文件大）。Redis 4.0+引入混合持久化：RDB快照 + AOF增量日志，兼顾恢复速度和数据安全，是生产环境首选。
🏮 生活类比：拍照 vs 录像
把技术概念映射到「记录生活」的场景，秒懂 RDB 和 AOF
📸 RDB = 定时拍照

你每隔一小时拍一张全家福（快照）。照片清晰、体积小、翻看快。但两次拍照之间的精彩瞬间——比如宝宝第一次走路——没拍到就永远丢了。

🎥 AOF = 全程录像

摄像机从进门就开始录（append），每一秒的画面都记录在案。任何细节都不会丢，但录像文件巨大，想找到某个瞬间得从头看到尾（恢复慢）。

📸+🎥 混合持久化

先拍一张全家福（RDB 快照），然后摄像机继续录后面发生的事（AOF 增量）。想看回放？从照片开始看，再看后续录像——又快又全。这就是 Redis 4.0 的混合持久化。

🗺️ 7 步回答模板

是什么（定义）

Redis是内存数据库，数据都在内存中，断电即丢失。持久化就是把内存数据保存到磁盘。Redis提供两种持久化机制：RDB（Redis Database Backup，全量快照）和AOF（Append Only File，增量命令日志）。

为什么需要（必要性）

① 灾难恢复：宕机或重启后恢复数据。② 主从全量同步的基础：主节点生成的RDB文件传给从节点。③ 冷热数据分离：内存只存热数据，冷数据从磁盘恢复。④ 数据备份和迁移。

解决什么问题

RDB解决：快速恢复（直接加载二进制快照，比逐条回放AOF快）。AOF解决：数据安全性（每一条写命令都记录，最多丢1秒数据甚至不丢）。混合持久化解决：恢复速度和数据安全的trade-off。

原理（RDB详解）

触发方式：save（阻塞主线程，生产禁用）/ bgsave（fork子进程后台执行）。
bgsave流程：① 主线程fork子进程 ② 子进程共享父进程的内存页（COW写时复制）③ 子进程遍历内存数据写入RDB文件 ④ 期间主线程继续处理请求，修改数据时触发COW复制内存页。
配置：save 900 1（900秒内1次修改触发）/ save 300 10 / save 60 10000。

AOF原理：每条写命令追加到AOF文件末尾（文本协议）。
fsync策略：always（每条命令都fsync，最安全最慢）/ everysec（每秒一次，默认，最多丢1秒数据）/ no（由操作系统决定，可能丢30秒数据）。
AOF重写：fork子进程分析AOF，将多条对同一Key的操作合并为一条（如lpush list a; lpush list b → lpush list a b），压缩文件体积。

缺点与权衡

RDB缺点：① 两次快照之间宕机会丢失数据 ② fork子进程时内存翻倍风险（COW情况下父进程大量写入导致复制整个内存）③ bgsave期间写操作变慢（COW开销）。
AOF缺点：① 文件比RDB大（文本协议，同样数据集AOF通常比RDB大数倍）② 恢复速度慢（需逐条回放命令）③ fsync always严重影响性能 ④ AOF重写期间fork也会占用资源。

如何解决缺点

RDB丢失问题 → 开启AOF补充 or 使用混合持久化（4.0+）。AOF文件大 → AOF重写自动压缩 + 混合持久化将RDB二进制写入AOF文件前半部分。fork内存问题 → 控制单实例内存（建议<10GB），或使用无fork持久化方案（如定时SAVE+多实例）。

落地建议

推荐配置（4.0+）：开启混合持久化（aof-use-rdb-preamble yes）+ AOF everysec。这样AOF文件前半部分是RDB二进制快照（恢复快），后半部分是增量AOF日志（数据安全）。
极端场景：纯缓存（允许数据丢失）→ 关闭持久化，性能最佳。金融/账务 → AOF always + 定期bgsave备份。注意监控：最近一次bgsave状态、AOF文件大小、fork耗时。

⏱️ 60 秒极简表达

⚡
面试官："RDB和AOF有什么区别？"
"RDB是定时全量快照，fork子进程通过COW写时复制把内存数据dump到磁盘，二进制格式，恢复快但可能丢数据（两次快照之间）。AOF是每条写命令追加到日志文件，支持三种fsync策略：always不丢数据但最慢，everysec最多丢1秒，no由OS决定。AOF文件大会自动重写压缩。Redis 4.0后推荐混合持久化——AOF文件前半是RDB快照后半是增量日志，兼顾速度和安全性。"

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"RDB就是AOF的压缩版"

"bgsave不会影响性能"

"AOF always模式生产环境常用"

"fork子进程不需要考虑内存"

✅ 正确表达

"RDB是内存快照的二进制dump，AOF是写命令的文本日志，两者格式和原理完全不同，不是谁压缩谁的关系"

"bgsave虽不阻塞主线程，但fork操作本身会短暂卡顿（大内存下fork耗时数十到数百毫秒），且COW期间父进程写操作需要复制内存页，有性能开销"

"AOF always每条命令都fsync，性能极差（QPS可能从10万跌到几千），只有金融等极端场景使用。生产99%用everysec"

"fork子进程时父子共享内存页，如果父进程大量写入触发COW复制，内存占用可能翻倍，需预留内存余量"

🌲 追问树

持久化
├─ RDB深入
├─ bgsave流程：fork → COW共享 → 子进程写RDB → 父进程继续服务
├─ COW原理：操作系统层面，父进程写时复制内存页（4KB粒度），子进程始终看到fork时刻的快照
├─ save vs bgsave：save阻塞主线程（生产禁用），bgsave fork子进程（生产用）
└─ 触发方式：手动SAVE/BGSAVE + 配置自动触发 + 主从全量同步 + shutdown save
├─ AOF深入
├─ 写回策略：always（每次事件循环都fsync）→ everysec（每秒刷盘一次）→ no（不主动fsync）
├─ AOF rewrite流程：fork子进程 → 子进程写新AOF → 期间新命令写入AOF重写缓冲区 → 子进程完成后父进程追加缓冲区内容
├─ rewrite触发：auto-aof-rewrite-percentage 100（增长100%）+ auto-aof-rewrite-min-size 64mb
└─ AOF文件损坏修复：redis-check-aof --fix 工具
├─ 混合持久化（4.0+）
├─ 配置：aof-use-rdb-preamble yes
├─ AOF文件结构：[RDB二进制快照][AOF增量日志]
├─ rewrite时将当前内存数据生成RDB格式写在AOF文件开头
└─ 优势：RDB的快速恢复 + AOF的少量丢失，最佳平衡
└─ 选择策略
├─ 数据安全优先 → 混合持久化 + AOF everysec
├─ 性能优先（允许丢数据）→ RDB only 或关闭持久化
├─ 极端安全 → AOF always + 定期bgsave备份
└─ 内存控制 → 单实例≤10GB，监控fork耗时
🧠 记忆钩子
"RDB = 拍照片（定时快照，快但可能错过瞬间），AOF = 开录像机（每条命令都记录，全但文件大回放慢），混合持久化 = 先拍一张照片，然后继续录像，下次看时从照片开始回放录像。COW（Copy-On-Write）= 父子共享一本笔记（内存），孩子抄笔记时（bgsave），爸爸只在要修改的那页复印一张新的再改（COW）。"
