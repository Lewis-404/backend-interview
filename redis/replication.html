#5 主从复制原理 (Replication)

📚 难度：中高级 ⏱️ 建议作答：4-5 分钟 🔖 标签：高可用 / 主从复制 / PSYNC
💡 一句话结论
Redis主从复制：从节点发送PSYNC + ReplicationID + offset给主节点，主节点判断走全量同步（fork子进程生成RDB+Dump命令缓冲区）还是增量同步（命令传播）。核心是replication buffer（环形缓冲区），从节点断线太久导致buffer被覆盖则回退到全量同步。

🗺️ 7 步回答模板

是什么（定义）

主从复制是Redis实现数据冗余和读写分离的基础机制。主节点（Master）处理写请求并维护数据，从节点（Slave/Replica）通过复制协议同步主节点的数据。一个主节点可以有多个从节点，从节点还可以有自己的从节点（级联复制）。

为什么需要（架构目的）

① 数据冗余：主节点故障时有备份数据。② 读写分离：主写从读，提升读并发量。③ 负载均衡：多个从节点分摊读请求。④ 高可用基础：哨兵和Cluster模式都基于主从复制。⑤ 备份：从节点上做bgsave，不影响主节点性能。

解决什么问题

单机Redis的问题：① 单点故障—数据丢失 ② 读压力集中—性能瓶颈 ③ 备份影响线上服务。主从复制通过多副本解决这些，但注意Redis主从是异步复制，不保证强一致性。

原理（核心流程）

建立连接：从节点执行REPLICAOF命令，与主节点建立TCP连接。
全量同步（Full Resynchronization）：① 从节点发送 PSYNC ? -1（首次同步）② 主节点fork子进程bgsave生成RDB ③ 同时将新写入的命令写入replication buffer（默认1MB环形缓冲区）④ bgsave完成后主节点发送RDB文件给从节点 ⑤ 从节点清空旧数据加载RDB ⑥ 主节点发送缓冲区中累积的命令给从节点。
增量同步（Partial Resynchronization）：① 从节点发送 PSYNC replicationID offset（上次复制的ID和偏移量）② 主节点检查replication buffer中是否还有offset起的数据 ③ 有则发送增量命令 ④ 没有则降级为全量同步。
命令传播：同步完成后，主节点将每条写命令异步发送给所有从节点。

缺点与权衡

① 异步复制导致主从数据不一致（可能存在几百ms的延迟）。② 全量同步时主节点fork+bgsave开销大。③ 从节点故障恢复后若buffer被覆盖只能全量同步（网络带宽开销大）。④ buffer太小导致频繁全量同步（两难：太大占内存，太小兜不住断线恢复）。⑤ 读写分离场景下从节点可能返回旧数据。

如何解决缺点

主从不一致 → 对一致性要求高的读走后主节点；或用Redis 6.0的WAIT命令等待指定数量从节点确认（半同步）。全量同步开销 → 控制数据量（分片到多个主节点）；或使用无盘复制（repl-diskless-sync）。buffer覆盖 → 调大repl-backlog-size（建议：断线时长 × 写入速率 × 2）。

落地建议

核心配置：repl-backlog-size（环形缓冲区大小，建议256MB-1GB）、repl-backlog-ttl（空闲多久释放，建议3600秒）、client-output-buffer-limit slave（防止缓冲区溢出导致从节点被踢）。监控：主从延迟（INFO replication的offset差值）、全量同步次数（避免频繁发生）。高并发写入场景建议设置repl-diskless-sync yes减少磁盘IO。

⏱️ 60 秒极简表达

⚡
面试官："Redis主从复制是怎么工作的？"
"从节点连主后发送PSYNC命令，带上replicationID和offset。首次同步或offset不在主节点buffer中就全量同步：主节点fork子进程bgsave生成RDB，同时新命令写入环形缓冲区，RDB发给从节点加载后再发缓冲区的增量命令。后续同步就增量传播：主节点把每条写命令发给从节点。关键参数是repl-backlog-size，如果太小，从节点断线重连时buffer被覆盖就只能重新全量同步，开销很大。"

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"主从复制是同步的，写主立即可读从"

"全量同步就是把内存数据全发给从"

"buffer越大越好"

"SYNC和PSYNC是一样的"

✅ 正确表达

"Redis主从复制是异步的，主节点执行完写命令就返回给客户端，再异步发给从节点。读写分离场景下从节点数据有"

"全量同步的关键是RDB文件+replication buffer组合：RDB保证基准数据，buffer保证同步期间新写入命令不丢失"

"buffer需要合理设置。太小→频繁全量同步；太大→浪费内存。公式：buffer_size ≈ 写入速率 × 最大可接受断线时间"

"SYNC是Redis 2.8之前的命令，只支持全量同步。PSYNC是2.8+引入的，支持增量同步（基于offset+replicationID），断线重连时先尝试增量而非直接全量"

🌲 追问树

主从复制
├─ 全量同步流程详解
├─ 步骤1：从→主 PSYNC ? -1（首次连接，replicationID为空，offset=-1）
├─ 步骤2：主节点 fork子进程 bgsave，同时所有新命令写入replication buffer
├─ 步骤3：bgsave完成，主发送RDB文件给从节点
├─ 步骤4：从节点清空自身数据，加载RDB文件
└─ 步骤5：主节点将replication buffer中积累的命令发送给从节点，追上主节点
├─ 增量同步条件
├─ 条件1：从节点携带上次复制的主节点replicationID
├─ 条件2：从节点offset在replication buffer有效范围内
├─ 条件3：主节点没有重启（重启生成新replicationID，缓冲区清空）
└─ 不满足任一条件 → 降级为全量同步
├─ replication buffer详解
├─ 数据结构：环形缓冲区（定长数组+写指针+读指针）
├─ 配置：repl-backlog-size（默认1MB，生产建议256MB-1GB）
├─ 覆盖场景：新命令写入覆盖最老数据 → 从节点offset在覆盖区外 → 全量同步
└─ repl-backlog-ttl：多从节点全部断开后，多久释放buffer（默认3600秒）
├─ 无盘复制（Diskless Replication）
├─ 配置：repl-diskless-sync yes
├─ 流程：主节点不再先写磁盘RDB，而是直接通过socket流式发送给从节点
├─ 优势：减少磁盘IO，适用于磁盘慢但网络快的场景（SSD/NVMe无此必要）
└─ 注意：多个从节点同时sync时会排队，可设置repl-diskless-sync-delay（等待其他从节点一起sync）
└─ 复制延迟与数据一致性
├─ 监控指标：master_repl_offset - slave_repl_offset（延迟字节数）
├─ 半同步复制：WAIT numreplicas timeout（阻塞主节点直到N个从节点确认）
├─ 读写分离一致性：对实时性要求高的读往后走主节点
└─ 从节点只读：replica-read-only yes（默认），防止误写到从节点
🧠 记忆钩子
"主从复制像师徒传承：新徒弟来了（首次连接）→ 师父复印整套教材（全量同步：RDB dump）并把复印期间新增的笔记也给他（replication buffer里的增量命令）。之后师父每天传授新知识（增量同步：命令传播）。徒弟请假太久（断线），进度跟不上（环形buffer被覆盖）→ 只能重新复印整套教材（全量同步）。"
