#7 Cluster集群原理

📚 难度：高级 ⏱️ 建议作答：5-6 分钟 🔖 标签：高可用 / Cluster / 分片 / Gossip
💡 一句话结论
Redis Cluster是去中心化分片方案：将16384个哈希槽分布在多个主节点，客户端通过CRC16(key) % 16384路由到对应节点。节点间通过Gossip协议传播拓扑和槽位信息，通过MOVED（永久重定向）和ASK（临时重定向，槽迁移中）处理请求路由。每个主节点可配置从节点实现自动故障转移。

🗺️ 7 步回答模板

是什么（定义）

Redis Cluster是Redis官方提供的分布式解决方案。它通过数据分片（Sharding）将数据分散到多个Redis节点，同时每个主节点可以有从节点保证高可用。与哨兵模式不同，Cluster是去中心化架构——没有单点的哨兵进程，每个节点都是对等的，通过Gossip协议互相通信。

为什么需要（与哨兵对比）

哨兵模式解决高可用但无法解决数据量超单机内存的问题。当数据量达到TB级、QPS达到百万级时，哨兵模式无能为力。Cluster通过分片将数据打散到多个节点，每个节点只存一部分数据，实现水平扩展。官方推荐：总节点数不超过1000个。

解决什么问题

① 数据水平分片（数据量超单机内存上限）。② 写性能水平扩展（多个主节点同时接受写入）。③ 自动故障转移（无需额外哨兵进程，集群内部选举）。④ 高可用+横向扩展一体化方案。

原理（核心机制）

哈希槽（Hash Slot）：总共16384个槽（0-16383）。每个Key通过HASH_SLOT = CRC16(key) % 16384计算所属槽。槽是数据迁移和路由的基本单位。如果Key包含{...}（hash tag），则只用花括号内的部分计算槽，确保相关Key在同一槽（如user:{1001}:profile和user:{1001}:orders）。

请求路由：客户端可连接任意节点。节点计算Key的槽位，如果槽在本节点则处理；否则返回MOVED错误（携带目标节点地址），客户端缓存路由表后重定向。CLUSTER SLOTS命令可一次性获取完整的槽-节点映射。智能客户端（Jedis/Lettuce）会自动缓存路由表和处理MOVED重定向。

槽迁移：使用CLUSTER SETSLOT命令迁移槽。迁移过程中槽在源节点标记为MIGRATING，目标节点标记为IMPORTING。此时请求命中源节点：如果Key还在则处理，已迁移则返回ASK重定向到目标节点。ASK是临时重定向（只对本次请求），客户端不更新路由表。

Gossip协议：每个节点定期（每秒随机选几个节点）发送PING消息交换信息，包括：节点自身信息、已知节点列表（带flag和ping/pong时间）、槽位分布。接收到PONG或超时则标记节点状态。Gossip保证了最终一致性——拓扑信息最终会传播到所有节点。

故障转移：节点间通过Gossip检测故障。超过半数主节点认为某主节点不可达+cluster-node-timeout超时，则标记为FAIL。该主节点的从节点发起选举：向其他主节点请求投票，获得多数票的提升为新主节点，接管原主的槽。

缺点与权衡

① 不支持多Key跨槽操作（如MSET key1 key2必须同槽，否则报错CROSSSLOT）。② 批量操作用hash tag解决了同槽问题但可能导致数据倾斜（热点槽）。③ 网络分区可能导致脑裂（分区两边各自认为自己是主）。④ 槽迁移是同步阻塞的（迁移单个Key时该Key不可用）。⑤ Gossip协议有收敛延迟（秒级），不如集中式元数据精确。⑥ 没有像哨兵一样的独立监控层，故障判断依赖节点间互PING。

如何解决缺点

跨槽操作 → 业务设计时按用户/商户维度聚合数据，使用hash tag分到同槽。数据倾斜 → 监控槽的Key数量分布，必要时手动rebalance。脑裂 → 合理设置cluster-node-timeout（不要太短），配置min-replicas-to-write。槽迁移阻塞 → 使用MIGRATE命令时控制pipeline大小，分批迁移。Gossip延迟 → 合理设置cluster-node-timeout（默认15s，生产可调为10-20s）。

落地建议

标准部署：至少3主3从（6节点），分布在不同物理机/可用区。cluster-node-timeout建议10-15s。核心配置：cluster-enabled yes、cluster-config-file nodes.conf、cluster-require-full-coverage yes（所有槽都有节点才接受写入）。监控：每个节点的槽数分布、节点状态（cluster nodes命令）、迁移进度。客户端选Lettuce（支持Cluster拓扑自动刷新）或Jedis。

⚡
面试官："Cluster是怎么工作的？"
"Cluster把数据分到16384个哈希槽，每个主节点负责一部分槽。Key通过CRC16(key)%16384计算属于哪个槽。客户端连接任意节点，如果Key不在当前节点会收到MOVED重定向（带目标节点地址），智能客户端会自动缓存路由表。节点之间通过Gossip协议交换拓扑信息，主节点挂了其从节点自动发起选举接管槽。槽迁移时用ASK临时重定向。不支持跨槽的多Key操作，需要用hash tag把相关Key分到同一槽。"

⚠️ 面试中注意 — 错误 vs 正确

❌ 错误表达

"Cluster就是哨兵加多了几个Redis"

"MOVED和ASK是一样的"

"16384个槽太少了应该65536"

"Cluster有中心化的元数据节点"

✅ 正确表达

"Cluster和哨兵是完全不同的架构：Cluster是去中心化+数据分片，哨兵是中心化（哨兵集群）+不拆分数据。数据量超单机内存必须Cluster"

"MOVED表示槽已被永久迁移（客户端应更新路由表），ASK表示槽正在迁移中（临时重定向，客户端不更新路由表，下次还走原节点）"

"选16384而不是65536的原因：① 节点间Gossip消息用bitmap表示槽位，16384=2KB，65536=8KB，4倍带宽差距 ② 实际节点数远小于16384（<1000），65536浪费 ③ 16384足够均匀分布"

"Cluster是完全去中心化的：没有配置中心、没有名字服务、没有元数据节点。每个节点都是对等的，通过Gossip协议了解全局"

🌲 追问树

Redis Cluster
├─ 16384哈希槽
├─ 为什么不是65536：Gossip消息头的bitmap为16384/8=2KB vs 65536/8=8KB（4倍带宽）
├─ 实际节点数（<1000）远小于16384，65536纯浪费
├─ CRC16(k) % 16384：CRC16输出0-65535，取模到16384
└─ hash tag：{user:1001} → 括号内部分计算槽，保证同用户的多个Key在同一节点
├─ MOVED vs ASK
├─ MOVED：-MOVED 3999 127.0.0.1:6380 → 槽已永久迁移，客户端更新路由表
├─ ASK：-ASK 3999 127.0.0.1:6380 → 槽迁移中，客户端先发ASKING命令再重试（一次性）
├─ 判断依据：节点维护的槽状态 → MIGRATING（源） / IMPORTING（目标）
└─ ASK流程：① 客户端收到ASK ② 向目标节点发ASKING ③ 目标节点临时处理一次 ④ 下次还走原节点
├─ Gossip协议
├─ 每条PING消息携带：节点自身信息 + 随机几个其他节点信息（gossip entries）
├─ PING/PONG/MEET三种消息类型
├─ 每秒随机选几个节点PING（cluster-node-timeout/2秒未通信的优先）
└─ 故障检测：PING超时 → PFAIL（疑似）→ 超半数主确认 → FAIL（确认下线）
├─ 节点间通信
├─ 每个节点有TCP服务端口（如6379）和Cluster总线端口（10000+服务端口，如16379）
├─ 服务端口：客户端连接；总线端口：节点间Gossip通信（二进制协议）
└─ 防火墙需要同时开放两个端口
└─ 故障转移（去中心化选举）
├─ 检测：PFAIL（单节点判断）→ FAIL（超半数主节点确认）
├─ 从节点发现主节点FAIL后，等待cluster-node-timeout*2后发起选举
├─ 请求所有主节点投票，获得≥N/2+1票提升为新主
├─ 新主接管槽位并向集群广播PONG，旧主恢复后自动成为新主从节点
└─ 与哨兵区别：无独立哨兵进程，选举逻辑内嵌在每个Redis节点中
🧠 记忆钩子
"Cluster像分仓物流系统：16384个货架（槽）分给不同仓库（主节点），货物（Key）按条码（CRC16）自动路由到对应仓库。仓库之间互发传单通报库存变化（Gossip PING/PONG）。货物搬家（槽迁移）期间，原仓库贴条'已搬走'（MOVED），搬运中的货架贴条'正在搬运，去那边看看'（ASK）。某个仓库着火（主节点挂），副仓库长立刻接管（从节点选举升级）。"
