🎯 Context 超时控制与值传递

编号 GO-04 分类：并发控制 难度：⭐⭐⭐ 频率：必问
💡 一句话结论
Context 是 Go 中传递请求范围数据（trace_id 等）和超时/取消信号的机制。采用树状结构向下传递，父 Context 取消时所有子 Context 自动取消，是控制 goroutine 生命周期的最佳实践。
🎬 生活类比 — 多米诺骨牌
推倒第一块，后面全部倒下——这就是 Context 的取消传播
🎲
场景1：推倒第一块(WithCancel) = 遥控引爆
你搭了一排多米诺骨牌，每块骨牌代表一个 goroutine 任务。当你推倒最前面那块（调用 cancel()），所有后面的骨牌（子 Context）依次倒下（自动取消）。一条请求链路上的所有 goroutine，一个 cancel() 全部停掉——干净利落。
⏰
场景2：定时器(WithTimeout) = 设定自动推倒时间
你给第一块骨牌装了个定时触发器，设定"3 秒后自动推倒"。时间一到，骨牌自动倒塌，后面的全部跟着倒。这就是 WithTimeout——请求 3 秒超时，超时后所有下游调用自动取消，不浪费任何资源。
📝
场景3：WithValue = 骨牌上贴便签信息
每块骨牌上贴了一张便签（如 trace_id、user_id），后面的骨牌需要时往回翻就能找到。但不能贴太多——骨牌是用来传递取消信号的，不是用来当快递盒的。业务参数走函数参数，只把请求元数据贴在骨牌上。
👂
场景4：Done() channel = 等骨牌倒的信号
你盯着第一块骨牌，等它倒下的那一刻（<-ctx.Done()）。一旦它倒了，你就知道该收工了——这就是 select + ctx.Done() 实现优雅退出。不倒 = 继续干活，倒了 = 立刻停手。
💣
WithCancel = 遥控引爆
手动调用 cancel()，推倒第一块骨牌，连锁取消所有子任务
⏱️
WithTimeout = 定时炸弹
设定倒计时，时间一到自动引爆，防止请求无限等待
🏷️
WithValue = 便签
骨牌上贴的元数据，沿链查找即可获取，不传业务参数
🔗
树状传播
父→子单向取消，像骨牌传播一样不可逆，子取消不影响父

🗺️ 7 步回答模板

Step 1 — 是什么

Context 是 Go 标准库 context 包提供的一个接口类型，用于在 goroutine 之间传递截止时间（Deadline）、取消信号（Cancel）和请求范围数据（Value）。它是树状结构，每个 Context 都有一个 parent，取消信号从根向下传播。

Step 2 — 为什么需要 Context

在微服务架构中，一个外部 HTTP 请求可能触发多个下游 RPC 调用，每个调用又可能创建多个 goroutine。如果没有统一的超时控制和取消机制：①某个下游超时后上游 goroutine 继续空转浪费资源；②无法在请求链路中传递 trace_id 等元数据；③goroutine 泄漏风险高。Context 正是解决这些问题的标准方案。

Step 3 — 四种创建方式

① context.Background()：根 Context，永不取消，用于 main、init、test 入口；② context.TODO()：不确定用哪个时的占位符；③ context.WithCancel(parent)：返回可手动 cancel 的子 Context；④ context.WithTimeout/WithDeadline(parent, d)：超时/截止时间自动 cancel；⑤ context.WithValue(parent, key, val)：携带键值对数据。

Step 4 — 核心原理

Context 的本质是一个不可变的树状链表：CancelContext 内部有 done channel（closed 表示已取消）和 children map（子 Context 集合）。调用 cancel() 时，先 close(done)，再递归 cancel 所有 children。ValueContext 通过 parent 链向上查找：ctx.Value(key) 沿 parent 链逐级查找直到 root 或找到匹配 key。

Step 5 — 常见错误与陷阱

① Context 存到 struct 中反复使用（应作为函数第一个参数传递）；② WithValue 的 key 用 string（应使用自定义类型，防止不同包 key 冲突）；③ 忘记调用 cancel()（WithTimeout/WithCancel 返回的 cancel 必须 defer cancel()，否则 goroutine 泄漏）；④ 在 select 中直接 <-ctx.Done() 而不处理（应 return ctx.Err()）；⑤ Context 作为值传递时拷贝了不该拷贝的数据。

Step 6 — 最佳实践

① 函数第一个参数：func Do(ctx context.Context, ...)；② 永远 defer cancel()；③ Value key 用未导出的自定义类型（type ctxKey struct{}），避免字符串冲突；④ 只传请求范围数据（trace_id, user_id），不传函数参数；⑤ 使用 context.WithoutCancel()（1.21+）在特定场景中断开取消传播。

Step 7 — 实际落地场景

① HTTP Server：每个请求自动获得带超时的 ctx（r.Context()）；② gRPC 调用链：客户端 ctx 自动传播超时和 metadata 到服务端；③ 数据库查询：db.QueryContext(ctx, sql, args) 支持超时取消；④ 定时任务：ctx, cancel := context.WithTimeout(bg, 30*time.Second)；⑤ 优雅退出：监听 SIGTERM 后 cancel 全局 ctx，所有 goroutine 通过 <-ctx.Done() 退出。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"Context 是 Go 的请求生命周期管理器。核心三功能：超时控制、取消信号、值传递。四种创建：Background 做根、WithCancel 手动取消、WithTimeout 自动超时、WithValue 传元数据。树状结构，parent cancel 后所有 children 的 Done() channel 都会被 close。最佳实践：永远作为函数第一个参数、defer cancel()、Value key 用自定义类型。典型场景：HTTP→gRPC→DB 的请求链路中传递 trace_id 和超时。"

⚠️ 千万别这么回答

❌ 错误表达

  • "Context 就是用来传参数的"（错！主要功能是超时/取消，传值是次要）
  • "context.Background() 可以随便用"（根 context 不应该频繁创建，应该复用）
  • "context.WithValue 的 key 用 string 就行了"（容易跨包冲突，应该用自定义类型）
  • "cancel() 不调也没关系"（会导致 goroutine 泄漏！）
  • "Context 是并发安全的"（Context 是不可变的 immutable，不是并发安全）

✅ 正确表达

  • "Context 核心功能是超时控制和 goroutine 取消，传值是附加能力"
  • "context.Background() 是根 Context，通常只在 main/init/test 中创建一次"
  • "WithValue 的 key 应该是未导出的自定义空 struct，避免字符串 key 冲突"
  • "WithCancel/WithTimeout 返回的 cancel 必须 defer cancel()，否则 goroutine 泄漏"
  • "Context 是不可变的，每个派生操作返回新 Context 而非修改原值"

🌲 追问树

🎯 Context
├─ → 为什么需要？
│ ├─ 解决 goroutine 泄漏（无统一退出机制时阻塞的 goroutine 无法回收）
│ ├─ 请求链路超时控制（上游超时→下游全部取消）
│ └─ 请求元数据透传（trace_id, user_id 在调用链中传播）
├─ → 四种创建方式
│ ├─ Background()：根，永不取消
│ ├─ TODO()：占位符
│ ├─ WithCancel(parent)：手动 cancel
│ ├─ WithTimeout(parent, dur) / WithDeadline(parent, time)：超时自动 cancel
│ └─ WithValue(parent, key, val)：键值对
├─ → 树状结构
│ ├─ 每个 Context 有 parent 指针和 children map
│ ├─ cancel() → close(done channel) → 递归取消 children
│ └─ 子 Context 取消不影响父 Context（单向传播）
├─ → Done() channel 机制
│ ├─ ctx.Done() 返回只读 channel（<-chan struct{}）
│ ├─ cancel 时 close(done), select 中 <-ctx.Done() 立即就绪
│ └─ ctx.Err() 返回取消原因：Canceled 或 DeadlineExceeded
└─ → 最佳实践与反模式
├─ ✅ 函数第一参数：func Do(ctx context.Context, arg string)
├─ ✅ defer cancel() 防止泄漏
├─ ✅ Value key 用自定义类型：type traceIDKey struct{}
├─ ❌ Context 存入 struct 反复复用
└─ ❌ 用 Context 传业务参数（应该走函数参数）
