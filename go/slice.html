🍕 Slice 底层结构与扩容机制

编号 GO-06 分类：数据结构 难度：⭐⭐⭐ 频率：必问
💡 一句话结论
Slice 是一个引用类型，由指针 ptr + 长度 len + 容量 cap 组成。append 扩容策略：cap < 1024 翻倍增长，cap ≥ 1024 增长 25%（约 1.25 倍）。切片操作共享底层数组是常见坑——修改子切片会影响原切片。nil slice（len=0, cap=0, ptr=nil）和 empty slice（len=0, cap=0, ptr 非 nil）语义不同。
🎬 生活类比 — 长颈鹿的身体
Slice 就是长颈鹿：头看哪里、脖子用多长、身体能装多少
🦒
场景1：头(ptr)、脖子(len)、身体(cap)——水面上的鹿
一只长颈鹿在水里：头（ptr）指向水面上某个位置，脖子（len）是从头开始能看到的长度，身体（cap）藏在水下——你看不到但确实存在。你只能使用脖子以上的部分（len），但身体（cap）才是真正的容量。s1[i:j] 就是让另一只鹿把头伸过来看同一具身体。
📈
场景2：append 长高——不够高整个身体换
append 就是让长颈鹿长高。如果身体（cap）还够，脖子直接伸长就行（在原底层数组上追加）。但如果身体不够了——那就得换一具更大的新身体（分配新底层数组），把原来的身体数据搬过去，旧身体丢掉（靠 GC 回收）。这就是扩容：cap<1024 时换 2 倍大的身体，cap≥1024 时换 1.25 倍。
🐘
场景3：共享身体——两只长颈鹿看同一具身体
s2 := s1[1:3] 是什么意思？两只长颈鹿共享同一具水下身体！s2 的头指向 s1 身体的不同位置。危险的是：如果 s2 的脖子还够长（len < cap），s2 append 可能改到 s1 的数据！就像一只鹿吃了某种药，结果两只鹿的身体都变了——因为它们根本是同一具身体。
👀
ptr = 头
指向底层数组的起始位置，决定视图从哪里开始
📏
len = 脖子
当前能用到的长度，只能看到脖子以内的元素
🏊
cap = 身体
藏在水下的真实容量，append 先吃身体，不够了换更大的
🔄
扩容 = 换身体
cap<1024 换 2 倍大身体，cap≥1024 换 1.25 倍，旧身体被 GC

🗺️ 7 步回答模板

Step 1 — 是什么，底层结构

Slice 是 Go 中动态数组，底层是 runtime 的 slice 结构体：
type slice struct { array unsafe.Pointer; len int; cap int }
ptr：指向底层数组的指针；len：当前元素个数；cap：从 ptr 开始到底层数组末尾的容量。Slice 是对底层数组的一个"窗口视图"。

Step 2 — 为什么需要 Slice（vs Array）

Array 长度固定且是类型的一部分（[3]int 和 [5]int 是不同类型），无法动态增长。Slice 提供动态长度的数组视图，可以 append 追加元素、切片操作创建子视图、作为函数参数传递时只复制 header（24 字节），避免了 array 的全量拷贝。

Step 3 — append 扩容策略详解

当 len + 新元素数 > cap 时触发扩容（growslice）：
① 预估新容量：若 old.cap < 1024 → newcap = 2 × old.cap；若 old.cap ≥ 1024 → newcap = old.cap × 1.25（循环直到够用）；
② 内存对齐：根据元素大小向上取整到 Go 内存分配器的大小类（如 8, 16, 32, 48, 64, 80, 96...）；
③ 分配新底层数组并复制旧数据，返回指向新数组的 slice。这也是 append 必须接收返回值的原因。

Step 4 — 共享底层数组的坑

// 经典坑：切片后共享底层数组 s1 := []int{1, 2, 3, 4, 5} s2 := s1[1:3] // s2 = [2,3], len=2, cap=4（cap = s1.cap - 1） s2 = append(s2, 99) // len=3 < cap=4，在底层数组位置 3 写入 99 // s1 变成了 [1,2,3,99,5]！s1 的第四个元素被修改了

Step 5 — nil slice vs empty slice

nil slice：var s []int → len=0, cap=0, ptr=nil。JSON 序列化为 null。可作为函数返回值表示"无结果"。
empty slice：s := []int{} 或 s := make([]int, 0) → len=0, cap=0, ptr 指向 zerobase 地址（非 nil）。JSON 序列化为 []。语义上 nil slice = 不存在，empty slice = 存在但为空。

Step 6 — 避免坑的最佳实践

① 切片操作后用 copy(dst, src) 创建独立副本：s2 := make([]int, len(s1[i:j])); copy(s2, s1[i:j])；② 预分配容量减少扩容：make([]int, 0, expectedLen)；③ 使用 s = append(s, x) 接受返回值，不要用 _ = append(s, x)；④ 函数返回 slice 时注意底层数组的生命周期（大 slice 的引用会阻止 GC 回收整个底层数组）。

Step 7 — 实际落地场景

① 批量数据处理：make([]Record, 0, batchSize) 预分配减少扩容次数；② 字符串处理：用 []byte 操作后转 string() 减少内存分配；③ 函数参数：传 slice header（24 字节）而非 array；④ 栈内存：小 slice 可分配在栈上（逃逸分析决定），大 slice 逃逸到堆。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"Slice 底层是 ptr + len + cap 三个字段的结构体。ptr 指向底层数组，len 是当前长度，cap 是容量。append 扩容时：cap 小于 1024 翻倍，大于等于 1024 增长 1.25 倍，然后内存对齐取整，分配新数组并复制数据。最大坑：切片操作共享底层数组，改子切片会影响原切片。解决办法：用 copy 创建独立副本。nil slice 的 ptr 是 nil，empty slice 的 ptr 是 zerobase 非 nil 地址。"

⚠️ 千万别这么回答

❌ 错误表达

  • "slice 就是动态数组"（没解释底层结构 ptr+len+cap）
  • "slice 和 array 是一样的"（array 长度是类型一部分且固定，slice 是 header 引用）
  • "append 一直翻倍"（cap ≥ 1024 后不是翻倍，是 1.25 倍）
  • "切片操作创建了新数组"（没有！切片操作共享底层数组，这是大坑）
  • "nil slice 和 empty slice 一样"（ptr 不同，JSON 序列化不同，语义不同）

✅ 正确表达

  • "slice 是 header 结构体（ptr+len+cap），底层数组独立存储"
  • "slice 是对数组的窗口视图，array 长度固定是类型一部分"
  • "扩容：cap<1024 → 2×，cap≥1024 → 1.25×，最后内存对齐到 size class"
  • "s1[i:j] 共享底层数组，append 后可能影响原 slice"
  • "nil slice: ptr=nil；empty slice: ptr=&zerobase（非 nil 地址）"

"Slice 像长颈鹿：头（ptr）看到哪里、脖子（len）用多长、身体（cap）能装多少。身体不够用了就找一具新的（扩容+复制），原来的身体丢掉（原底层数组靠 GC 回收）。切片操作就像让另一只长颈鹿把头伸过来看同一具身体——一只吃东西（append），另一只也会受影响。"
