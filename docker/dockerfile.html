🐳 Dockerfile 最佳实践

编号 DO-01 分类：镜像构建 难度：⭐⭐⭐ 频率：高频
💡 一句话结论
Dockerfile 是镜像构建蓝图。最佳实践：多阶段构建（编译+运行分离减小镜像）、利用构建缓存（COPY/ADD 在最后）、.dockerignore 排除无关文件、CMD 运行（可覆盖）vs ENTRYPOINT 入口（不可覆盖）、尽量用非 root 用户。
🎬 生活类比 — 用「做菜」看懂 Dockerfile 全部指令
用生活场景看懂全部概念
🛒
场景1：FROM = 采购食材
做菜前你得先决定去哪买菜——是去精品超市（Alpine ~5MB）还是大卖场（Ubuntu ~70MB）。FROM alpine:3.19 就是选食材来源，食材越精简，最后端上桌的菜越轻巧。
→
🥬
🔪
场景2：RUN = 处理食材
买完菜开始切、洗、腌——RUN apt install 就是处理食材。聪明厨师会把好几步处理合在一起（RUN a && b && c），而不是切一下停一下。处理完记得清理台面（apt clean）。
📦
场景3：COPY = 把菜放到盘子里
COPY . /app 就是把你的食材（代码/配置）从厨房台面搬到盘子里。用 .dockerignore 就是先挑掉烂菜叶（node_modules/.git），别一股脑全倒进去——盘子装脏东西，整道菜就毁了。
🍽️
场景4：CMD vs ENTRYPOINT = 上菜方式
CMD 是「今天推荐的菜」（可被客人换掉——docker run 传参覆盖），ENTRYPOINT 是「本店固定上菜仪式」（不能换），两者组合就是「固定仪式+默认菜品」——比如 ENTRYPOINT ["nginx"] + CMD ["-g", "daemon off;"]。
👨‍🍳
场景5：多阶段构建 = 先备菜再烹饪
大厨做法：第一阶段在后厨备料（FROM golang AS builder）编译二进制，第二阶段在明档区烹饪摆盘（FROM alpine），COPY --from=builder 只把成品端过来。客人看到的只有精致的摆盘（最终镜像 15MB），看不到乱糟糟的后厨（编译工具链）。

🗺️ 7 步回答模板

Step 1 — 是什么

Dockerfile 是一个文本文件，包含一系列指令，每条指令构建一层，最终生成 Docker 镜像。它是镜像的"源代码"，定义了应用运行所需的完整环境。

Step 2 — 为什么需要

保证环境一致性：开发、测试、生产环境完全一致；可版本控制（提交到 Git）；可复用和共享（Docker Hub/私有仓库）；声明式构建，消除"在我机器上能跑"的问题。

Step 3 — 解决什么核心问题

解决"环境不一致导致线上事故"的痛点。传统部署：手动装依赖 → 版本冲突 → 回滚困难。Dockerfile 将环境代码化，实现了 Infrastructure as Code。

Step 4 — 核心原理

每条指令创建一层（UnionFS），层是只读的，只有最顶层（容器层）可写。构建时利用层缓存：如果某层和之前完全一样，直接用缓存跳过。因此 COPY 和 ADD 放在后面（前面是变化少的 RUN apt install），.dockerignore 排除 node_modules 等避免缓存失效。

Step 5 — 缺点/局限

① 层数过多导致镜像臃肿；② 不合理的指令顺序会导致缓存频繁失效；③ ADD 有自动解压 tar 的副作用，容易误用；④ 如果不在最后清理临时文件，镜像会很大；⑤ root 用户运行有安全风险。

Step 6 — 如何解决缺点

① 多阶段构建：第一阶段编译，第二阶段只拷贝二进制；② 合并 RUN 指令（用 && 连接），减少层数；③ 优先用 COPY 而非 ADD；④ RUN 后立即清理 apt/yum 缓存；⑤ USER 1000 切换到非 root 用户运行。

Step 7 — 实际落地

Go 项目典型 Dockerfile：第一阶段 FROM golang:1.21-alpine AS builder 编译二进制；第二阶段 FROM alpine:3.19 只拷贝编译产物 + 配置文件，最终镜像仅 15MB。配合 .dockerignore 排除 vendor/.git/ 等无关文件。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"Dockerfile 是镜像构建蓝图，每条指令一层。最佳实践五条：①多阶段构建——编译和运行分开，最终镜像只放二进制；②利用构建缓存——变化少的放前面，COPY 放最后；③.dockerignore——排除 node_modules/.git 等避免缓存失效；④CMD vs ENTRYPOINT——CMD 可被 docker run 覆盖，ENTRYPOINT 不行，两者常组合用；⑤用非 root 用户——USER 指令切换到普通用户，安全第一。"

⚠️ 千万别这么回答

❌ 错误表达

  • "Dockerfile 就是写一堆指令"（没说每条指令一层，没说缓存）
  • "CMD 和 ENTRYPOINT 一样"（完全不同，一个是默认命令可覆盖，一个是入口不可覆盖）
  • "把所有东西都打到一个镜像里"（臃肿、不安全、违反单一职责）
  • "用 root 跑就行"（安全红线，生产环境大忌）
  • "多阶段构建就是分两个 Dockerfile"（是在一个 Dockerfile 中用多个 FROM）

✅ 正确表达

  • "每条指令一层，利用 UnionFS 层缓存加速构建"
  • "CMD 是默认命令可被 docker run cmd 覆盖；ENTRYPOINT 是固定入口不可覆盖，组合用实现默认参数+固定入口"
  • "多阶段构建：FROM ... AS builder → FROM alpine → COPY --from=builder，最终镜像只有运行时"
  • "USER 1000:1000 非 root 运行，RUN apt clean && rm -rf /var/lib/apt/lists/* 清理缓存"
  • ".dockerignore 排除.git/、node_modules/、*.md 等，避免 COPY 阶段带入脏文件"

🌲 追问树

🐳 Dockerfile
├─ → 指令
│ ├─ FROM（基础镜像）
│ ├─ RUN（执行命令，构建时）
│ ├─ COPY vs ADD（COPY 原样复制；ADD 有 tar 自动解压 + URL 下载）
│ ├─ CMD vs ENTRYPOINT（CMD 可覆盖默认参数；ENTRYPOINT 固定入口）
│ ├─ WORKDIR（设置工作目录，后续指令基于此目录）
│ └─ EXPOSE（声明端口，实际映射靠 -p）
├─ → 多阶段构建
│ ├─ AS 命名阶段（FROM golang AS builder）
│ ├─ COPY --from=builder（从第一阶段拷贝产物）
│ └─ 最终镜像只含运行时依赖（Alpine + 二进制）
├─ → CMD vs ENTRYPOINT 组合
│ ├─ 只有 CMD：docker run 传参覆盖整个 CMD
│ ├─ 只有 ENTRYPOINT：docker run 传参作为 ENTRYPOINT 的参数
│ └─ 组合：ENTRYPOINT 定义程序，CMD 定义默认参数（最佳实践）
├─ → 构建缓存机制
│ ├─ 每层有唯一 ID（基于指令+内容哈希）
│ ├─ 相同指令+相同上下文 → 命中缓存
│ └─ 某层变化 → 该层及之后全部重建
└─ → 镜像瘦身
├─ Alpine 基础镜像（~5MB vs Ubuntu ~70MB）
├─ RUN 后清理 apt 缓存
└─ 多阶段构建只拷贝最终产物
🧠 记忆钩子
"Dockerfile = 菜谱：FROM = 买材料，RUN = 处理食材，COPY = 放菜，CMD = 上菜（可以换菜），ENTRYPOINT = 固定的上菜方式"
