🐳 镜像分层与优化

编号 DO-02 分类：镜像原理 难度：⭐⭐⭐⭐ 频率：高频
💡 一句话结论
Docker 镜像由多层只读层叠加（UnionFS），每层对应 Dockerfile 一条指令。镜像是分层 + 写时复制（COW）实现共享和复用。优化：合并 RUN 减少层数 / 清理临时文件 / .dockerignore / 多阶段构建 / Alpine 基础镜像。

场景1：镜像分层 = 千层蛋糕的每一层
Dockerfile 每条指令就是一层可丽饼皮。底层是基础镜像（Alpine饼底），往上是 RUN（抹奶油）、COPY（撒水果）。全部叠在一起就是一块完整的蛋糕（UnionFS 统一视图）。每层都有唯一编号（SHA256），一样的层（缓存）直接复用，不用重烤。

✂️
场景2：COW 写时复制 = 蛋糕师复刻
容器运行时想改某个共享层里的文件？不是直接改原版蛋糕——而是先把那一小块复制到顶层（容器层），再在顶层改。原版蛋糕纹丝不动，其他容器继续共享。这就是 COW（Copy-on-Write），读完就改的才复制，不动的继续共享。
🗑️
场景3：删除文件不释放空间 = 贴白纸遮住
删除蛋糕上的一颗草莓？不是真的拿走——是在上面贴一张白纸（whiteout 文件）遮住。下面的草莓还在，体积没变小。所以 rm -rf /tmp/bigfile 不会让镜像变小，要在同一层完成「安装+清理」。
🔁
场景4：缓存复用 = 一样的层不重做
今天烤的蛋糕底和昨天一模一样？直接用昨天的，跳过。但如果第三层换了奶油品牌，第三层及之后全部重新烤——这就是为什么 COPY 要放最后：代码天天变，但 apt install 基本不变，放前面能最大化缓存命中。

🗺️ 7 步回答模板

Step 1 — 是什么

Docker 镜像由多个只读层（layer）堆叠而成，底层是基础镜像（如 Alpine），上层是 Dockerfile 中每条指令产生的增量。所有层通过 UnionFS（Union File System）合并成一个统一的文件系统视图。

Step 2 — 为什么需要

分层设计实现了镜像复用和高效传输。多个镜像共享相同的基础层（如都基于 Alpine）时，只存储一份；拉取镜像时只需拉取本地没有的层（增量下载）；构建时利用层缓存加速。

Step 3 — 解决什么核心问题

解决镜像存储空间爆炸和传输效率问题。如果不分层，每个镜像都是完整的文件系统快照，10 个 Go 项目镜像就要存储 10 份 Alpine。分层后 10 个镜像共享一份 Alpine 基础层，节省 90% 存储和传输。

Step 4 — 核心原理

存储驱动通常是 Overlay2：将多个目录（lowerdir）叠加，upperdir 存修改，merged 是统一视图。COW（写时复制）：读取共享层，只有修改时才复制到容器层。每层有唯一的 SHA256 哈希 ID，层之间通过父子关系链式引用。

Step 5 — 缺点/局限

① 层数过多导致元数据开销大，mount 变慢；② 删除文件不会释放空间（上层用 whiteout 文件标记删除，下层文件仍在）；③ 构建缓存失效后，有变化的层及之后所有层全部重建；④ 不同存储驱动性能差异大（Overlay2 > devicemapper > vfs）。

Step 6 — 如何解决缺点

① 合并 RUN 指令（&& 连接），减少层数；② 在同一层完成安装+清理（apt install && apt clean && rm -rf /var/lib/apt/lists/*）；③ .dockerignore 排除无关文件；④ 多阶段构建，最终镜像只包含运行时产物；⑤ 优先使用 Overlay2 存储驱动。

Step 7 — 实际落地

用 docker history <image> 查看镜像分层；用 docker image inspect 查看层的 RootFS 信息。优化前后对比：未优化 Go 镜像 800MB → 合并 RUN + 多阶段 + Alpine 后 15MB，层数从 18 层减到 5 层。

⚠️ 千万别这么回答

❌ 错误表达

  • "镜像就是一个压缩包"（忽略了分层和复用机制）
  • "删了文件镜像就变小了"（whiteout 机制，下层文件还在）
  • "层越多越好"（层数多导致元数据开销大）
  • "所有基础镜像差不多"（Ubuntu ~70MB vs Alpine ~5MB，差距巨大）
  • "COPY 放前面没事"（会频繁破坏缓存，慢）

✅ 正确表达

  • "镜像是分层叠加的只读文件系统，由 UnionFS（Overlay2）实现统一视图"
  • "删除文件是上层创建 whiteout 文件遮蔽下层，原文件仍在，空间不释放"
  • "应尽可能减少层数——合并 RUN 指令，清理在同一层完成"
  • "依赖少放前面、变化多放后面，这样前面的层能缓存复用"
  • "Alpine（~5MB）比 Ubuntu（~70MB）小 14 倍，是生产首选基础镜像"

🌲 追问树

🐳 镜像分层
├─ → UnionFS / Overlay2
│ ├─ lowerdir：只读层（镜像层）
│ ├─ upperdir：可写层（容器层）
│ ├─ merged：统一视图
│ └─ workdir：Overlay2 内部工作目录
├─ → COW 写时复制
│ ├─ 读：直接从共享层读取
│ ├─ 写：先复制到容器层再修改
│ └─ 删：容器层创建 whiteout 文件标记
├─ → 层缓存复用
│ ├─ 每层有唯一 SHA256 哈希
│ ├─ 相同指令+相同上下文 → 命中缓存
│ └─ 多镜像共享基础层（一份 Alpine 多项目用）
├─ → 优化
│ ├─ 合并 RUN（&& 连接，减少层数）
│ ├─ 多阶段构建（编译+运行分离）
│ ├─ Alpine 基础镜像（~5MB）
│ └─ 同一层清理缓存（apt clean）
├─ → 镜像仓库
│ ├─ Docker Hub（公共）
│ ├─ Harbor（私有企业级）
│ └─ 分层推送/拉取（只传缺失层）
└─ → 镜像签名（Notary）
├─ Docker Content Trust（DCT）
├─ 对镜像层进行数字签名
└─ 确保镜像未被篡改
🧠 记忆钩子
"镜像 = 千层蛋糕：每一层是一条指令的产物，一模一样的层（缓存）可以复用省的重新做；有变化才重新烤那一层"
