🐳 容器生命周期

编号 DO-03 分类：容器原理 难度：⭐⭐⭐ 频率：高频
💡 一句话结论
容器生命周期：create（创建）→ start（启动）→ running / stop（停止）→ kill（强制停）→ rm（删除）。容器本质是进程（Namespace 隔离 + Cgroup 限制），退出后文件系统保留直到 rm。PID 1 问题：容器内主进程需正确处理信号。
🎬 生活类比 — 用「临时工位」看懂容器生命周期
用生活场景看懂全部概念
🏗️
场景1：create = 搭隔间
公司给新员工分配了一个格子间——装了隔板（Namespace 隔离）、拉了网线（Network）、给了预算帽（Cgroup 资源限制），但人还没来。这就是 docker create：文件系统和网络都分配好了，进程还没启动。

🪑
▶️
场景2：start → running = 开工干活
docker start——员工坐进格子间，打开电脑开始工作，状态变成 running。docker run = create + start 一步到位，搭好隔间直接开工。
🌙
场景3：stop = 正常下班
docker stop——HR 敲门说「下班了」（SIGTERM），给你 10 秒收拾东西。员工自己关电脑走人。如果 10 秒后还不走（不响应 SIGTERM），保安过来拉闸（SIGKILL）。这就是 PID 1 问题的根源——很多程序不听敲门！
👮
场景4：kill = 被保安拖走
docker kill——不敲门，保安直接冲进去拉闸断电（SIGKILL），员工连保存的机会都没有。和 stop 的区别：stop 给 10 秒优雅期，kill 直接杀，不留情面。
🔨
场景5：rm = 拆掉隔间
docker rm——员工离职后，隔板拆了、桌子搬走、网线拔了。文件系统彻底清理。注意：只有 rm 后文件才真没——stop 了容器文件还在，跟员工下班了但私人物品还在工位一样。

🗺️ 7 步回答模板

Step 1 — 是什么

容器是 Docker 镜像的运行实例。从"镜像"这个静态模板创建出"容器"这个动态进程。容器本质上就是一个被隔离和限制的 Linux 进程 + 独立的文件系统视图。

Step 2 — 为什么需要

容器提供了轻量级的应用隔离方案，没有虚拟机的完整 OS 开销。启动速度秒级（VM 分钟级），资源利用率高（共享宿主机内核），一台物理机可以跑成百上千个容器。

Step 3 — 解决什么核心问题

解决"应用运行环境标准化"问题。同一个镜像在开发/测试/生产环境运行的容器行为完全一致。同时解决了进程隔离（不互相影响）和资源管控（不互相抢占）的需求。

Step 4 — 核心原理

底层依赖 Linux 两大内核特性：① Namespace 实现隔离（PID/Network/Mount/UTS/IPC/User/Cgroup Namespace 共 7 种），让容器看起来像独立系统；② Cgroup 实现资源限制（CPU/内存/IO/网络），防止一个容器耗尽宿主机资源。生命周期六个状态：created → running → paused → stopped → deleted，外加 restarting 过渡态。

Step 5 — 缺点/局限

① PID 1 问题：容器内 PID 1 进程不处理信号（默认忽略 SIGTERM），docker stop 可能等到超时后才 SIGKILL；② 隔离不彻底：共享宿主机内核，内核 bug 可能影响所有容器；③ 退出后文件残留（docker rm 才真正清理）；④ 状态管理需要注意 zombie 容器。

Step 6 — 如何解决缺点

① PID 1 问题：使用 dumb-init 或 tini 作为 PID 1（正确转发信号+回收僵尸进程），或用 docker run --init；② 内核隔离：安全敏感场景可结合 Kata Containers（轻量 VM）；③ 清理策略：docker run --rm 自动删除；④ docker system prune 定期清理僵尸容器和悬空镜像。

Step 7 — 实际落地

开发环境常用 docker run --rm -it image sh 临时调试。生产环境用 docker run -d --restart=always 保证自动重启。配合 --memory=512m --cpus=1 限制容器资源。用 docker inspect 查看容器状态、Pid、Mounts 等详细信息。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"容器生命周期六步：create 创建 → start 启动 → running 运行 → stop 正常停止 → kill 强制停止 → rm 删除。容器本质是 Namespace 隔离 + Cgroup 限制的 Linux 进程。关键坑是 PID 1 问题——容器主进程不处理信号，docker stop 发 SIGTERM 它不理，等到 10 秒超时被 SIGKILL 强杀。解决方案是 dumb-init/tini 或 --init 参数。"

⚠️ 千万别这么回答

❌ 错误表达

  • "容器就是轻量级虚拟机"（本质是进程，不是 VM）
  • "docker stop 马上停"（先 SIGTERM，10s 超时后 SIGKILL）
  • "docker rm 和 docker stop 一样"（stop 停进程保留容器，rm 删除）
  • "PID 1 能正常处理信号"（默认 shell/bash 不转发信号）
  • "容器完全隔离很安全"（共享内核，隔离不如 VM）

✅ 正确表达

  • "容器是 Namespace+Cgroup 隔离的进程，不是虚拟机"
  • "docker stop = SIGTERM（10s grace period）→ SIGKILL"
  • "容器退出后文件系统还在，文件真正删除在 docker rm"
  • "PID 1 问题：需要 dumb-init/tini 转发信号、回收僵尸进程"
  • "7 种 Namespace：PID/Network/Mount/UTS/IPC/User/Cgroup"

🌲 追问树

🐳 容器
├─ → 生命周期
│ ├─ create：分配文件系统和网络，未启动
│ ├─ start：启动容器进程
│ ├─ run = create + start（一步到位）
│ ├─ stop：SIGTERM → 10s → SIGKILL
│ ├─ kill：直接 SIGKILL（无优雅期）
│ └─ rm：删除容器和文件系统
├─ → 底层机制：Namespace + Cgroup
│ ├─ PID Namespace：进程隔离（容器内 PID 1 是宿主机某 PID）
│ ├─ Network Namespace：独立网络栈（网卡/IP/端口）
│ ├─ Mount Namespace：独立文件系统挂载点
│ └─ Cgroup：CPU/内存/IO/网络带宽限制
├─ → PID 1 问题
│ ├─ 容器内 PID 1 不自动处理 SIGTERM
│ ├─ 解决方案：dumb-init / tini（轻量 init 进程）
│ └─ Docker 内置：docker run --init
├─ → docker exec 原理
│ ├─ 进入已有容器的 Namespace（setns 系统调用）
│ ├─ 新进程加入 PID/Mount/Network 等 Namespace
│ └─ 因此 exec 能看见容器内的进程和文件
├─ → exit vs stop
│ ├─ exit：容器内主进程自己退出（正常结束）
│ ├─ stop：从外部发信号让容器停止
│ └─ 容器退出码：0 正常，非 0 异常
└─ → restart 策略
├─ no：不自动重启（默认）
├─ always：总是重启（Docker daemon 启动时也重启）
├─ unless-stopped：除非手动 stop，否则重启
└─ on-failure:N：退出码非 0 时重启，最多 N 次
