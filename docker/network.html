🐳 Docker 网络模式

编号 DO-05 分类：网络 难度：⭐⭐⭐⭐ 频率：高频
💡 一句话结论
四种网络模式：bridge（默认，容器在隔离网络中通过 NAT 访问外网）、host（与宿主机共享网络，性能好但不隔离）、none（无网络，纯隔离）、overlay（跨主机 Swarm 通信）。容器间通过容器名 DNS 解析通信（同一个自定义 bridge 网络）。
🎬 生活类比 — 用「合租房」看懂 Docker 四种网络模式
用生活场景看懂全部概念
🏠
场景1：bridge = 合租房各房间独立
默认模式。每间房（容器）有自己的门牌号（IP），要出门（访问外网）走公共大门 NAT（docker0 网桥 + iptables MASQUERADE）。自定义 bridge 网络更高级——室友之间可以叫名字（DNS 解析容器名），默认 bridge 不行，只能叫门牌号（IP）。

场景2：host = 睡客厅
不用房间了，直接睡客厅沙发——和房东（宿主机）共用所有空间、网络、门牌号。好处是零隔离开销，性能最佳；坏处是你在客厅做的事所有人都看得见（端口冲突、安全风险）。-p 端口映射在 host 模式下无效，因为你就睡在房东的床上。
🚫
场景3：none = 禁闭室
--network none——把你关进小黑屋，只有 lo 回环接口，没有任何网络。只能自言自语。适合处理敏感数据的批处理任务，最高级别的网络隔离。
🌉
场景4：overlay = 跨楼连通
Swarm 模式——不同楼栋（不同宿主机）的租客需要通信？通过 VXLAN 隧道把两个房子的局域网连起来，看起来像在同一个屋里。代价是封装/解封装的性能损耗约 10-15%。替代方案：Calico（纯路由/BGP）、Flannel。

🗺️ 7 步回答模板

Step 1 — 是什么

Docker 网络是容器间及容器与外部通信的基础设施。Docker 为每个容器创建独立的网络命名空间（Network Namespace），通过虚拟网卡（veth pair）连接到虚拟网桥（docker0），实现容器间及容器与外部的通信。

Step 2 — 为什么需要

微服务架构中多个容器需要互相通信（如 Web 连接 MySQL、Redis）。需要网络隔离（不让数据库暴露公网）、跨主机通信（Swarm/K8s）、性能优化（host 模式减少 NAT 开销）。

Step 3 — 解决什么核心问题

解决"容器间如何互相发现和通信"以及"外部如何访问容器服务"两个核心问题。通过自定义网络和内置 DNS，容器可以通过名称互相调用，无需关心 IP 变化。

Step 4 — 核心原理

bridge 模式：宿主机上创建 docker0 网桥，每个容器一个 veth pair（一端在容器内 eth0，一端在 docker0），容器通过 NAT（iptables MASQUERADE）访问外网。同一个自定义 bridge 网络的容器可通过名称互相 DNS 解析（内置 DNS Server 127.0.0.11）。host 模式直接复用宿主机网络栈。overlay 模式用 VXLAN 隧道实现跨主机二层互通。

Step 5 — 缺点/局限

① 默认 bridge 网络不支持 DNS 解析（只有自定义 bridge 支持）；② host 模式端口冲突——容器端口直接占用宿主机端口；③ overlay 有 VXLAN 封装开销，性能损耗约 10-15%；④ none 模式完全没有网络用途很受限；⑤ iptables 规则多了之后性能下降。

Step 6 — 如何解决缺点

① 生产环境用自定义 bridge 网络而非默认 bridge（获得 DNS 解析能力）；② 高性能场景用 host 模式但需注意端口规划；③ 大规模 overlay 网络考虑 Calico/Flannel（纯路由/BGP）替代 Docker Swarm overlay；④ 用 docker network ls/inspect 管理网络。

Step 7 — 实际落地

典型三容器场景：docker network create app-net → Web、MySQL、Redis 都加入 app-net → Web 代码里用 mysql:3306 和 redis:6379 即可通信（DNS 解析容器名）。端口暴露：-p 8080:80 将容器 80 映射到宿主机 8080，外网通过宿主机 IP:8080 访问。

⏳
60 秒内说清
"Docker 四种网络模式：①bridge 默认模式——每个容器独立 IP，通过 NAT 访问外网，通过端口映射让外部访问；②host——和宿主机共享网络栈，性能最好但没有隔离，端口直接冲突；③none——纯隔离，没网络；④overlay——跨主机通信（Swarm 模式用 VXLAN）。关键点：同一个自定义 bridge 网络中的容器可以通过容器名互相 DNS 解析，默认 bridge 网络不支持。"

⚠️ 千万别这么回答

❌ 错误表达

  • "bridge 和 host 一样"（bridge 有隔离 NAT，host 不隔离共享网络栈）
  • "任何 bridge 网络都能用容器名通信"（默认 bridge 不支持 DNS，自定义 bridge 才支持）
  • "-p 和 --network host 可以一起用"（host 模式下 -p 无效，因为直接共享网络）
  • "overlay 没有性能损耗"（VXLAN 封装/解封装有 10-15% 开销）
  • "容器 IP 是固定的"（重启后 IP 可能变化，应该用容器名/服务名通信）

✅ 正确表达

  • "bridge 通过 docker0 网桥 + NAT 隔离通信；host 直接共享宿主机网络栈"
  • "自定义 bridge 网络内置 DNS Server（127.0.0.11），支持容器名解析"
  • "host 模式下 -p 参数无效，容器直接监听宿主机端口"
  • "overlay 用 VXLAN 隧道跨主机，性能损耗约 10-15%"
  • "容器间通信应使用容器名/服务名，不依赖 IP"

🌲 追问树

🐳 Docker 网络
├─ → bridge（默认）
│ ├─ docker0 网桥 + veth pair（容器内 eth0 ↔ 宿主机 veth）
│ ├─ NAT（iptables MASQUERADE）访问外网
│ ├─ 默认 bridge：不支持 DNS，只能用 IP 通信
│ └─ 自定义 bridge：内置 DNS（127.0.0.11），支持容器名解析
├─ → host（共享宿主机）
│ ├─ 直接复用宿主机网络栈（网卡/IP/端口）
│ ├─ 性能最优（无 NAT 开销）
│ ├─ 不隔离：端口冲突、安全风险
│ └─ -p 参数无效
├─ → none（无网络）
│ ├─ 只有 lo 回环接口
│ ├─ 最高隔离级别
│ └─ 场景：处理敏感数据的批量任务
├─ → overlay（跨主机 Swarm）
│ ├─ VXLAN 隧道封装二层帧
│ ├─ 跨主机容器像在同一个局域网
│ ├─ 性能损耗约 10-15%
│ └─ 替代方案：Calico（BGP）、Flannel
├─ → 容器间通信
│ ├─ 同网络：容器名 DNS 解析 → 内网 IP 直连
│ ├─ 自定义 bridge：内置 DNS Server
│ └─ 不同网络：需要 docker network connect 加入多个网络
└─ → 端口映射（-p host:container）
├─ iptables DNAT 规则转发
├─ -p 8080:80（宿主机 8080 → 容器 80）
└─ -P：随机映射 EXPOSE 声明的端口
