☸️ Service 服务发现

编号 K8S-03 分类：网络 难度：⭐⭐⭐⭐ 频率：必问
💡 一句话结论
Service 通过 Label Selector 将请求负载均衡到 Pod。三种类型：ClusterIP（集群内访问，默认）、NodePort（节点端口映射，30000-32767）、LoadBalancer（云厂商 LB）。通过 kube-proxy 实现（iptables / IPVS 模式）。

顾客（请求）来了，不管后厨哪个厨师掌勺、换了几拨人（Pod 增删变化），只认领位员（Service 的稳定 ClusterIP）。领位员通过标签（Label Selector：app=nginx）识别谁是后厨员工（后端 Pod），把客人带到空闲厨师面前（负载均衡）。

场景2：ClusterIP = 内部订餐热线
只有餐厅内部员工能打的订餐电话（集群内可达）。微服务之间互相调用就靠这个——my-svc.default.svc.cluster.local 是 DNS 名称，Pod 重建了 IP 变了也没关系，名字不变。默认类型，最常用。
🪟
场景3：NodePort = 外卖窗口
在餐厅墙上开了个外卖窗口（每个节点开端口 30000-32767），路人可以通过 节点IP:端口 点餐。方便是方便，但窗口多了管理麻烦，直接暴露也有安全风险。一般只在测试环境用，前面要再加个 LB。
🏛️
场景4：LoadBalancer = 大堂入口
商场给餐厅分配了一个正式大堂入口（云厂商 LB），路人从商场正门进来直接被引导到餐厅。底层其实还是先建了外卖窗口（NodePort）+ 商场前台（LB）转发过来。每开一家新餐厅就要一个新入口，成本高——所以后来有了 Ingress（共享前台）。

🗺️ 7 步回答模板

Step 1 — 是什么

Service 是 K8s 中将一组 Pod 暴露为网络服务的抽象。Pod IP 会变（重建后变化），Service 提供一个稳定的虚拟 IP（ClusterIP）和 DNS 名称，客户端通过 Service 访问后端 Pod，无需关心 Pod 的 IP 变化。

Step 2 — 为什么需要

Pod 是"易失的"——IP 不固定、随时可能被替换。如果前端直接连 Pod IP，Pod 重启后 IP 变了就失联。Service 提供稳定的访问入口 + 自动负载均衡 + 服务发现，解决了"动态 Pod 如何被稳定访问"的问题。

Step 3 — 解决什么核心问题

实现"服务发现"和"负载均衡"：①前端不需要知道后端 Pod IP 和数量变化；②自动负载均衡到健康的 Pod（通过 readiness probe）；③通过 DNS 名称（如 my-svc.default.svc.cluster.local）简化服务间调用。

Step 4 — 核心原理

Service 通过 Label Selector 匹配目标 Pod，维护 Endpoints 列表（健康 Pod 的 IP:Port）。kube-proxy 在每个节点上运行，将访问 Service ClusterIP 的流量转发到后端 Pod。iptables 模式（随机负载均衡，规则随 Pod 增多而膨胀）→ IPVS 模式（支持多种调度算法：rr/lc/dh/sh，性能更好）。

Step 5 — 缺点/局限

① ClusterIP 只在集群内可访问；② iptables 模式下大量 Service 时规则爆炸（IPVS 改善）；③ 默认负载均衡是随机/轮询（无会话保持时需要额外配置 sessionAffinity）；④ NodePort 端口范围有限（30000-32767）且安全风险高；⑤ 不支持 7 层路由（需要 Ingress）。

Step 6 — 如何解决缺点

① 外部访问用 NodePort/LoadBalancer 或 Ingress；② 大规模集群用 IPVS 模式替代 iptables；③ 需要会话保持配置 sessionAffinity: ClientIP；④ NodePort 生产环境避免直接暴露，前面加 LB；⑤ 7 层路由用 Ingress + Ingress Controller。

Step 7 — 实际落地

典型三层架构：前端 Deployment → ClusterIP Service（内部）→ 后端 Deployment → ClusterIP Service（内部）→ 数据库 StatefulSet。外部入口：LoadBalancer Service 或 Ingress。配置示例：kubectl expose deploy/nginx --port=80 --target-port=80 --type=ClusterIP。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"Service 是 K8s 的服务发现和负载均衡抽象。三种类型：①ClusterIP——集群内部访问，默认；②NodePort——在每个节点开一个端口（30000-32767）；③LoadBalancer——云厂商分配公网 LB。底层由 kube-proxy 实现：iptables（规则链，默认）或 IPVS（内核 L4 LB，性能更好）。Pod 通过 Service 名 + DNS 互相访问，IP 变化了也没关系。"

⚠️ 千万别这么回答

❌ 错误表达

  • "Service 和 Pod 是一一对应的"（一对多，负载均衡到多个 Pod）
  • "ClusterIP 可以从外部访问"（只能集群内部访问）
  • "NodePort 端口可以随便设"（范围 30000-32767）
  • "Service 是 7 层负载均衡"（是 L4 四层，按 IP:Port 转发）
  • "iptables 和 IPVS 一样"（IPVS 支持多种调度算法、性能更好）

✅ 正确表达

  • "Service 通过 Label Selector 匹配后端 Pod，一对多负载均衡"
  • "ClusterIP 只能集群内部访问，外部访问需 NodePort/LoadBalancer/Ingress"
  • "NodePort 默认范围 30000-32767，可通过 --service-node-port-range 调整"
  • "Service 是 L4 四层负载均衡，7 层路由需 Ingress"
  • "IPVS 支持 rr/lc/dh/sh 等调度算法，iptables 只有随机"

🌲 追问树

☸️ Service
├─ → Label Selector
│ ├─ 通过标签匹配后端 Pod（app=nginx）
│ ├─ 动态维护 Endpoints 列表
│ └─ Pod 增删 → Endpoints 自动更新
├─ → ClusterIP（内部访问，默认）
│ ├─ 分配虚拟 IP（仅集群内可达）
│ ├─ DNS 名称：svc-name.namespace.svc.cluster.local
│ └─ 最常用：微服务间互相调用
├─ → NodePort（节点端口映射）
│ ├─ 每个节点开放端口（30000-32767）
│ ├─ 外部可通过 <NodeIP>:<NodePort> 访问
│ └─ 测试/演示环境常用
├─ → LoadBalancer（云厂商 LB）
│ ├─ 云厂商自动创建 LB（AWS ELB/Aliyun SLB）
│ ├─ 底层创建 NodePort + LB 转发
│ ├─ 每 Service 一个 LB（成本高）
│ └─ 生产环境推荐用 Ingress 统一入口
├─ → kube-proxy 模式
│ ├─ iptables：规则链，随机负载均衡，规则多时性能下降
│ ├─ IPVS：内核 L4 LB，支持 rr/lc/dh/sh/sed/nq 调度算法
│ └─ 大规模集群强烈建议 IPVS
├─ → Headless Service
│ ├─ clusterIP: None，无虚拟 IP
│ ├─ DNS 直接返回所有 Pod IP（不经过 Service 代理）
│ └─ 场景：StatefulSet 需要直接访问特定 Pod、自定义负载均衡
└─ → Endpoints / EndpointSlice
├─ Endpoints：保存健康 Pod 的 IP:Port
├─ EndpointSlice（K8s 1.21+）：替代 Endpoints，更高效
└─ 按 Service 组织，kube-proxy 据此更新转发规则
