☸️ Ingress 流量入口

编号 K8S-04 分类：网络 难度：⭐⭐⭐⭐ 频率：高频
💡 一句话结论
Ingress 是 7 层（HTTP/HTTPS）路由规则，通过 Ingress Controller（Nginx/Traefik/Istio）实现域名路由、TLS 终结、路径转发。不同于 Service（4 层），Ingress 可以做基于 host/path 的流量分发。

🗺️ 7 步回答模板

Step 1 — 是什么

Ingress 是 K8s 中管理外部 HTTP/HTTPS 流量进入集群的 API 对象。它定义了域名到 Service 的路由规则（根据 host 和 path 转发），配合 Ingress Controller（如 Nginx Ingress）实际执行流量转发和 TLS 终结。

Step 2 — 为什么需要

如果用 LoadBalancer Service，每个服务都要一个 LB（成本高、管理复杂）。Ingress 用单一入口统一管理多个服务的路由：api.example.com → api-svc、web.example.com → web-svc，一个 LB 搞定所有服务。

Step 3 — 解决什么核心问题

解决"7 层流量路由"问题——基于域名、路径、Header 等 HTTP 层信息分发流量，这是 Service（L4）做不到的。同时统一管理 TLS 证书、限流、认证等网关功能。

Step 4 — 核心原理

Ingress 资源定义路由规则（host + path → backend Service），Ingress Controller（实际是 Nginx/Traefik 等反向代理）watch API Server 的 Ingress 变化，动态更新自己的配置文件（如 nginx.conf）并热加载。Ingress Controller 本身通常通过 LoadBalancer 或 NodePort Service 暴露。

Step 5 — 缺点/局限

① Ingress 只支持 HTTP/HTTPS（7 层），不支持 TCP/UDP（需用其他方式）；② 不同 Ingress Controller 实现有差异（Nginx 支持 annotation，Traefik 用 middleware）；③ 复杂路由场景 annotation 配置膨胀；④ 跨命名空间 TLS 证书管理复杂；⑤ 作为 K8s 原生 API 功能有限（催生了 Gateway API）。

Step 6 — 如何解决缺点

① TCP/UDP 流量用 ConfigMap 配合 Nginx Ingress 或使用 Gateway API；② 统一团队使用同一款 Ingress Controller 并用 GitOps 管理配置；③ 证书管理用 cert-manager 自动化（ACME 自动申请/续期 Let's Encrypt）；④ 复杂场景迁移到 Gateway API（K8s 下一代入口标准）。

Step 7 — 实际落地

部署 Nginx Ingress Controller → 创建 Ingress 资源定义路由 → 配合 cert-manager 自动签发 TLS 证书。示例：api.example.com/api/* → api-service:8080、api.example.com/web/* → web-service:3000。配置限流 annotation：nginx.ingress.kubernetes.io/limit-rps: "10"。

⏳
60 秒内说清
"Ingress 是 K8s 的 7 层 HTTP 路由。和 Service（4 层）最大区别：Ingress 能根据域名和路径转发——比如 api.example.com/user 到一个 Service，api.example.com/order 到另一个。它只是一组路由规则，实际转发靠 Ingress Controller（Nginx/Traefik）。配合 cert-manager 自动管理 TLS 证书。新一代标准是 Gateway API，功能更强但 Ingress 目前仍是主流。"

⚠️ 千万别这么回答

❌ 错误表达

  • "Ingress 自带转发功能"（只定义规则，实际转发靠 Controller）
  • "Ingress 和 Service 一样都是四层"（Ingress 是 7 层 L7）
  • "所有 Ingress Controller 一样"（Nginx/Traefik/Istio 实现差异大）
  • "TLS 证书需要手动管理"（cert-manager 可自动化 ACME 申请续期）
  • "Ingress 支持 TCP"（Ingress 只支持 HTTP/HTTPS，TCP 需特殊配置）

✅ 正确表达

  • "Ingress 定义路由规则，Ingress Controller 实际执行转发"
  • "Ingress 是 OSI 7 层（HTTP），Service 是 4 层（TCP/UDP）"
  • "主流 Controller：Nginx（最普及）、Traefik（云原生）、Istio Gateway（服务网格）"
  • "cert-manager 自动签发和续期 TLS 证书（Let's Encrypt ACME 协议）"
  • "TCP/UDP 场景需用 ConfigMap 或 Gateway API，不在标准 Ingress 范围内"

🌲 追问树

☸️ Ingress
├─ → Ingress Controller
│ ├─ Nginx Ingress：最普及，功能全，annotation 驱动
│ ├─ Traefik：云原生，自动发现，middleware 机制
│ ├─ Istio Gateway：服务网格场景，功能最强
│ └─ Controller 本身是一个 Deployment + Service（LB/NodePort）
├─ → 域名路由（host 规则）
│ ├─ api.example.com → api-service
│ ├─ web.example.com → web-service
│ └─ 默认 backend（无匹配时的兜底）
├─ → 路径转发（path 规则）
│ ├─ /api/* → backend-api
│ ├─ /web/* → backend-web
│ └─ PathType：Prefix（前缀匹配）/ Exact（精确匹配）/ ImplementationSpecific
├─ → TLS 终结
│ ├─ Secret 存储 TLS 证书（tls.crt + tls.key）
│ ├─ Ingress 引用 Secret 进行 TLS 终结
│ └─ cert-manager：自动申请 Let's Encrypt 证书（ACME）
├─ → Ingress vs Service（7 层 vs 4 层）
│ ├─ Service：L4，按 IP:Port 转发
│ ├─ Ingress：L7，按 Host/Path/Header 转发
│ └─ 两者配合使用（Ingress → Service → Pod）
├─ → Gateway API（下一代）
│ ├─ 比 Ingress 更强大、更可扩展
│ ├─ 角色分离：基础设施（GatewayClass）→ 集群管理员（Gateway）→ 应用开发者（HTTPRoute）
│ └─ 原生支持 HTTP/TCP/UDP/TLS/gRPC
└─ → 限流 / 认证等高级功能
├─ 限流：nginx.ingress.kubernetes.io/limit-rps
├─ 认证：nginx.ingress.kubernetes.io/auth-url
├─ 金丝雀：nginx.ingress.kubernetes.io/canary: "true"
└─ 跨域 CORS、重写 URL、白名单等
