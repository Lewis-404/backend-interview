☸️ Pod 详解

编号 K8S-01 分类：核心概念 难度：⭐⭐⭐⭐ 频率：必问
💡 一句话结论
Pod 是 K8s 最小调度单元，共享 Network Namespace 和 IPC，可包含多个容器（共享 localhost + 共享 Volume）。Sidecar 模式（Pod 内辅助容器，如日志采集 / Envoy 代理）。Init 容器先于主容器运行完成。

场景1：Pod = 合租的卧室
一间卧室住两个人——共享 WiFi（Network Namespace）、共享书架（Volume）、共享门牌号（同一个 IP）。两个人是室友（容器），命运绑定：一起入住（调度）、一起退租（销毁）。这就是 Pod 把多个容器"打包"在一起的哲学。

场景2：Sidecar = 室友帮你收快递
你（主容器）在写代码，室友（Sidecar）帮你收快递、倒垃圾、记日志——你专注自己的事，辅助工作他全包。典型：Nginx 容器跑 Web 服务 + Filebeat 容器采集日志。住在同一间房（Pod）的好处是可以用 localhost 通信，又快又简单。
🧹
场景3：Init 容器 = 先打扫房间再入住
搬进去之前，先派个清洁阿姨（Init 容器）把房间打扫干净、检查水电（等数据库就绪）、铺好床单（初始化配置）。Init 容器做完就撤（退出），主容器才住进来。如果 Init 失败（打扫不干净），房间锁着不让住（Pod 卡在 Init 状态）。
🏥
场景4：探针 = 健康检查
liveness 探针——每隔一会敲敲门「你还活着吗？」没回应就踹门做心肺复苏（重启容器）。readiness 探针——「你准备好接客了吗？」没准备好就把门牌从服务名录里摘掉（从 Service 摘除）。startup 探针——慢热的室友，前 5 分钟不催他，启动了再换常规检查。

🗺️ 7 步回答模板

Step 1 — 是什么

Pod 是 Kubernetes 中最小的可调度和可管理单元。一个 Pod 包含一个或多个容器，这些容器共享相同的网络命名空间（localhost 互通）、IPC 命名空间和挂载的 Volume。Pod 是 K8s 的原子调度单位。

Step 2 — 为什么需要

Pod 的设计解决了一个核心问题：哪些进程需要紧密协作？比如 Web 服务 + 日志采集、应用 + 本地缓存。如果没有 Pod，这些紧密耦合的容器需要单独调度，网络通信和管理变得复杂。Pod 把它们"打包"在一起。

Step 3 — 解决什么核心问题

解决"紧密耦合的容器如何作为一个整体被调度和管理"的问题。Pod 内所有容器共享网络（localhost）、共享存储（Volume）、命运共享（一起调度、一起终止）。

Step 4 — 核心原理

每个 Pod 有一个 Infra 容器（pause 容器），它创建并持有 Network Namespace 和其他 Namespace，Pod 内其他容器加入这些 Namespace。这就是为什么 Pod 内容器可以通过 localhost 通信，且共享同一个 IP。多容器模式：Sidecar（辅助进程，如日志采集）、Ambassador（代理外部服务，如 Envoy）、Adapter（标准化输出）。

Step 5 — 缺点/局限

① Pod 内容器耦合度高（共享资源的容器一起被调度和扩缩）；② 一个容器崩溃可能影响整个 Pod；③ Pod IP 不是静态的（Pod 重建 IP 会变）；④ Init 容器失败会导致 Pod 一直处于 Init 状态无法启动；⑤ 不适合将无关服务放在同一个 Pod 中（违反单一职责）。

Step 6 — 如何解决缺点

① 只有真正紧密耦合的容器才放同一 Pod（Sidecar 模式）；② 使用 liveness/readiness/startup 探针实现健康检查和自动重启；③ 用 Service 提供稳定的访问入口（不依赖 Pod IP）；④ Init 容器加超时和重试机制；⑤ 无关服务用独立的 Deployment 管理。

Step 7 — 实际落地

典型 Sidecar 场景：主容器跑 Nginx，Sidecar 容器跑 Filebeat 采集日志到 ES。Init 容器等待数据库就绪后再启动主容器：initContainers: [{image: busybox, command: ['sh','-c','until nc -z mysql 3306; do sleep 2; done']}]。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"Pod 是 K8s 最小调度单元。核心特点：Pod 内所有容器共享网络（localhost 互通）和存储（Volume），通过 Infra 容器实现。三种多容器模式：Sidecar（辅助功能，如日志采集）、Ambassador（代理外部，如 Envoy）、Adapter（标准化输出）。Init 容器在主容器启动前运行且必须完成。Pod 的哲学是：把需要紧密协作的容器打包在一起，享受共享资源和命运共享。"

⚠️ 千万别这么回答

❌ 错误表达

  • "Pod 就是一个容器"（Pod 可以包含多个容器）
  • "Pod 内容器用 IP 通信"（直接 localhost 即可）
  • "Init 容器和普通容器一样"（Init 必须运行完成并退出，且顺序执行）
  • "随便把多个容器放一个 Pod"（应该是紧密耦合的容器才放一起）
  • "Pod IP 是固定的"（Pod 重建后 IP 会变化）

✅ 正确表达

  • "Pod 是最小调度单元，可包含一个或多个容器"
  • "Pod 内容器共享 Network Namespace，可通过 localhost 通信"
  • "Init 容器顺序执行，每个必须成功退出后下一个才开始"
  • "只有紧密耦合的容器才放同一 Pod（如 Sidecar），松耦合用独立 Service"
  • "Pod IP 随 Pod 重建而变化，应通过 Service 访问"

🌲 追问树

☸️ Pod
├─ → 为什么是最小单元？
│ ├─ 共享 Namespace（Network/IPC）
│ ├─ Infra/pause 容器持有 Namespace
│ └─ 紧密耦合进程需要共享资源
├─ → 多容器模式
│ ├─ Sidecar：辅助功能（日志采集 Filebeat、服务网格 Envoy）
│ ├─ Ambassador：代理外部服务（连接池、重试、发现）
│ └─ Adapter：标准化输出（统一日志/监控格式）
├─ → 共享网络（localhost 互通）
│ ├─ 同一 IP 地址，不同端口
│ ├─ 通过 localhost 通信（无需服务发现）
│ └─ 共享端口空间（注意端口冲突）
├─ → Init 容器
│ ├─ 先于主容器运行
│ ├─ 顺序执行，每个必须成功退出
│ ├─ 场景：等待依赖就绪、初始化配置、数据库迁移
│ └─ 与主容器隔离（可用不同镜像、不同安全上下文）
├─ → Pod 生命周期
│ ├─ Pending：已提交到集群，容器未运行
│ ├─ Running：至少有一个容器在运行
│ ├─ Succeeded：所有容器成功退出（restartPolicy: Never）
│ └─ Failed：至少一个容器非 0 退出
└─ → 健康检查探针
├─ liveness：容器是否存活（失败 → 重启）
├─ readiness：是否可接收流量（失败 → 从 Service 摘除）
└─ startup：启动探针（慢启动应用，成功后切到 liveness）
