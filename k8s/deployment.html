☸️ Deployment 滚动更新

编号 K8S-02 分类：工作负载 难度：⭐⭐⭐⭐ 频率：必问
💡 一句话结论
Deployment 管理 ReplicaSet 实现声明式更新和滚动升级。滚动更新：逐步替换旧 Pod（maxSurge 超出数 / maxUnavailable 不可用数），支持回滚（kubectl rollout undo）。ReplicaSet 保证 Pod 副本数始终为期望值。

场景2：滚动更新 = 一幕一幕拍
导演要换演员：不能全场全撤——要一个一个换。让新演员先上场（新 ReplicaSet 扩容 1 个 Pod），旧演员再下场（旧 ReplicaSet 缩容 1 个），交替进行。maxSurge 控制「最多同时几个新演员在后台准备」，maxUnavailable 控制「最多几个旧演员可以不在场」。观众（用户）全程有戏看，零停机。

场景3：回滚 = 重拍
新演员上台后演砸了（新版本有 Bug），导演喊「undo！」——kubectl rollout undo deploy/nginx，旧演员立刻重新上场。旧 ReplicaSet 一直保留着（revisionHistoryLimit 控制保留几个版本），随时可以喊回之前的任何一个版本。
📊
场景4：HPA 自动扩缩 = 根据观众人数自动加座
演出太火爆，观众（流量）爆满——HPA 自动通知场记：「再加 5 个演员！」CPU/内存使用率飙高就自动扩容，闲下来自动缩容，不用导演操心。kubectl autoscale deploy/nginx --min=2 --max=10 --cpu-percent=80。

🗺️ 7 步回答模板

Step 1 — 是什么

Deployment 是 K8s 中管理无状态应用的工作负载控制器，提供声明式更新能力。它通过控制 ReplicaSet 来间接管理 Pod，实现滚动更新、版本回滚、水平扩缩等功能。是生产环境最常用的工作负载类型。

Step 2 — 为什么需要

Pod 本身是"易失的"——挂了不会自动恢复，需要控制器保证副本数。Deployment 比直接管理 ReplicaSet 更高级：支持版本记录、暂停/恢复更新、自动回滚。对开发者来说只需声明"我要 3 个 nginx:1.21 的副本"即可。

Step 3 — 解决什么核心问题

解决"如何零停机更新应用"的核心问题。通过滚动更新机制，新版本 Pod 逐步替换旧版本 Pod，保证在更新过程中始终有一定数量的可用副本处理流量。

Step 4 — 核心原理

Deployment → 创建/更新 ReplicaSet → ReplicaSet 确保 Pod 副本数 = spec.replicas。滚动更新时：创建一个新 ReplicaSet 逐步扩容，旧 ReplicaSet 逐步缩容。参数 maxSurge（最多超出多少 Pod，默认 25%）和 maxUnavailable（最多多少不可用 Pod，默认 25%）控制滚动速度。更新后旧 ReplicaSet 保留（副本数 0）用于回滚。

Step 5 — 缺点/局限

① 只适合无状态应用（StatefulSet 管理有状态应用）；② 滚动更新期间新旧版本可能同时处理流量（需考虑兼容性）；③ 回滚只回滚 Pod 模板，不回滚外部依赖（数据库 schema 等）；④ 大量 Deployment 时 ReplicaSet 会积累（需配置 revisionHistoryLimit）；⑤ 不适合需要有序更新和稳定网络标识的场景。

Step 6 — 如何解决缺点

① 有状态应用用 StatefulSet；② 滚动更新时做好 API 版本兼容（新旧共存）；③ 数据库 migration 单独管理，配合 Deployment 版本；④ 设置 revisionHistoryLimit: 5 限制保留的 ReplicaSet 数量；⑤ 结合 readiness probe 确保新 Pod 就绪后才接入流量。

Step 7 — 实际落地

kubectl set image deploy/nginx nginx=nginx:1.22 触发滚动更新 → kubectl rollout status deploy/nginx 查看状态 → 发现问题用 kubectl rollout undo deploy/nginx 一键回滚。发布策略：开发环境用 Recreate（更快），生产环境用 RollingUpdate + readiness probe。

⏱️ 60 秒极简表达

⏳
60 秒内说清
"Deployment 是 K8s 管理无状态应用的控制器，通过 ReplicaSet 保证 Pod 副本数。滚动更新是核心能力——创建新 ReplicaSet 逐步扩容、旧 ReplicaSet 逐步缩容，由 maxSurge（最多超几个）和 maxUnavailable（最多少几个）控制速度。更新失败能一键回滚。记住层级：Deployment → ReplicaSet → Pod，Deployment 是导演，ReplicaSet 是场记，Pod 是演员。"

⚠️ 千万别这么回答

❌ 错误表达

  • "Deployment 直接管理 Pod"（中间有 ReplicaSet）
  • "滚动更新和蓝绿部署一样"（蓝绿是一次切，滚动是逐步换）
  • "maxSurge=0 最安全"（可能永远完成不了，新 Pod 无法创建）
  • "回滚什么都能回"（只回滚 Pod 模板，不回滚数据库等外部状态）
  • "ReplicaSet 更新后旧 RS 就删了"（保留到 revisionHistoryLimit 用于回滚）

✅ 正确表达

  • "Deployment → 创建/更新 ReplicaSet → ReplicaSet 管理 Pod"
  • "滚动更新 = 逐步替换（新起一个删一个），蓝绿 = 起完一批再切流量"
  • "maxSurge 控制新建速度，maxUnavailable 控制不可用容忍度"
  • "回滚仅回滚 Pod 模板，数据库迁移需单独处理"
  • "旧 ReplicaSet 保留（副本调 0），可通过 --to-revision 回滚到指定版本"

🌲 追问树

☸️ Deployment
├─ → Deployment ↔ ReplicaSet 关系
│ ├─ Deployment 是更上层抽象
│ ├─ 每个版本对应一个 ReplicaSet
│ └─ 层级：Deployment → ReplicaSet → Pod
├─ → 滚动更新机制
│ ├─ maxSurge：超出 replicas 的 Pod 数（默认 25%）
│ ├─ maxUnavailable：最多不可用 Pod 数（默认 25%）
│ ├─ 新 RS 扩容 + 旧 RS 缩容，交替进行
│ └─ readiness gate：新 Pod 就绪才计入 available
├─ → 回滚
│ ├─ kubectl rollout undo deploy/xxx（回滚到上一版）
│ ├─ kubectl rollout undo deploy/xxx --to-revision=2（指定版本）
│ ├─ kubectl rollout history deploy/xxx（查看版本历史）
│ └─ revisionHistoryLimit 控制保留数量
├─ → 更新策略对比
│ ├─ RollingUpdate：逐步替换（零停机，默认）
│ ├─ Recreate：先全删再全建（有停机，但干净）
│ ├─ 蓝绿部署：新版本全量部署后切流量
│ └─ 金丝雀发布：少量新版本验证后再全量
└─ → HPA 水平扩缩
├─ 基于 CPU/内存/自定义指标自动调整 replicas
├─ kubectl autoscale deploy/nginx --min=2 --max=10 --cpu-percent=80
└─ HPA 修改 Deployment 的 replicas → ReplicaSet 执行
