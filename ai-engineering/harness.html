🏗️ Agent Harness 五层架构

编号 AI-02 分类：系统设计 · Agent 工程 难度：⭐⭐⭐⭐⭐ 标签：差异化必杀技
💡 一句话结论
Agent Harness 是 Agent 的操作系统——从身份定义到多 Agent 编排，五层递进：身份（Identity）→ 能力（Capabilities）→ 门控（Guardrails）→ 记忆（Memory）→ 编排（Orchestration）。下层出错上层全部失效，所以设计必须从第一层开始逐层收敛。
🎬 生活类比 — 用「造房子」看懂五层架构
为什么顺序不能乱？
🏠
盖一栋智能建筑
L1 身份：这是医院还是商场？先定义用途。
L2 能力：允许进哪些房间？手术室需要刷卡。
L3 门控：药房双人核对——一个人不能取药。
L4 记忆：病历系统——上次来开了什么药。
L5 编排：挂号→看诊→取药——流程不能反。

反模式：先装智能门锁（L3）再决定这楼是干嘛的（L1）→ 门锁装错位置全白费。

🧱 五层架构详解

1
身份层 Identity
"你是谁" → 角色先于能力
System Prompt 不是工具说明书。先定义 Agent 的「不做什么」，再列「能做什么」。
否决条件 > 能力清单 硬约束用祈使句 软约束用原则句 实战：SOUL.md 角色定义先于工具列表
2
能力层 Capabilities
"你能碰什么" → 操作粒度隔离
工具白名单精确到操作粒度（读/写/执行分开），不是整块 file access。网络访问按「能力」建模，不是「可以上网」。
先环境层后模型层 出站白名单按能力授权 隔离强度匹配用户监督能力 实战：Agent 只能写 workspace，不能碰系统文件
3
门控层 Guardrails
"什么绝对不能做" → 硬约束不可绕过
每一步跳过必须有可验证的条件，不是 Agent「觉得没必要」。子 Agent 声称「文件已写入」→ 必须验证。
held-out gate 硬门控 验证优先于信任 去重是门控的一种 失败必须可见 实战：OTC 确认码机制 / 子 Agent 产出验证协议
4
记忆层 Memory
"你记得什么，记得多久" → 分层淘汰
热数据 7 天 → 温数据 30 天 → 精华永久。写便宜读贵，所以要有自动 promote 机制。
T1/T2/T3 三层淘汰 多级降级搜索路径 搜索结果标注置信度 实战：brain-first-lookup 5步查找链 / WAL 先写后回
5
编排层 Orchestration
"谁在什么时候做什么" → Pipeline 优于 Supervisor
线性管道不需要 Agent 框架。自治 Agent 必须有记忆闭环——cron job 对自身历史产出无感知是设计盲区。
Pipeline > Supervisor context_from 链式注入 子 Agent 验证协议 实战：10 bot 团队分工 + 失败传播边界定义

📐 层间依赖关系

L1 身份错 → 所有决策偏离
↓
L2 能力不设限 → 门控形同虚设
↓
L3 门控缺失 → Agent 绕过所有约束
↓
L4 记忆丢失 → 编排失去上下文
↓
L5 编排断裂 → 系统不可观测、不可恢复

🎤 一分钟话术模板

📋 面试标准版（60秒）
我从第一性原理推导了一套 Agent Harness 五层架构——这不是「用 AI 工具」，而是设计 Agent 的操作系统。

五层从下到上：
L1 身份层——先定义 Agent 不做什么，再列能做什么
L2 能力层——工具白名单精确到操作粒度，不是「可以上网」
L3 门控层——不可绕过的硬约束，子 Agent 产出必须验证
L4 记忆层——三层淘汰，写便宜读贵，自动 promote
L5 编排层——Pipeline 优于 Supervisor，自治 Agent 必须有记忆闭环

设计原则很简单：下层出错，上层全部失效。身份定义错了，后面的能力、门控、记忆全都白做。所以我们从 L1 开始逐层设计，不是反过来。
📋 精简版（30秒·电梯演讲）
我的 AI 工程核心不是「用工具」，而是设计 Agent 的操作系统。从身份到编排五层递进——先定义 Agent 是谁，再控制它能碰什么，再设不可绕过的硬门控，再建分层记忆，最后编排多 Agent 协作。下层出错上层全部失效，所以设计必须从身份层开始逐层收敛。这套方法论来自部署 10 个 Bot 协作的实战经验。

🔥 常见追问 & 回答

Q: 这和普通的「用 AI 工具」有什么区别？
普通人用 AI：打开 ChatGPT 问问题。
工程师用 AI：把 Claude Code 接入开发流程。
我做的是 设计 Agent 的运转规则——它在什么条件下可以跳过某一步？它的产出谁来验证？它的记忆什么时候淘汰？

打个比方：普通人用 ChatGPT 就像租车开；工程师用 Claude Code 就像自己改装车；我做的五层架构是 设计交通规则和车辆安全标准——不针对某一辆车，而是让所有车安全运转。
Q: 你说「子 Agent 产出必须验证」，具体怎么验证？
三个层面：
① 存在性验证——子 Agent 说「文件写好了」，主 Agent 用 find/grep 确认文件确实存在
② 内容正确性——抽样检查输出内容是否符合预期格式
③ 不重复验证——cron job 写入前先扫描历史产物，避免重复生成

来源是真实踩坑：子 Agent 声称完成但文件未落盘，导致静默失败。
Q: 记忆层为什么要分层淘汰？
因为 Agent 的 context window 有限，记忆会膨胀。
T1（7天热数据）保证最近的决策能快速召回
T2（30天温数据）保留阶段脉络
T3（精华永久）沉淀跨场景通用的经验

核心原则是「读写不对称」——写入成本低（一句话的事），读取成本高（要搜要推理）。所以要有自动 promote 机制，把高价值记忆从 T1 升到 T3，而不是永远手工整理。
Q: Pipeline 优于 Supervisor 是什么意思？
很多团队一上来就搞 Agent 框架，让 AI 自己决策、自己调度。
我的实践是：确定性流程不需要 Agent 框架。如果流程是 A→B→C 固定顺序，用 cron + script 就够了。
什么时候才需要 Agent？当步骤之间需要非确定性判断——比如「这段代码需要 refactor 吗？」需要 AI 决策。否则就是在用大炮打蚊子。
另一个关键是「自治 Agent 的记忆闭环」——cron job 如果不知道自己上次写过什么，就会重复产出。必须显式扫描历史。
Q: 这套方法论有什么实际效果？
三项可量化：
① 零静默失败——子 Agent 产出验证协议覆盖所有自动化流程，Agent 协作事故率从「偶发」降到零
② 去重闭环——自治 cron job 写入前扫描历史，公众号选题等场景杜绝重复生成
③ 知识传承——bot-shared-knowledge 开源仓库，新 Bot 通过知识库自学上手，无需人工培训

定性效果：多 Agent 团队从「能用」升级到「可观测、可恢复、可审计」。

🎯 面试中的战术用法

🥊
何时主动抛出
当面试官问「你做过什么 AI 相关的」或「你的技术亮点是什么」——不要只说「我用了 Claude Code」，而是说「我设计了一套 Agent Harness 五层架构」。

为什么有效：大部分候选人只会说「用 AI 写代码」，你能说出「身份→能力→门控→记忆→编排」这个框架，面试官立刻知道你不只是使用者，而是设计者。
⚠️
何时收住
面试官没追问细节就别展开到 5 层全讲。先抛一句「五层架构」，等他追问再展开。如果他只对某一层感兴趣，就专注那一层讲——不要自顾自把五层全倒出来。

记住：面试是对话，不是演讲。这五层是弹药库，不是演讲稿。
