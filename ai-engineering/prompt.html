✍️ Prompt 工程

编号 AI-05分类：Agent 工程难度：⭐⭐⭐标签：基础能力
💡 一句话结论
Prompt 工程不是「怎么写提示词」，而是 把人类意图结构化地编译成模型可执行的指令。真正的工程化包括：模板管理、版本控制、效果评估、持续迭代——和写代码一样严肃。
🎬 一句话类比
Prompt = 给大厨的菜谱，不是「随便做点好吃的」
👨‍🍳
你跟大厨说「做点好吃的」——他可能做一桌满汉全席，也可能下碗面。你写清楚「4 人份、预算 200、忌辣、40 分钟上桌、前菜+主菜+甜点」——这才是能稳定交付的菜谱。
Prompt 工程的本质是把模糊需求变成确定性指令。

四种核心模式

🎯 角色 + 约束
「你是谁 + 不做什么」
先定义身份边界，再列能力。硬约束祈使句，软约束原则句。例：「你是代码审查专家。你不能直接改代码，只能给建议。」
🧱 结构化输出
指定输出格式
JSON Schema、Markdown 标题层级、表格。不是「写个报告」，而是「输出：1.摘要 2.问题列表 3.建议方案」。
🔄 Few-shot / Chain-of-Thought
给例子 + 让模型展示推理
「请逐步分析，先列问题再给方案。」复杂任务不说「直接给我答案」，强制展示推理过程。
🛡️ Guardrails 嵌入
在 prompt 里写不可绕过的规则
「遇到 X 情况必须先确认。」不是「尽量」，是「必须」。这是 Agent Harness L3 门控层在 prompt 层的投影。

🎤 面试话术模板

📋 60秒完整版
我的 Prompt 工程实践不是「怎么写提示词」，而是把 prompt 当成代码来管理：

① 模板化——所有 prompt 统一 Git 管理，有版本历史，就像管理代码库
② 效果评估——每次修改 prompt 后对比输出质量，不是感觉「好像好了一点」
③ 四种模式——角色+约束、结构化输出、Few-shot/CoT、Guardrails 嵌入，按场景选用
④ CI 集成——代码提交自动触发 AI Code Review，Review 用的 prompt 同样是版本管理的

核心认知：prompt 是 Agent 系统的配置层，不是一次性手写的聊天内容。它需要测试、迭代、回滚。

🔥 常见追问

Q: 你的 prompt 效果怎么评估？
两个维度：
① 格式符合率——输出是否满足预期的结构（标题层级、字段完整性、禁止项是否出现）
② 内容质量——通过对比修改前后的输出，看是否解决了之前的典型问题（遗漏、幻觉、啰嗦）

不是靠感觉，而是靠对比测试——同一组输入，新老 prompt 各跑一遍，逐项打分。
Q: Prompt 工程和 Fine-tuning 怎么选？
Prompt 工程是外部约束——改指令不改模型，成本低、可快速迭代、可版本管理。Fine-tuning 是内部参数调整——改模型本身，成本高、需要训练数据、部署复杂。

我的选择策略：先 prompt 迭代到天花板，确定 prompt 确实解决不了（比如需要领域特定知识内化），再考虑 fine-tuning。大多数场景 prompt 就够了。
Q: 你怎么管理 10 个 Bot 的 prompt？
每个 Bot 有自己的 SOUL.md（身份+角色约束），还有共享的 AGENTS.md（通用执行规则）。修改走 Git PR 流程，改完先 dry-run 测试，确认效果后才推送到生产。

这本质上是配置即代码的思路——prompt 就是 Agent 的配置文件，和 nginx.conf 一样需要版本管理。
