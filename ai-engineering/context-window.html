🪟 Context Window

编号 AI-06分类：Agent 工程难度：⭐⭐⭐⭐⭐标签：核心概念 · 必问
💡 一句话结论
Context Window 是 LLM 的 「工作记忆」容量上限——模型一次能「看到」多少 token。它不是越大越好：更大的窗口意味着更高的成本和更低的注意力密度（Lost in the Middle 效应）。工程化的关键是 Context Engineering——不是塞更多内容，而是塞对的内容。
🎬 一句话类比
Context Window = 你的办公桌面
🪟
桌面只有这么大——摊开 5 份文件还能看，摊开 50 份就什么都找不到了。
Context Window 同理——塞 200K token 进去，模型对中间部分的注意力急剧下降（Lost in the Middle）。不是「加内存」就解决，而是放什么、放多少、什么时候换——这才是 Context Engineering 的工程价值。

🎤 面试话术模板

📋 60秒完整版
Context Window 的本质是模型的注意力预算——不是内存，是注意力。

关键认知：
① Lost in the Middle——论文证明模型对 context 开头和结尾的内容注意力强，中间部分急剧衰减。不是「128K 窗口就能喂 128K 内容」
② Context 是护城河——Google 的 Gemini 200K、Claude 200K 都是竞争壁垒。因为高质量 context 管理 = 更好的 Agent 决策质量
③ 工程实践三原则：Progressive Disclosure（渐进加载）、Compaction（定期压缩）、Prioritization（高价值内容靠前）

我在 Agent 系统里落地：Skills 按需加载（不是全塞）、记忆分层淘汰（T1→T2→T3）、对话过长时自动摘要压缩——目标不是「塞满窗口」，而是让窗口里的每个 token 都值得花注意力。
📋 30秒精简版
Context Window 是模型的注意力预算，不是内存。核心挑战是 Lost in the Middle——中间部分注意力断崖下降。我的工程策略：Skills 按需加载、记忆分层淘汰、长对话自动压缩。目标是让窗口里的每个 token 都值得模型的注意力，而不是无脑塞满。

🔥 常见追问

Q: Lost in the Middle 是什么？怎么应对？
斯坦福/伯克利论文发现：模型对 context 开头 25% 和末尾 25% 的内容召回率最高，中间 50% 断崖式下降——尤其在多文档问答场景。

工程应对：
① 最重要的信息放开头或结尾，不要埋在中间
② 减少总 token 数——用摘要替代全文，用指令替代啰嗦描述
③ 分块处理——长文档分多次推理，每次上下文短而精
Q: 你说的 Compaction 具体怎么实现？
Compaction 是定期把长对话压缩成摘要——就像 Git 的 squash commit。

触发条件：对话 token 数超过阈值（如 50K），或者新任务开始前。

压缩策略：保留决策结论、丢弃重复内容、保留待办事项。压缩后的摘要替代原始对话注入下一轮 context。

核心原则：保留「后续决策需要知道的信息」，丢掉「已经执行完的过程描述」。
Q: Context Window 越来越大，还需要 Context Engineering 吗？
更需要。

三个原因：
① 成本——200K token 的推理成本远高于 10K，按 API 调用量算，塞满窗口月费可以差 10 倍
② 延迟——更大的 context 意味着更长的首 token 延迟，影响用户体验
③ 注意力稀释——窗口再大，Lost in the Middle 依然存在。问题不在容量，在信息密度

所以结论是反直觉的：窗口越大，Context Engineering 越重要。因为你有更多空间可以浪费，但浪费的代价也更高。
Q: 你在实际项目中怎么管理 context？
四个机制：
① Progressive Disclosure——Skills 先加载目录（名称+一行描述），用户选择后才加载全文
② 分层记忆——T1 热数据常驻、T2 温数据按需加载、T3 精华永久保留但精简
③ Compaction 自动摘要——长对话超过阈值自动触发压缩，保留关键决策丢弃过程描述
④ 子 Agent 隔离——大任务拆给子 Agent，每个子 Agent 独立 context，不相互污染
