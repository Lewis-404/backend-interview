🔌 MCP 协议

编号 AI-03分类：Agent 工程难度：⭐⭐⭐⭐标签：工具链核心
💡 一句话结论
MCP（Model Context Protocol）是 AI 模型的 USB-C 接口——定义了一套标准协议让 LLM 安全地调用外部工具和数据源，替代了过去每个工具各写一套胶水代码的局面。我用它在自己的 Agent 系统中统一管理工具接入。
🎬 一句话类比
MCP = AI 世界的 USB-C
🔌
以前每个设备有自己的充电口——Micro-USB、Lightning、DC 圆口。USB-C 统一了物理接口。
MCP 统一了 AI 调用外部工具的协议接口——不管是查数据库、调 API 还是读文件，都用同一套协议。你不需要为每个工具写 adapter 了。

🎤 面试话术模板

📋 60秒完整版
MCP 是 Anthropic 开源的 Model Context Protocol，可以理解为 AI 模型与外部工具的标准化通信协议。

它的核心价值是 统一工具接入层——传统做法里，每个工具（数据库、文件系统、API）都要单独写对接代码，MCP 之后工具提供方实现一个 MCP Server，AI 客户端通过 MCP Client 统一调用。

我在实际项目里：
① 用 MCP 接入 Jira/Confluence API，Agent 可以直接查 Issue 和 Wiki
② 通过 MCP 暴露文件系统和 Shell 给 Agent，但按操作粒度限制权限（读/写/执行分开）
③ 理解 MCP 的 Tools/Prompts/Resources 三种原语——不是所有能力都叫 Tool，静态知识应该用 Resource 暴露
📋 30秒精简版
MCP 是 AI 调用外部工具的标准协议层——就像 USB-C 统一了硬件接口。它解决的核心问题是：以前每个工具要单独写 adapter，现在工具提供方实现一个 MCP Server，AI 统一通过 MCP Client 调用。我在 Agent 系统里用它统一管理 Jira、文件系统、Shell 等工具的接入和权限控制。

🔥 常见追问

Q: MCP 和 Function Calling 有什么区别？
Function Calling 是单次调用的格式约定（OpenAI 定义的工具调用 JSON schema），MCP 是持续连接的协议层。打个比方：Function Calling 是「打电话说一件事」，MCP 是「专线电话」——建立连接后可以持续双向通信。MCP 还多了 Resource（静态知识暴露）和 Prompt Templates（可复用提示模板），比纯 Function Calling 更完整。
Q: MCP 的 Tools、Prompts、Resources 分别是什么？
Tools：模型可以调用的可执行函数（查数据库、调 API）
Prompts：预定义的提示模板，用户选择后注入对话（代码审查模板、commit 消息模板）
Resources：静态数据暴露（文档、schema、配置），模型可以读取但不执行

很多新手把所有能力都塞进 Tools，这是错的——文档应该用 Resource，模板应该用 Prompt。
Q: 怎么保证 MCP 调用的安全性？
三个层面：
① 操作粒度白名单——不是「允许访问文件系统」，而是「允许读 workspace 目录」「允许执行白名单命令」
② 用户确认机制——高风险操作（删除、外部发送）必须经过用户确认，类似 sudo
③ 沙箱隔离——MCP Server 运行在受限环境中，不能随意访问宿主机

这和我在 Agent Harness 五层架构中的「能力层 + 门控层」是一脉相承的。
Q: 你在项目中实际用过 MCP 吗？
用过。我的 10 个 Bot Agent 团队中，通过 MCP 接入了 Jira API、Confluence Wiki、文件系统和 Shell。核心实践是不直接暴露原始能力——比如文件系统不暴露「读写任意路径」，而是封装成「读 workspace」「写 memory 目录」等有限操作，这样即使 Agent 被恶意 prompt 攻击，破坏范围也是可控的。
