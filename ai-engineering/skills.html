📦 Skills 机制

编号 AI-04分类：Agent 工程难度：⭐⭐⭐⭐标签：工具链核心
💡 一句话结论
Skills = AI Agent 的可复用知识插件——把「怎么用某个工具/完成某类任务」封装成按需加载的模块，而不是把所有知识全塞进 system prompt。核心理念：技术定义（MCP/Tools）和业务指导（Skills）分离。
🎬 一句话类比
Skills = 外卖骑手的导航 App
🛵
骑手不需要记住全城每一条路——他打开导航 App，导航告诉他「这个订单怎么送最优」。
Skills 就是 Agent 的导航 App——用 Docker 时加载 Docker Skill，用 K8s 时加载 K8s Skill。不加载时这些知识不在 context 里，省 token、防干扰。

MCP vs Skills — 职责分离

维度MCP / ToolsSkills

🎤 面试话术模板

📋 60秒完整版
Skills 是 AI Agent 的可复用知识插件系统，核心理念是「技术定义和业务指导分离」。

MCP/Tools 负责「能调用什么」——是技术契约，docstring 保持精简；Skills 负责「怎么用好」——是工作流、最佳实践、故障排查手册。

设计原则：
① 按需加载——不把所有 Skill 全塞进 system prompt，触发时才读取注入，省 token、防干扰
② 渐进式加载——先用简短目录（名称+描述），用户选择后再加载 SKILL.md 全文
③ 五维审计标准——我建立了一套评估体系：格式约束/证据绑定/失败恢复/验证纪律/决策规则各 5 分，满分 25，低于 15 分不准启用
📋 30秒精简版
Skills 是Agent 的按需知识插件——MCP 告诉你「能调用什么」，Skills 告诉你「怎么用好」。我用它管理 113 个专业技能的加载和审计，核心理念是技术定义和业务指导分离，避免把所有说明书塞进 system prompt。还建立了一套五维审计标准评估每个 Skill 的质量。

🔥 常见追问

Q: 为什么要把 MCP 和 Skills 分开？
因为它们的生命周期完全不同。MCP Server 的接口相对稳定（Docker API 不会天天变），但最佳实践会持续演进（「这个镜像构建技巧」是踩坑踩出来的）。

把它们混在一起有两个代价：① 修改 Skill 内容时需要重新部署 MCP Server；② system prompt 爆炸——113 个 Skill 的全文塞进去 model 直接超时。
Q: 五维审计标准是什么？
我建立的一套 Skill 质量评估体系：
① 格式约束（5分）——输出有没有「别写废话」的规定
② 证据绑定（5分）——查到的东西要不要独立验证
③ 失败恢复（5分）——一个步骤挂了，下一步做什么
④ 验证纪律（5分）——任务完成后有没有自检清单
⑤ 决策规则（5分）——多选项时按什么规则选

满分 25，低于 15 分的不准启用。这套标准来自微软 SkillOpt 论文的 bounded edit + validation gate 理念。
Q: Skills 机制和 RAG 有什么区别？
RAG 是通用知识检索——喂文档、建索引、语义搜索，适合「这段代码怎么写」这类开放问题。

Skills 是程序性知识——它是一套操作流程：「第一步检查配置，第二步执行命令，第三步验证结果」。不是检索一段文档，而是执行一套工作流。

实际项目中两者互补：RAG 查「Redis 持久化原理」，Skills 指导「现在给这个项目加 Redis 缓存怎么做」。
