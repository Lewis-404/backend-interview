netstat / ss — 网络连接诊断工具

编号：LINUX-05难度：⭐⭐⭐频率：🔥🔥🔥🔥
💡 一句话结论
netstat（已逐步被 ss 替代）用于查看网络连接、路由表、接口统计。ss -tlnp 查看 TCP 监听端口、ss -anp 查看所有连接。连接状态：LISTEN 监听 / ESTABLISHED 已建立 / TIME_WAIT 等待关闭 / CLOSE_WAIT 对方关闭。
🎬 生活类比 — netstat/ss = 港口调度中心
用「港口」看懂 TCP 连接状态
🚢
场景1：LISTEN = 泊位开放，ESTABLISHED = 轮船停靠
港口调度中心的大屏幕上，每个泊位（端口）都有状态。LISTEN = 泊位已开放，欢迎轮船停靠（服务器在监听）。ESTABLISHED = 轮船已停稳，正在装卸货物（三次握手完成，正常通信）。ss -tlnp = 查看所有开放的泊位和对应的码头工人（进程）。
🌊
场景2：TIME_WAIT = 轮船刚走，等浪平静
一艘轮船离开泊位后，港口规定必须等 2MSL（约 60 秒）浪才能平静。为什么？①确保最后发送的"再见信号"（ACK）对方收到了，如果丢了对方会重发"我要走了"（FIN），这边还能回应。②让上一艘船掀起的浪花（残留报文）完全消失，不影响下一艘船。
🪜
场景3：CLOSE_WAIT = 船走了但跳板没撤
对方轮船发来"我要走了"的信号（FIN），但码头工人忘了撤跳板（应用没调 close()）。泊位就这么空挂着，别人也用不了。CLOSE_WAIT 大量堆积一定是应用 bug——不是调内核参数能解决的，要去查代码。

📋 7步答题模板

第一步：一句话定义

netstat（network statistics）是网络连接查看工具。注意：netstat 已经被 ss（socket statistics）逐步替代，ss 更快（直接读 /proc/net 而非遍历 /proc/PID/fd），面试应该主动提 ss。

第二步：常用命令

ss -tlnp：查看所有 TCP 监听端口（t=TCP, l=LISTEN, n=数字格式不解析域名, p=显示进程）。ss -anp：查看所有连接（含 UDP、UNIX）。ss -s：汇总统计。ss -i：TCP 详细信息（拥塞窗口、重传等）。

第三步：TCP 连接状态（必考点）

LISTEN：服务器在监听端口。ESTABLISHED：三次握手完成，正常通信。TIME_WAIT：主动关闭方等待 2MSL（约 60s）确保最后的 ACK 到达。CLOSE_WAIT：被动关闭方收到 FIN 但还没调用 close()，应用层 bug 常见。

第四步：TIME_WAIT 详解

为什么等 2MSL？①确保最后一个 ACK 丢包时对方能重发 FIN；②让旧连接的残留报文在网络上消失。大量 TIME_WAIT 优化：tcp_tw_reuse=1（复用）、tcp_timestamps=1（时间戳判断）。

第五步：CLOSE_WAIT 泄漏

CLOSE_WAIT 大量堆积 = 应用收到对方 FIN 但没有调用 close()。经典原因：应用代码忘记关闭连接、线程阻塞、连接池未释放。排查：ss -anp | grep CLOSE_WAIT 找到进程，检查应用代码。

第六步：netstat vs ss

ss 比 netstat 快很多（特别是大量连接时），因为 ss 直接从内核 netlink socket 获取数据，而 netstat 要遍历 /proc/PID/fd 目录。现代 Linux 默认应该用 ss。

第七步：实战场景

端口占用排查：ss -tlnp | grep :8080（谁在监听 8080）。连接统计：ss -s（看建立了多少连接）。大量 TIME_WAIT：ss -an state time-wait | wc -l（计数）。CLOSE_WAIT 异常：ss -an state close-wait。

⏱️
60秒面试表达
"现在更推荐用 ss 替代 netstat，ss 更快。TCP 有 11 种状态，面试最常问的是四个：LISTEN 监听、ESTABLISHED 正常通信、TIME_WAIT 主动关闭后等 2MSL、CLOSE_WAIT 被动关闭但应用没关。TIME_WAIT 多可以调内核参数优化，CLOSE_WAIT 多一定是应用 bug——忘了 close 连接。"
